Необходимые поля для переноса товаров между Wildberries и Ozon
Обязательные поля при создании карточки на Wildberries

API Wildberries (метод POST contentv2cardsupload) принимает список карточек. Каждая карточка включает категорию (subjectID) и список variants. В каждой вариации нужно указать

subjectID — идентификатор категории. Выбирается из каталога WB. Выбор категории определяет набор обязательных характеристик, получаемых через метод GET contentv2objectcharcs{id_категории}; поле required в ответе показывает, что характеристика обязательна.

vendorCode — артикул продавца. Это уникальный код товара, по которому WB идентифицирует позицию.

title — краткое название товара. По правилам WB не должно превышать 60 символов и не должно содержать повторов, пола или рекламных фраз.

description — описание товара. Должно быть информативным, без маркетинговых приёмов; WB советует указывать только факты.

brand — торговая марка. Выбирается из справочника WB; если бренда нет, можно использовать «Нет бренда».

characteristics — массив характеристик. Каждая характеристика передаётся парой id и value. Идентификаторы и типы характеристик зависят от категории; обязательные параметры нужно заполнить.

sizes — массив размеров (минимум один). Для каждой позиции указываются

techSize — размер (например «41‑45»);

wbSize — размер в формате WB (обычно строка типа «RU»);

price — цена в рублях;

skus — список штрихкодов. Штрихкод обязателен для каждой вариации; он должен быть уникальным внутри магазина.

Wildberries создаёт карточку асинхронно и проверяет корректность данных. Если обязательное поле пропущено или значение неверно, метод вернёт ошибку. При обновлении карточки (POST contentv2cardsupdate) нужно отправлять все поля, даже если меняется один атрибут.

Обязательные поля при создании карточки на Ozon

Ozon допускает загрузку товаров через API (POST v2productimport) или XLS‑шаблоны. В обоих случаях нужно заполнить базовые поля. Руководство по массовой загрузке (2026 г.) выделяет следующие обязательные колонки

offer_id — уникальный артикул продавца (до 50 символов). Значение должно быть алфавитно‑цифровым и не должно начинаться с «_» или «-», иначе загрузка отклоняется.

category_id — числовой идентификатор категории Ozon. Устаревшие категории возвращают код 410, поэтому категорию необходимо выбирать по актуальному дереву категорий.

name — название товара (до 250 символов), без HTML и маркетинговых лозунгов («BEST SELLER» и т. п.).

price — цена товара (только цифры и точка). Ozon принимает рубли; для других валют нужно указать currency_code в методе API.

stock — количество товара на складе. В шаблоне это обязательная колонка; в API остатки можно отправить отдельно через метод v1productimportstocks.

Дополнительные, но крайне важные поля

attributes — массив характеристик с идентификаторами атрибутов (attribute_id) и списком значений. Для каждой категории есть обязательные атрибуты (тип REQUIRED), получаемые через метод v3categoryattribute. Значения справочных атрибутов нужно искать через v2categoryattributevalues.

images — до 10 URL‑адресов изображений. Хотя графика не обязательна, Ozon снижает видимость товаров без фото и требует изображения не менее 1200×1200 пикселей.

barcode — штрихкод EAN‑13UPC‑AISBN‑13. Не обязательное, но рекомендуется поле; Ozon сверяет штрихкоды с реестром GS1 и задерживает модерацию при ошибках.

dimensions и weight — габариты упаковки и вес. В API эти поля присутствуют в объекте dimensions (height, width, depth, weight, dimension_unit, weight_unit). Они облегчают логистику и в ряде категорий обязательны.

При массовой загрузке Ozon рекомендует сначала заполнить обязательные колонки offer_id, category_id, name, price, stock, затем пройти локальную валидацию. Без указания обязательных атрибутов по категории (через attributes) товар не пройдет модерацию.

Сопоставление полей и приведение к единому виду

Чтобы перенести товар между маркетплейсами, необходимо получить данные из источника, привести их к требуемому формату и загрузить в целевую систему.

1. Категория

Wildberries → Ozon используйте subjectID источника, чтобы определить категорию. Постройте таблицу соответствий между subjectID и category_id Ozon или попросите пользователя выбрать категорию вручную. Для выбора категории Ozon вызовите метод v2categorytree (или v1description-categorytree) и найдите подходящий category_id. Затем запросите список обязательных атрибутов через v3categoryattribute.

Ozon → Wildberries аналогично определите subjectID на основе category_id товара Ozon. Получите список категорий WB через GET contentv2objectparentall и GET contentv2objectall, либо храните заранее подготовленную карту соответствий.

2. Артикул

vendorCode ↔ offer_id. При переносе из WB в Ozon используйте vendorCode как offer_id. Убедитесь, что значение уникально, соответствует правилам Ozon (не более 50 символов, без подчеркиваний или дефисов в первых трёх символах). При переносе из Ozon в WB присвойте offer_id в поле vendorCode.

3. Название

title ↔ name. Название товара нужно адаптировать под правила площадки. Для WB название до 60 символов, без пола и рекламных слов. Для Ozon допустимо до 250 символов, но запрещены маркетинговые лозунги и HTML. При переносе сокращайте или расширяйте строку и убирайте запрещённые фразы. Хорошая практика — формировать название по шаблону «Бренд — модель — ключевая характеристика».

4. Бренд

Оба маркетплейса требуют указания бренда. Проверьте, чтобы бренд присутствовал в справочниках площадок. В Ozon бренд обычно передаётся как значение атрибута (идентификатор бренда). Используйте метод v2categoryattributevalues для поиска нужного dictionary_value_id. В WB бренд указывается строкой; если бренда нет, впишите «Нет бренда».

5. Описание

Поле description присутствует в обеих системах. В Ozon описание можно передавать в отдельном поле JSON или как атрибут; избегайте HTML‑разметки, эмодзи и рекламных фраз. В WB описание следует писать лаконично и по существу; оно не должно противоречить действительному товару. При переносе очистите разметку и переадаптируйте текст к требованиям площадки.

6. Характеристикиатрибуты

В WB список характеристик определяется категорией GET contentv2objectcharcs{subjectID} возвращает id, name, type и признак required. При переносе в Ozon нужно сопоставить каждую характеристику с атрибутом Ozon. Сначала получите список обязательных атрибутов через v3categoryattribute (указав attribute_type REQUIRED). Затем найдите соответствие по названию и единицам измерения. Для справочных значений (цвет, страна, пол) запросите возможные варианты через v2categoryattributevalues и выберите dictionary_value_id. Значения, отсутствующие в словаре, можно передать как произвольный текст в value.

В обратном направлении (Ozon → WB) получите характеристики товара через POST v2productsinfoattributes или выгрузите товар целиком (v2productinfo). Для каждого атрибута найдите соответствующий id в WB по названию. Заполните массив characteristics в JSON‑запросе WB, указывая id и value согласно типу данных.

7. Размеры, количество и цена

В WB размеры и цены хранятся в массиве sizes techSize, wbSize, price, skus. Если товар имеет несколько размеров, каждая вариация должна быть отдельным элементом массива. Для Ozon подобная логика реализована через атрибуты (например, атрибут «Размер одежды») и опционально через отдельные товары. При переносе из WB в Ozon для каждого размера можно

Создать отдельную позицию Ozon с собственным offer_id (например, добавить суффикс размера) и указать индивидуальную цену и количество;

Либо использовать complex_id и списки параметров, если поддерживается в API.

Цены. В Ozon цена передаётся как price в рублях. В WB цена указывается внутри sizes. При переносе возьмите цену основной вариации или рассчитайте цену для каждой позиции. Не забудьте указать НДС (vat) в Ozon, если это требуется.

Stockquantity. В Ozon наличие передаётся в отдельном методе v1productimportstocks либо в файле. В WB наличие обновляется через API раздела «Статистика». После создания товаров синхронизируйте остатки.

8. Штрихкоды (barcodes)

В WB штрихкоды передаются в списке skus внутри каждого размера; они обязательны. В Ozon штрихкод (barcode) не обязателен, но рекомендуется и должен соответствовать стандартам EAN‑13UPC‑A. При переносе используйте те же значения. Если в WB несколько штрихкодов, выберите один для Ozon (или добавьте остальные как дополнительные barcodes, если поддерживается).

9. Изображения

Ozon принимает до 10 URL‑адресов изображений; фотографии должны быть не менее 1200×1200 пикселей и оптимизированы. В WB изображения нужно загружать через методы contentv3mediafile или contentv3mediasave и использовать их идентификаторы в карточке. При переносе загрузите изображения с исходной площадки на свой сервер или хранилище, получите новые URL и передайте в формате, соответствующем целевой системе.

Рекомендации по автоматизации

Создайте карту соответствия категорий между subjectID WB и category_id Ozon и храните её в базе. Обновляйте эту таблицу периодически, так как категории меняются.

Модули конвертации атрибутов должны учитывать типы данных (строка, число, справочник) и языки. Для справочников используйте API Ozon для поиска возможных значений.

Валидация перед загрузкой. Проверяйте названия, артикулы и штрихкоды на соответствие требованиям обеих платформ. Например, удаляйте недопустимые символы из offer_id для Ozon или сокращайте название для WB.

Мониторинг статуса. После отправки данных проверяйте статусы задач (метод Ozon v1productimportinfo) и список ошибок в WB (GET contentv2cardserrorlist), чтобы оперативно исправлять проблемы.

Следуя этим рекомендациям и правильно сопоставляя поля, можно автоматизировать перенос товаров между Wildberries и Ozon и обеспечить успешную модерацию на обеих площадках.
