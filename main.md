Сервис переноса карточек с Wildberries в Ozon и обратно
Задача

Необходимо разработать веб‑сервис, который позволяет пользователю подключить личные кабинеты Wildberries (WB) и Ozon, выбрать товары на одной площадке и создать карточки этих товаров на другой площадке. Сервис должен работать на Python и использовать API каждого маркетплейса для получения и загрузки карточек. Основные действия:

Авторизация и подключение площадок (получение токенов/ключей).

Загрузка списка товаров из источника по API.

Выбор пользователем нужных товаров для переноса.

Получение категорий и обязательных характеристик для целевой площадки.

Маппинг (преобразование) атрибутов между WB и Ozon.

Создание карточек на целевой площадке и отслеживание статуса импорта.

Ниже собрана актуальная информация о методах API Wildberries и Ozon, которые пригодятся для этой задачи. На момент поиска (15 марта 2026 г.) документация WB и Ozon регулярно обновляется; рекомендуется периодически проверять официальные источники.

Аутентификация и получение токенов
Ozon

В личном кабинете Ozon Seller необходимо сгенерировать пару Client ID и API‑key. Официальное руководство подчёркивает, что эти ключи следует передавать в каждом запросе к Seller API.

При загрузке товаров API позволяет передавать до 100 позиции за один запрос, а также требует указать категорию и обязательные характеристики. После импорта карточки проходят модерацию, и только затем становятся доступными для продажи.

Wildberries

Для работы с API Wildberries нужен Seller Token. Его можно получить в личном кабинете WB и передавать в заголовке Authorization при каждом запросе. На момент 2026 г. у WB несколько типов токенов (личный, сервисный, базовый, тестовый); для управления контентом используется сервисный токен.

Для вызовов, описанных ниже, используется базовый URL https://content-api.wildberries.ru/.

Методы Wildberries для работы с карточками
Получение иерархии категорий

Список категорий первого уровня – GET /content/v2/object/parent/all. Запрос возвращает список верхнеуровневых категорий с полями id и name; пример ответа приведён в статье на Хабр.

Список категорий второго уровня – GET /content/v2/object/all. Можно передать параметр parentID, чтобы получить дочерние категории, и параметры limit/offset для постраничного получения .

Список характеристик категории – GET /content/v2/object/charcs/{id_категории}. Запрос возвращает список атрибутов (характеристик) с их идентификаторами, названием, типом и обязательностью . Эти данные используются при создании карточки.

Создание карточки товара

Метод: POST /content/v2/cards/upload.

Тело запроса: массив карточек. Каждая карточка содержит subjectID (идентификатор категории), список variants, где указывается vendorCode (артикул продавца), title, description, brand, список characteristics (каждая характеристика — пара {id, value}) и список sizes (размеры, цена, штрихкоды). Пример тела запроса для категории «Носки» приведён в статье на Хабр.

Важно: Отправка запроса создаёт карточку асинхронно; результат возвращается после обработки. Wildberries советует формировать каждую карточку с указанием обязательных атрибутов и корректной структуры категорий.

Обновление и поиск карточек

Получение списка карточек – POST /content/v2/get/cards/list. Можно фильтровать по артикулу продавца и получить nmID — внутренний идентификатор номенклатуры WB.

Обновление карточек – POST /content/v2/cards/update. В тело запроса передаётся массив карточек с полным набором полей, включая nmID, vendorCode, characteristics, sizes и др. При обновлении необходимо передавать все поля карточки, даже если часть из них не меняется.

Проверка ошибок – GET /content/v2/cards/error/list возвращает список ошибок по карточкам (например, дубликаты штрихкодов).

Методы Ozon Seller API для работы с карточками
Получение информации о товарах

Список товаров – POST /v1/product/list. В запросе можно указывать фильтры (по SKU, offer_id, статусам) и настройки пагинации.

Информация о товаре – POST /v2/product/info. Запрос принимает product_id, offer_id или sku и возвращает подробные сведения о карточке (название, описание, характеристики).

Список характеристик конкретных товаров – POST /v2/products/info/attributes. Позволяет получить значения атрибутов для указанного списка товаров.

Категории и атрибуты

Дерево категорий – Ozon предоставляет методы POST /v2/category/tree (устаревает) и POST /v1/description-category/tree. Они возвращают иерархию категорий, из которой нужно выбрать подходящую категорию для нового товара.

Список характеристик категории – POST /v3/category/attribute. В тело запроса передаётся массив category_id и тип характеристик (например, REQUIRED), результат включает перечень атрибутов, которые нужно заполнить при создании товара.

Справочник значений атрибута – POST /v2/category/attribute/values. Позволяет получить возможные значения для конкретного атрибута (например, список брендов или размеров).

Создание и обновление товаров

Импорт товаров – POST /v2/product/import. В запросе указывается массив items (до 100 штук), где для каждого товара передаются offer_id, name, category_id, список attributes (каждый атрибут включает id, complex_id, values), price, old_price, размеры, вес и список URL‑адресов изображений. Пример в коллекции Postman показывает использование этого метода для создания гидрокостюма с разными характеристиками.

Статус импорта – запрос POST /v1/product/import/info принимает task_id и возвращает состояние задачи (успешно, в обработке, ошибки).

Обновление остатков – POST /v1/product/import/stocks позволяет обновить количество товаров на складах.

Обновление цен и других параметров – у Ozon отдельные методы для обновления цен (/v1/product/import/prices) и описаний; они требуют повторного прохождения модерации (об этом напоминает руководство).

Общая схема работы сервиса
1. Авторизация и подключение площадок

Пользователь регистрируется в вашем сервисе и переходит в личный кабинет.

Для WB он вводит seller token. Ваш сервис хранит его в зашифрованном виде.

Для Ozon он указывает Client ID и API‑key; сервис также сохраняет эти ключи.

При каждом обращении к API добавляйте нужные заголовки (Authorization для WB, Client-Id и Api-Key для Ozon).

2. Получение списка товаров

Из Wildberries: вызовите POST /content/v2/get/cards/list без фильтров или с фильтром по статусу, чтобы получить все карточки. Если карточек много, используйте курсорную пагинацию (метод v3/cards/cursor/list) — она позволяет последовательно получать порции карточек.

Из Ozon: используйте POST /v1/product/list с необходимыми фильтрами. При необходимости получите подробные данные по каждому товару через POST /v2/product/info или POST /v2/products/info/attributes.

3. Выбор товаров и парсинг атрибутов

Отобразите пользователю список товаров из источника (например, карточки WB) с основными полями (название, артикул, цена, наличие).

После выбора товаров загрузите полную информацию о каждом товаре (характеристики, размеры, изображения).

Получите ID категории WB (subjectID) и перечень характеристик (через GET /content/v2/object/charcs/{subjectID}) — это понадобится при маппинге.

4. Подбор категории на целевой площадке

Получите список категорий Ozon: вызовите POST /v2/category/tree или POST /v1/description-category/tree и постройте навигационное дерево.

Автоматически сопоставьте категорию WB и категорию Ozon (можно хранить таблицу соответствий; в сложных случаях предложите пользователю выбрать категорию вручную).

Получите список обязательных атрибутов для выбранной категории Ozon через POST /v3/category/attribute (передайте attribute_type: "REQUIRED").

Получите справочники значений для атрибутов, которые имеют фиксированные списки (например, бренды) через POST /v2/category/attribute/values.

5. Преобразование характеристик и формирование запроса

Создайте функцию, которая принимает характеристики товара из WB и находит для них соответствующие атрибуты Ozon. Например, атрибут «Цвет» у WB может иметь строковое значение «Красный», а у Ozon — справочный ID цвета; этот ID можно получить через справочник значений. Если нужного значения нет, можно передать значение в поле value с dictionary_value_id = 0 (такой подход допускается в примере запроса).

Аналогично преобразуйте размеры: в WB поле sizes.techSize и wbSize; в Ozon размеры описываются атрибутами (например, «Размер одежды») и габаритами (height, width, depth, weight и единицы измерения). При необходимости конвертируйте миллиметры и граммы.

Сформируйте JSON‑массив items для запроса POST /v2/product/import. Для каждого товара укажите:

offer_id — уникальный артикул продавца из WB (vendorCode).

category_id — ID выбранной категории Ozon.

name и description — названия и описания (можно взять из WB или составить заново).

attributes — массив преобразованных характеристик.

price, old_price, vat, размеры (height, width, depth, weight, dimension_unit, weight_unit).

images — массив URL изображений. Изображения с WB лучше скачивать и загружать на собрый сервер, чтобы избежать проблем с доступностью; для WB существует отдельный метод POST /content/v3/media/file для загрузки медиафайлов, но при переносе в Ozon достаточно передать публичные ссылки.

6. Отправка товаров на целевую площадку и отслеживание статуса

Отправьте сформированный массив в Ozon через POST /v2/product/import. В ответе вы получите task_id.

Периодически проверяйте статус импорта с помощью POST /v1/product/import/info и показывайте пользователю прогресс. После успешного завершения карточка появится в личном кабинете Ozon.

Если нужно переносить товары с Ozon в WB, выполняйте обратные шаги: получайте товары из Ozon, выбирайте категорию WB (через методы /content/v2/object/parent/all и /content/v2/object/all), запрашивайте характеристики и формируйте запрос на POST /content/v2/cards/upload. Помните, что WB требует передачу всех полей при создании и обновлении карточек, и после создания полезно проверять список ошибок.

После успешной загрузки обновите остатки и цены при помощи POST /v1/product/import/stocks (Ozon) и аналогичных методов WB (например, POST /supplier/stocks в statistics API) для синхронизации склада. Ozon также предлагает методы для обновления цен.

7. Обработка ошибок и пользовательские уведомления

При возникновении ошибок API WB (например, дублирование штрихкода, отсутствие обязательного атрибута) получите описание проблемы через GET /content/v2/cards/error/list и выведите пользователю.

В Ozon ошибки приходят в ответе на импорт или в статусе задачи. Анализируйте поле result.errors в ответе product/import/info и отображайте человеку, какие поля необходимо скорректировать.

Реализуйте уведомления (push‑сообщения или сообщения в интерфейсе), когда импорт завершён.

Особенности и рекомендации

Соблюдайте лимиты. Wildberries вводит отдельные лимиты на методы cards/upload, cards/upload/add и cards/update; при превышении лимитов запросы будут отклоняться. Ozon позволяет импортировать до 100 товаров за раз.

Категории и атрибуты. От выбора правильной категории зависит список обязательных характеристик и комиссия. Всегда проверяйте актуальность категорий и полный перечень обязательных полей. Оба API регулярно обновляются.

Модерация и задержка. На Ozon изменения в карточках вступают в силу только после модерации; обновлённая информация может появиться через несколько дней.

Безопасность. Храните токены, API‑ключи и штрихкоды в зашифрованном виде и избегайте их логирования. Разрешайте пользователю отозвать токены.

Вывод

API обеих маркетплейсов предоставляет достаточно методов, чтобы автоматически переносить карточки товаров. Главная сложность — корректно подобрать категорию и сопоставить атрибуты, поскольку структуры справочников WB и Ozon отличаются. Используя описанные выше методы (GET /content/v2/object/charcs/{id}, POST /v3/category/attribute, POST /v2/category/attribute/values) и основываясь на примерных маппингах атрибутов, можно реализовать эффективный конвертер. После создания товара обязательно проверяйте статус задания (Ozon) и список ошибок (WB), чтобы гарантировать, что карточка корректно прошла модерацию и отображается на целевой площадке.
