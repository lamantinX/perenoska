# Дизайн трассировочного логирования переноса

## Цель

Добавить в MVP сквозное логирование каждого отдельного запуска переноса так, чтобы по внутреннему токену можно было найти:

- внешние HTTP-запросы и ответы маркетплейсов;
- внутренние шаги форматирования и сборки payload;
- ошибки и результаты промежуточных вычислений.

## Границы

Один отдельный вызов `preview` или `launch` считается самостоятельным переносом для трассировки. Для каждого такого вызова создаётся новый базовый токен, а записи внутри него получают последовательные токены вида `<base_token>_1`, `<base_token>_2`, `<base_token>_3`.

`sync` не продолжает токен предыдущего вызова и при необходимости получает собственный новый базовый токен.

## Архитектура

### Контекст трассировки

В сервисе переноса создаётся лёгкий runtime-контекст трассировки:

- `base_token` для одного запуска;
- локальный счётчик последовательности;
- методы записи лог-события в БД.

Контекст передаётся в сервис маппинга и в клиентов маркетплейсов только на время текущего вызова.

### Типы событий

Поддерживаются два основных типа:

- `http` — внешний HTTP-запрос/ответ;
- `function` — внутренний шаг вычисления, форматирования или подготовки payload.

Для `http`:

- `request_url` содержит URL;
- `request_headers` и `response_headers` содержат заголовки;
- `request_body` и `response_body` содержат сериализованные тела.

Для `function`:

- `request_url` содержит имя операции или псевдо-URL вида `function://mapping/build_import_payload`;
- `request_body` содержит имя шага и входные данные;
- `response_body` содержит результат шага;
- `response_headers` обычно пустой JSON-объект.

### Точки логирования

Минимально обязательные точки:

- старт и результат `TransferService.preview`;
- старт и результат `TransferService.launch`;
- вызов `MappingService.build_import_payload`;
- внешние HTTP-вызовы клиентов маркетплейсов, которые используются во время preview/import;
- ошибки на любом из этих шагов.

### Маскирование данных

Перед записью в БД должны маскироваться чувствительные данные:

- `Authorization`;
- `Api-Key`;
- `Client-Id`;
- `token`;
- другие похожие ключи, если они встречаются в заголовках или JSON.

Маскирование должно сохранять структуру, но скрывать секретное значение.

## Хранилище

Добавляется таблица `transfer_logs`.

Обязательные поля:

- `created_at`;
- `token`;
- `request_url`;
- `request_headers`;
- `request_body`;
- `response_headers`;
- `response_body`.

Дополнительные поля:

- `base_token`;
- `sequence_no`;
- `event_type`;
- `operation`;
- `job_id`;
- `source_marketplace`;
- `target_marketplace`;
- `status_code`;
- `duration_ms`;
- `error_text`.

Эти поля нужны для фильтрации, анализа производительности и поиска ошибок без разбора больших JSON-тел.

## Поведение

### Генерация токенов

- при входе в `preview` или `launch` генерируется новый `base_token`;
- каждый вызов `log_event(...)` увеличивает локальный счётчик;
- итоговый токен собирается как `<base_token>_<sequence_no>`.

### Обработка ошибок

Если шаг завершился исключением:

- в лог пишется отдельная запись с очередным токеном;
- `error_text` заполняется текстом ошибки;
- `response_body` содержит сериализованную информацию об ошибке;
- исключение продолжает пробрасываться по текущей логике сервиса.

## Тестирование

Реализация делается через TDD.

Первый набор тестов:

- генерация последовательных токенов внутри одного контекста;
- запись строки в `transfer_logs` через слой БД;
- маскирование секретов.

Второй набор тестов:

- `preview` создаёт набор `function`-логов с токенами `_1`, `_2`, ...;
- `launch` создаёт новый независимый `base_token`;
- логи содержат входные и выходные данные ключевых этапов.

Третий набор тестов:

- клиентский HTTP-вызов пишет `http`-лог;
- ошибка клиента тоже пишет лог с `error_text`.

## Ограничения

- не менять продуктовую логику переноса;
- не переписывать существующие файлы целиком;
- использовать минимальные точечные изменения;
- сохранять UTF-8 и существующие русскоязычные тексты.
