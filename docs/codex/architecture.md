# Architecture

## Верхний уровень

- `app/main.py` создаёт приложение, инициализирует контейнер сервисов, БД и подключает роуты.
- `app/api/routes/` содержит HTTP-слой.
- `app/services/` реализует бизнес-логику.
- `app/clients/` инкапсулирует обращения к API WB и Ozon.
- `app/db.py` хранит схему SQLite и CRUD-операции.
- `app/static/` обслуживается как встроенный SPA.

## Поток данных

1. Пользователь проходит регистрацию или логин.
2. Токен сессии используется для обращения к API сервиса.
3. Подключения маркетплейсов сохраняются зашифрованно.
4. Каталог и категории загружаются через marketplace clients.
5. `TransferService` собирает preview и payload для импорта.
6. Задача сохраняется в SQLite и отправляется в целевой marketplace client.
7. Статус импорта синхронизируется отдельным вызовом.

## Ключевые зависимости

- `ServiceContainer` связывает `Database`, `AuthService`, `ConnectionService`, `CatalogService`, `TransferService` и client factory.
- `TransferService` зависит от каталога, подключений, mapping и marketplace clients.
- UI зависит от стабильности API-контрактов в `app/api/routes/`.

## Архитектурные ограничения

- нет очередей и фонового воркера;
- нет отдельной схемы миграций;
- нет выделенного frontend-приложения;
- часть маппинга категорий и атрибутов эвристическая.
