## Принципы

> **Миграция выполняется СРАЗУ после изменения стандарта.**

> **Порядок: сначала Workflows, затем Экземпляры.**

> **Содержание важнее версии.** Обновить `standard-version` недостаточно — нужно обновить шаги, чек-листы, примеры.

> **⚠️ КРИТИЧНО: `check-version-drift.py` проверяет только ВЕРСИИ в frontmatter, НЕ содержание.** Версии могут совпадать, но содержание не соответствовать стандарту. LLM ОБЯЗАН выполнять сравнение содержания вручную.

> **⚠️ КРИТИЧНО: Сравнивать ТЕКУЩЕЕ содержание.** Версии разные = содержание Workflows не соответствует стандарту. Прочитать ТЕКУЩИЙ стандарт, прочитать ТЕКУЩИЙ Workflow, найти расхождения и устранить их.

> **⚠️ КРИТИЧНО: Миграция скриптов и скиллов ОБЯЗАТЕЛЬНА.** Нельзя считать миграцию завершённой без обновления скриптов. Если скрипт реализует проверки — он ДОЛЖЕН реализовывать ВСЕ проверки из стандарта.

## 1. Что такое миграция

Миграция — процесс приведения зависимых файлов в соответствие с новой версией стандарта.

**Когда нужна миграция:**
- Изменился `standard-*.md` (увеличилась версия)
- Зависимые файлы ссылаются на старую версию

**Каскад изменений:**

```
standard-{object}.md (vX.Y → vX.Z)
         │
         ▼ Workflows: Связанные документы (для ТОГО ЖЕ объекта)
┌────────┴────────────────────────────┐
│ validation-{object}.md             │  ← обновить frontmatter (standard-version)
│ create-{object}.md                 │  ← обновить шаги/чек-листы/примеры
│ modify-{object}.md                 │
└────────┬────────────────────────────┘
         │
         ▼ Экземпляры
┌────────┴────────────────────────────┐
│ Все файлы с:                       │
│ standard: <путь к стандарту>       │  ← должны соответствовать новой версии
│ standard-version < версии стандарта│
└─────────────────────────────────────┘
```

## 2. Два уровня обновления

### 2.1. Workflows: Связанные документы

Workflows — это три типа инструкций для ИЗМЕНЁННОГО стандарта (для того же объекта):
- `validation-{object}.md` — валидация экземпляров этого объекта
- `create-{object}.md` — создание экземпляров этого объекта
- `modify-{object}.md` — модификация экземпляров этого объекта

Где `{object}` — имя объекта из `standard-{object}.md`.

**Что обновлять:**
1. Frontmatter: `standard-version: vX.Y` → `vX.Z`
2. Содержание: шаги, чек-листы, примеры — чтобы соответствовали новым требованиям стандарта
3. Валидация: каждая инструкция должна пройти проверку после обновления

**Как обновлять:** `/modify-instruction`

**Обязательная проверка содержания (⚠️ КРИТИЧНО):**

> **Версии разные = содержание не соответствует. Нужно найти ЧТО именно.**

**Процедура:**

1. **Прочитать ТЕКУЩИЙ стандарт** — понять все требования
2. **Прочитать ТЕКУЩИЙ Workflow** — понять что уже покрыто
3. **Найти расхождения:**
   - Что есть в стандарте, но НЕТ в Workflow?
   - Какие шаги устарели или неполные?
4. Извлечь секции стандарта: `grep -n "^## " <стандарт>`
5. Заполнить таблицу соответствия:

| Секция стандарта | validation | create | modify | Расхождение |
|------------------|------------|--------|--------|-------------|
| §1 ... | Шаг N | Шаг N | — | ✅ / ❌ описать |

6. Добавить недостающие шаги

### 2.2. Экземпляры

Файлы, СОЗДАННЫЕ по стандарту (имеют `standard: <путь>` в frontmatter).

**Критерии устаревания экземпляра:**
- Файл имеет `standard: <путь к стандарту>` в frontmatter
- `standard-version` в файле < `standard-version` в стандарте

**Проверка:**
```bash
python .instructions/.scripts/check-version-drift.py <стандарт>
```

**Что обновлять:** Структура, секции, формат — согласно новым требованиям.

**Как обновлять:** `/modify-{type}` (instruction, skill, agent, rule...)

### 2.3. Приоритеты экземпляров

| Приоритет | Критерий | Действие |
|-----------|----------|----------|
| Критичный | Экземпляр используется другими документами (входящие ссылки) | Обновить ОБЯЗАТЕЛЬНО |
| Обычный | Экземпляр используется в разработке | Обновить при возможности |
| Отложенный | Экземпляр помечен как deprecated или в архиве | Не обновлять |

## 3. Порядок миграции

> **Сначала Workflows, затем Экземпляры.**

1. Обновить связанные документы (validation, create, modify)
2. Обновить экземпляры
3. Обновить README.md области (если изменилась суть стандарта)

**Почему:** Нельзя обновлять экземпляры через `modify-{object}.md`, если эта инструкция ещё не обновлена под новую версию стандарта.

**Шаг 3: Обновление индексов**

После обновления Workflows и Экземпляров:

1. Проверить `README.md` в папке стандарта
2. Обновить описание в секции "# 1. Стандарты" (если изменилась суть стандарта)
3. Обновить ссылки на связанные документы

## 4. Формат отчёта о миграции

После миграции создаётся отчёт:

```markdown
## 📋 Отчёт о миграции

**Стандарт:** {путь} (v{старая} → v{новая})
**Дата:** {дата}

### Workflows
- ✅ validation-{object}.md
- ✅ create-{object}.md
- ✅ modify-{object}.md

### Экземпляры
- ✅ {N} файлов обновлено (100% критичных)
- ⏭️ {M} файлов отложено (причина: {причина}, план: {дата или условие})

### Валидация
- ✅ check-version-drift.py: 0 расхождений
- ✅ Содержание проверено

### Критерий успеха
- Все Workflows обновлены: ✅
- Все критичные Экземпляры обновлены: ✅
- Валидация пройдена: ✅
```

## 5. Обработка ошибок

| Ситуация | Действие |
|----------|----------|
| Ошибка при обновлении Workflows | Откатить изменения (git restore), исправить проблему, повторить |
| Ошибка при обновлении экземпляра | Пропустить файл, отметить в отчёте, продолжить |
| Ошибка валидации после миграции | Пересмотреть изменения, исправить, повторить валидацию |

---

## 6. Откат миграции

Если миграция завершилась с ошибками:

1. **Откатить изменения:**
   ```bash
   git restore --staged --worktree .
   git clean -fd
   ```
2. **Проверить состояние:**
   ```bash
   python .instructions/.scripts/check-version-drift.py <стандарт>
   ```
3. **Исправить проблему в стандарте или инструкциях**
4. **Повторить миграцию**

**Важно:** Откат возможен только до коммита изменений. После push в main — используется forward-миграция (создаётся новая версия стандарта с исправлением).

## 7. Частичная миграция

Если часть экземпляров нельзя обновить сейчас:

1. **Создать драфт с планом:**
   ```
   .claude/drafts/YYYY-MM-DD-migration-pending-{object}.md
   ```

   Содержание драфта:
   - Что было сделано (обновлённые файлы)
   - Что осталось (отложенные файлы)
   - Причина отложения
   - План завершения (когда и как)

2. **Отметить в отчёте:**
   > ⏭️ 3 файла отложены (причина: ожидают слияния feature-X, план: см. драфт)

3. **Когда пользователь сможет — завершить миграцию по драфту**

## 8. Каскадная миграция

Если обновлённый стандарт является **зависимостью** для других стандартов:

**Пример:**
- `standard-instruction.md` v1.2 → v1.3 (изменилась структура frontmatter)
- `standard-migration.md` ссылается на `standard-instruction.md` в frontmatter

**Действия:**

1. Обновить зависимые стандарты (Уровень 0)
2. Затем обновить Workflows каждого стандарта
3. Затем обновить Экземпляры каждого стандарта

**Проверка зависимостей:**
```bash
grep -r "standard: .*standard-{object}.md" .instructions/
```

## 9. Миграция скриптов и скиллов

При изменении стандарта проверить:

**1. Скрипты в `.instructions/.scripts/`:**
- Если скрипт парсит структуру объекта — обновить логику парсинга
- Если скрипт создаёт объект — обновить шаблон

**2. Скиллы в `/.claude/skills/`:**
- Если скилл ссылается на изменённую инструкцию — обновить промпт
- Если скилл использует старые термины/секции — обновить формулировки

**Воркфлоу:**
1. Найти скрипты в секции "Скрипты" Workflows (`validation-{object}.md`, `create-{object}.md`)
2. Найти скиллы в секции "Скиллы" Workflows
3. Проверить реализацию проверок — заполнить таблицу соответствия:

| Секция стандарта | Код ошибки в скрипте | Статус |
|------------------|---------------------|--------|
| §1 ... | E001 / ❌ | ✅ / Добавить |

4. Обновить через `/script-modify` и `/skill-modify`

## 10. Валидация корректности миграции

После обновления версий LLM должен проверить:

1. **Проверить структуру:**
   - Все обязательные секции из стандарта присутствуют
   - Формат соответствует новым требованиям

2. **Проверить содержание:**
   - Шаги, чек-листы, примеры обновлены под новую версию
   - Логика инструкций соответствует изменениям в стандарте
   - Нет устаревших формулировок или ссылок

3. **Если обнаружены расхождения:**
   - Исправить содержание экземпляра
   - Повторить валидацию

## 11. Примеры

### Миграция standard-api.md v1.0 → v1.1

**Сценарий:** Добавлена секция "Обработка ошибок"

**Шаг 1: Workflows**
- Обновить `validation-api.md`: добавить проверку секции "Обработка ошибок"
- Обновить `create-api.md`: добавить шаг создания секции "Обработка ошибок"
- Обновить `modify-api.md`: добавить шаг обновления секции "Обработка ошибок"

**Шаг 2: Экземпляры**
- Найти все файлы с `standard: .specs/.instructions/standard-api.md`
- Обновить каждый файл: добавить секцию "Обработка ошибок"
- Обновить `standard-version: v1.1` в frontmatter

**Шаг 3: Валидация**
```bash
python .instructions/.scripts/check-version-drift.py .specs/.instructions/standard-api.md
# Ожидаемый результат: 0 расхождений
```

**Шаг 4: Проверка содержания**
- LLM проверяет, что секция "Обработка ошибок" добавлена корректно
- Проверяет соответствие шаблону из стандарта
