## Когда валидировать

- После изменения `standard-*.md`
- После выполнения `/migration-create`
- Перед коммитом (автоматически через pre-commit хук)
- При подозрении на незавершённую миграцию

## Шаги

### Шаг 1: Проверка версий (version drift)

```bash
python .instructions/.scripts/check-version-drift.py <стандарт>
```

**Вывод (JSON):**
```json
{
  "standard": ".instructions/standard-migration.md",
  "standard_version": "v1.0",
  "workflows": [
    {"file": "validation-migration.md", "version": "v1.0", "status": "ok"},
    {"file": "create-migration.md", "version": "v1.0", "status": "ok"}
  ],
  "instances": [
    {"file": "example.md", "version": "v0.9", "status": "outdated"}
  ],
  "summary": {"total": 3, "ok": 2, "outdated": 1}
}
```

**Критерий успеха:** `summary.outdated = 0`

### Шаг 1.1: Проверка контентного покрытия

```bash
python .instructions/.scripts/check-content-drift.py <стандарт>
```

**Критерий успеха:** 0 непокрытых секций.

### Шаг 2: Проверка Workflows (⚠️ КРИТИЧНО — сравнить содержание)

> **Версии разные = содержание не соответствует. Нужно найти ЧТО именно.**

Связанные документы (validation, create, modify) должны:

1. Иметь `standard-version` = версии стандарта
2. Содержать шаги, соответствующие ТЕКУЩЕЙ версии стандарта

**Процедура проверки:**

1. **Прочитать ТЕКУЩИЙ стандарт** — понять все требования
2. **Прочитать ТЕКУЩИЙ Workflow** — понять что уже покрыто
3. **Найти расхождения:**
   - Что есть в стандарте, но НЕТ в Workflow?
   - Какие шаги устарели или неполные?

**Автоматическая проверка:**

```bash
python .instructions/.scripts/validate-instruction.py validation-{object}.md
python .instructions/.scripts/validate-instruction.py create-{object}.md
python .instructions/.scripts/validate-instruction.py modify-{object}.md
```

**Ручная проверка содержания (LLM):**
- Таблица соответствия "секция стандарта → шаг" заполнена
- Шаги обновлены под ТЕКУЩУЮ версию стандарта
- Чек-листы актуальны
- Примеры соответствуют новым требованиям

### Шаг 3: Проверка Экземпляров

Все экземпляры должны:

1. Иметь `standard-version` = версии стандарта
2. Соответствовать новым требованиям формата

**Как проверить:**

Для каждого файла из списка `instances` в выводе Шага 1:

```bash
python .instructions/.scripts/validate-{type}.py <файл>
```

Где `{type}` определяется по стандарту экземпляра.

### Шаг 4: Проверка содержания

> **ВАЖНО:** Формальное обновление версии не гарантирует корректность содержания.

LLM проверяет для каждого обновлённого файла:

1. **Структура:** Все обязательные секции из стандарта присутствуют
2. **Содержание:** Шаги/чек-листы/примеры обновлены под новую версию
3. **Ссылки:** Нет устаревших ссылок или формулировок

### Шаг 5: Проверка скриптов и скиллов

Скрипты и скиллы указаны в секциях "Скрипты" и "Скиллы" Workflows:

```bash
# Извлечь скрипты из Workflows
grep -A5 "^## Скрипты" validation-{object}.md | grep "\.py"
grep -A5 "^## Скрипты" create-{object}.md | grep "\.py"

# Извлечь скиллы из Workflows
grep -A5 "^## Скиллы" validation-{object}.md | grep "SKILL.md"
grep -A5 "^## Скиллы" create-{object}.md | grep "SKILL.md"
```

Для каждого найденного:
- Заполнить таблицу соответствия "секция стандарта → код ошибки в скрипте"
- Проверить актуальность логики парсинга
- Проверить актуальность шаблонов
- Проверить актуальность промптов

## Типичные ошибки

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `standard-version` устарел | Файл не обновлён после миграции | `/modify-{type}` |
| Структура не соответствует | Новые секции не добавлены | `/modify-{type}` |
| Workflows не обновлены | Сначала обновить инструкции | `/modify-instruction` |
| Скрипт возвращает ошибку | Логика не обновлена под новый формат | `/script-modify` |
| Скилл работает некорректно | Промпт ссылается на старую структуру | `/skill-modify` |
| Частичная миграция | Часть файлов отложена | См. драфт `migration-pending-*.md` |
