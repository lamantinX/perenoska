## Принципы

> **Миграция выполняется СРАЗУ после изменения стандарта.**

> **Порядок: сначала Workflows, затем Экземпляры.**

> **Содержание важнее версии.** Обновить `standard-version` недостаточно — нужно обновить шаги, чек-листы, примеры.

> **⚠️ КРИТИЧНО: КАЖДЫЙ затрагиваемый файл = контент + версия.** Миграция — это НЕ bump версии. Для каждого Workflow/экземпляра/скрипта LLM ОБЯЗАН:
> 1. Определить **что конкретно** изменилось в стандарте
> 2. Найти **все места** в файле, где это изменение должно быть отражено (шаги, чек-листы, таблицы ошибок, шаблоны, примеры)
> 3. **Обновить контент** в каждом таком месте
> 4. **Только после** обновления контента — обновить `standard-version`
>
> **Типичная ошибка:** LLM обновляет версии во всех файлах, но НЕ обновляет содержание. `check-version-drift.py` показывает "0 расхождений", но контент не соответствует стандарту. Это **провал миграции**.

> **⚠️ КРИТИЧНО: `check-version-drift.py` проверяет только ВЕРСИИ в frontmatter, НЕ содержание.** Версии могут совпадать, но содержание не соответствовать стандарту. LLM ОБЯЗАН выполнять сравнение содержания вручную.

> **⚠️ КРИТИЧНО: Сравнивать ТЕКУЩЕЕ содержание.** Версии разные = содержание Workflows не соответствует стандарту. Прочитать ТЕКУЩИЙ стандарт, прочитать ТЕКУЩИЙ Workflow, найти расхождения и устранить их.

> **⚠️ КРИТИЧНО: Шаг 6 (скрипты/скиллы) ОБЯЗАТЕЛЕН.** Нельзя считать миграцию завершённой без обновления скриптов. Если скрипт реализует проверки — он ДОЛЖЕН реализовывать ВСЕ проверки из стандарта.

> **⚠️ КРИТИЧНО: ОБЯЗАТЕЛЬНОЕ чтение документов.** Перед любыми изменениями LLM ОБЯЗАН прочитать (через Read tool) ПОЛНОСТЬЮ:
> 1. Стандарт (`standard-*.md`) — весь документ
> 2. Все Workflows (`validation-*.md`, `create-*.md`, `modify-*.md`) — весь документ каждого
> 3. Все Экземпляры — frontmatter + первые 50 строк каждого
>
> **ЗАПРЕЩЕНО:** Обновлять версии без предварительного чтения. Даже если "версии синхронизированы" — нужно прочитать и сравнить содержание.

## Шаги

### Шаг 1: Определить изменённый стандарт

Какой `standard-*.md` был изменён?

```bash
# Показать изменения
git diff --name-only HEAD~1 | grep "standard-"
```

### Шаг 2: Проверить версию стандарта

Убедиться, что версия стандарта увеличена:

```bash
python .instructions/.scripts/bump-standard-version.py --check <стандарт>
```

Если версия не увеличена — увеличить:

```bash
python .instructions/.scripts/bump-standard-version.py <стандарт>
```

### Шаг 3: Найти зависимые файлы

```bash
python .instructions/.scripts/check-version-drift.py <стандарт>
```

```bash
# Контентное покрытие
python .instructions/.scripts/check-content-drift.py <стандарт>
```

**Вывод покажет:**
- Workflows: связанные документы (validation, create, modify)
- Экземпляры
- Контентное покрытие: таблица «секция стандарта → шаг workflow → ERROR_CODE скрипта»

### Шаг 4: Обновить Workflows

**⚠️ ОБЯЗАТЕЛЬНАЯ ПРОЦЕДУРА — выполнить для КАЖДОГО Workflow:**

#### 4.1. Прочитать стандарт ПОЛНОСТЬЮ (⚠️ ОБЯЗАТЕЛЬНО)

> **ЗАПРЕЩЕНО** пропускать чтение. Даже если "уже знаешь содержание" — прочитать заново.

```
Read tool: <путь-к-стандарту> (весь документ, без limit)
```

После чтения — извлечь список секций:
```bash
grep -n "^## " <путь-к-стандарту>
```

Записать список секций (например: §1 Определение, §2 Формат, §3 Примеры...).

#### 4.2. Прочитать ВСЕ Workflows ПОЛНОСТЬЮ (⚠️ ОБЯЗАТЕЛЬНО)

> **ЗАПРЕЩЕНО** пропускать чтение. Каждый документ — полностью.

```
Read tool: validation-{object}.md (весь документ, без limit)
Read tool: create-{object}.md (весь документ, без limit)
Read tool: modify-{object}.md (весь документ, без limit)
```

#### 4.3. Сравнить СОДЕРЖАНИЕ стандарта и Workflows (⚠️ КРИТИЧНО)

> **Версии разные = содержание не соответствует. Нужно найти ЧТО именно.**

**Процедура:**

1. **Проанализировать прочитанный стандарт** — понять все требования
2. **Проанализировать прочитанные Workflows** — понять что уже покрыто
3. **Найти расхождения:**
   - Что есть в стандарте, но НЕТ в Workflow?
   - Какие шаги устарели или неполные?

**Для каждой секции стандарта проверить:**
- Есть ли соответствующий шаг в `validation-{object}.md`?
- Есть ли соответствующий шаг в `create-{object}.md`?
- Есть ли соответствующий шаг в `modify-{object}.md`?

**Таблица соответствия:**

> Скопировать таблицу из вывода `check-content-drift.py` (Шаг 3). Проверить маппинг, дополнить колонку «Расхождение» описанием конкретных пробелов.

| Секция стандарта | validation | create | modify | script | Расхождение |
|------------------|------------|--------|--------|--------|-------------|
| §1 ... | Шаг N | Шаг N | — | E001 | ✅ / ❌ описать |
| §2 ... | Шаг N | Шаг N | Шаг N | E002 | ✅ / ❌ описать |

#### 4.4. Обновить контент КАЖДОГО Workflow (⚠️ КРИТИЧНО)

> **Для КАЖДОГО Workflow выполнить ОБА типа обновлений:**

**Тип A — Добавить недостающее:** Если новая секция/правило стандарта не покрыты — добавить шаг/проверку/пункт чек-листа.

**Тип B — Обновить существующее:** Если секция стандарта **изменена/расширена** — найти и обновить **все места** в Workflow, где эта секция упоминается:
- **Шаги** — обновить описание проверки/создания
- **Чек-листы** — добавить/обновить пункты
- **Таблицы ошибок** — добавить новые ERROR_CODES
- **Примеры** — обновить если содержат устаревшие данные
- **Шаблоны** — обновить если содержат устаревшие значения (например, `standard-version`)

**Контрольный вопрос для каждого Workflow:** _"Если LLM будет выполнять этот Workflow — он узнает о новом правиле/изменении из текста Workflow?"_ Если нет — контент не обновлён.

```
/instruction-modify validation-{object}.md
# → Добавить/обновить шаг для секции §N + чек-лист + таблицу ошибок
```

#### 4.5. Обновить frontmatter (ПОСЛЕ контента)

> **ЗАПРЕЩЕНО:** Обновлять frontmatter ДО обновления контента. Версия — подтверждение что контент актуален.

```yaml
standard-version: vX.Z  # новая версия — ТОЛЬКО после обновления контента
```

### Шаг 5: Обновить Экземпляры

**⚠️ ОБЯЗАТЕЛЬНЫЙ ШАГ — выполняется СРАЗУ после Workflows.**

#### 5.1. Определить режим обновления

| Условие | Режим | Действие |
|---------|-------|----------|
| Экземпляров ≤ 5 | Интерактивный | Вызывать `/modify-{type}` последовательно |
| Экземпляров > 5 И изменения минимальные | Драфт-план | Создать драфт с планом обновления |
| Экземпляров > 5 И изменения существенные | Интерактивный | Вызывать `/modify-{type}` последовательно |

**Изменения минимальные** = только обновление `standard-version` в frontmatter (содержание Workflows уже соответствовало стандарту).

**Изменения существенные** = добавлены/изменены шаги, секции, требования в Workflows.

#### 5.2. Интерактивный режим

Для каждого экземпляра из списка `instances`:

```
/modify-{type} <файл>
```

Где `{type}` определяется по стандарту:
- `standard-instruction.md` → `/instruction-modify`
- `standard-skill.md` → `/skill-modify`
- `standard-agent.md` → `/agent-modify`
- и т.д.

#### 5.3. Режим драфт-плана

> **Когда:** Много экземпляров (> 5) + изменения минимальные.

**Создать драфт:**

```
/.claude/drafts/{дата}-migration-instances-{object}.md
```

**Содержание драфта:**

1. **Контекст:** Какой стандарт, какие изменения в Workflows
2. **Список экземпляров:** Таблица с путями, текущими версиями, скиллами
3. **План выполнения:** Последовательность команд `/modify-{type}` для ВСЕХ экземпляров
4. **Валидация:** Команда проверки после выполнения

**Формат команд в плане:**

```bash
# Область {область}/
/instruction-modify {путь1}
/instruction-modify {путь2}
...
```

**После создания драфта:** Выполнить команды из плана ПОСЛЕДОВАТЕЛЬНО.

#### 5.4. Приоритеты экземпляров

| Приоритет | Критерий | Действие |
|-----------|----------|----------|
| Критичный | Используется другими документами | Обновить ОБЯЗАТЕЛЬНО |
| Обычный | Используется в разработке | Обновить при возможности |
| Отложенный | Deprecated или в архиве | Отметить в драфте, не обновлять |

### Шаг 6: Обновить скрипты и скиллы

**⚠️ ЭТОТ ШАГ ОБЯЗАТЕЛЕН — нельзя пропускать!**

#### 6.1. Найти связанные скрипты

```bash
# Скрипты, упомянутые в Workflows
grep -r "\.py" <путь-к-validation>.md | grep -v "^#"

# Скрипты, ссылающиеся на стандарт
grep -r "standard-{object}.md" **/.scripts/
```

#### 6.2. Проверить реализацию проверок в скриптах

Для каждого найденного скрипта (например, `validate-{object}.py`):

1. Прочитать скрипт
2. Извлечь ERROR_CODES или реализованные проверки
3. Сравнить с секциями стандарта

**Таблица соответствия (заполнить):**

| Секция стандарта | Код ошибки в скрипте | Статус |
|------------------|---------------------|--------|
| §1 ... | E001 / ❌ | ✅ / Добавить |
| §2 ... | E002, E003 / ❌ | ✅ / Добавить |

#### 6.3. Добавить недостающие проверки

```
/script-modify validate-{object}.py
# → Добавить проверку E00N для секции §N стандарта
```

#### 6.4. Проверить связанные скиллы

```bash
grep -r "SSOT:" .claude/skills/ | grep "{object}"
```

Для каждого скилла проверить, что SSOT-ссылка актуальна.

**ЗАПРЕЩЕНО:** Считать миграцию завершённой, если скрипты не проверены/обновлены.

### Шаг 7: Обновить README

Если изменилась суть стандарта:

1. Обновить описание в `README.md` области
2. Обновить ссылки на связанные документы

### Шаг 8: Валидация

```bash
python .instructions/.scripts/check-version-drift.py <стандарт>
```

```bash
# Контентное покрытие
python .instructions/.scripts/check-content-drift.py <стандарт> --check
```

**Критерий успеха:** 0 расхождений (версии) + 0 непокрытых секций (контент).

Дополнительно: `/migration-validate <стандарт>`

### Шаг 9: Отчёт

Вывести отчёт по шаблону из [standard-migration.md § 4](./standard-migration.md#4-формат-отчёта-о-миграции).

## Чек-лист

### Подготовка
- [ ] Определён изменённый стандарт
- [ ] Версия стандарта увеличена
- [ ] Найдены зависимые файлы

### Workflows (⚠️ СОДЕРЖАНИЕ, не только версия!)
- [ ] **Read tool:** Прочитан стандарт ПОЛНОСТЬЮ (весь документ)
- [ ] **Read tool:** Прочитан `validation-{object}.md` ПОЛНОСТЬЮ
- [ ] **Read tool:** Прочитан `create-{object}.md` ПОЛНОСТЬЮ
- [ ] **Read tool:** Прочитан `modify-{object}.md` ПОЛНОСТЬЮ
- [ ] Найдены расхождения (что в стандарте есть, а в Workflow нет)
- [ ] Таблица соответствия "секция → шаг" заполнена
- [ ] **Для КАЖДОГО Workflow — контент обновлён:**
  - [ ] `validation-{object}.md`: шаги, чек-лист, таблица ошибок отражают изменение
  - [ ] `create-{object}.md`: шаги, чек-лист, примеры отражают изменение
  - [ ] `modify-{object}.md`: шаги, чек-лист, примеры отражают изменение
- [ ] **Шаблоны** в стандарте обновлены (standard-version, новые поля/секции)
- [ ] **Только после контента** — frontmatter `standard-version` обновлён в каждом Workflow

### Экземпляры (⚠️ ОБЯЗАТЕЛЬНО после Workflows!)
- [ ] **Read tool:** Прочитан frontmatter + первые 50 строк КАЖДОГО экземпляра
- [ ] Определён режим: интерактивный (≤ 5) или драфт-план (> 5 + минимальные изменения)
- [ ] **Если драфт-план:** создан драфт `/.claude/drafts/{дата}-migration-instances-{object}.md`
- [ ] **Если драфт-план:** драфт содержит список ВСЕХ Экземпляров + команды `/modify-{type}`
- [ ] Выполнены ВСЕ команды `/modify-{type}` для экземпляров
- [ ] Критичные экземпляры обновлены
- [ ] Обычные экземпляры обновлены
- [ ] Отложенные экземпляры отмечены в драфте (причина)

### Скрипты и скиллы (⚠️ ОБЯЗАТЕЛЬНО!)
- [ ] Скрипты найдены (`grep` по Workflows)
- [ ] Таблица соответствия "секция → код ошибки" заполнена
- [ ] Скрипты реализуют ВСЕ проверки из стандарта
- [ ] Скиллы проверены (SSOT-ссылки актуальны)

### Завершение
- [ ] README обновлён (если нужно)
- [ ] `check-version-drift.py`: 0 расхождений (версии)
- [ ] `check-content-drift.py`: 0 непокрытых секций (контент)
- [ ] Содержание проверено вручную (секции ↔ шаги ↔ скрипты)
- [ ] Отчёт сформирован

## Примеры

### Миграция standard-instruction.md v1.1 → v1.2

**Сценарий:** Добавлена секция "Каскадное обновление" в стандарт.

```bash
# 1. Проверить расхождения версий
python .instructions/.scripts/check-version-drift.py .instructions/standard-instruction.md

# 2. Извлечь секции стандарта
grep -n "^## " .instructions/standard-instruction.md
# Вывод:
# 15: ## 1. Четыре типа инструкций
# 30: ## 2. Структура файла
# 50: ## 3. Каскадное обновление  ← НОВАЯ СЕКЦИЯ
# ...

# 3. Заполнить таблицу соответствия для Workflows
# | Секция | validation | create | modify |
# |--------|------------|--------|--------|
# | §1     | Шаг 1      | Шаг 2  | —      |
# | §2     | Шаг 2      | Шаг 3  | Шаг 1  |
# | §3 NEW | ❌         | ❌     | ❌     | ← ДОБАВИТЬ!

# 4. Обновить Workflows
/instruction-modify .instructions/validation-instruction.md
# → Добавить "Шаг N: Проверить секцию Каскадное обновление"

/instruction-modify .instructions/create-instruction.md
# → Добавить "Шаг N: Создать секцию Каскадное обновление"

# 5. Найти связанные скрипты
grep -r "\.py" .instructions/validation-instruction.md
# Вывод: validate-instruction.py

# 6. Проверить реализацию проверок в скрипте
# Прочитать validate-instruction.py, найти ERROR_CODES
# Сравнить с секциями стандарта:
# | Секция | Код в скрипте |
# |--------|---------------|
# | §1     | I001          |
# | §2     | I002, I003    |
# | §3 NEW | ❌            | ← ДОБАВИТЬ I004!

# 7. Обновить скрипт
/script-modify .instructions/.scripts/validate-instruction.py
# → Добавить проверку I004 для секции "Каскадное обновление"

# 8. Обновить Уровень 2 (все инструкции)
/instruction-modify <файл>
# → Обновить standard-version: v1.3
# → Добавить секцию если требуется

# 9. Валидация
python .instructions/.scripts/check-version-drift.py .instructions/standard-instruction.md
# Ожидаемый результат: 0 расхождений
```

### Частичная миграция

**Сценарий:** 3 экземпляра нельзя обновить сейчас (ожидают ревью).

```bash
# После обновления основных экземпляров:

# 1. Создать драфт
# .claude/drafts/2026-02-02-migration-pending-instruction.md

# Содержание драфта:
# - Обновлено: 47 файлов
# - Отложено: 3 файла (specs/api-v2.md, specs/auth.md, specs/db.md)
# - Причина: ожидают ревью PR #123
# - План: обновить после слияния PR

# 2. Отметить в отчёте
# ⏭️ 3 файла отложены (причина: PR #123, план: см. драфт)
```
