## Принципы

> **Скрипты автоматизируют шаги в инструкциях.** Если шаг инструкции требует поиска, фильтрации или трансформации данных — он должен быть реализован скриптом.

> **Скрипты создаются только для validation/create/modify.** Инструкции типа standard не имеют скриптов.

> **Не дублировать существующие скрипты.** Перед созданием — проверить через `parse-docstrings.py`.

> **Вызывается при создании/модификации инструкций.** Если в инструкции появляется шаг, требующий автоматизации — создать скрипт.

> **Шаблоны — из примеров SSOT.** При создании файлов использовать шаблоны из секции "Примеры". Запрещено придумывать свой формат.

**Принципы программирования** (SSOT: [standard-principles.md](./standard-principles.md)):

| Принцип | Суть |
|---------|------|
| **KISS** | Максимально простой код |
| **DRY** | Не дублировать код |
| **YAGNI** | Не добавлять функционал "на будущее" |
| **Наименьшее удивление** | Код делает то, что ожидает пользователь |
| **SOLID (S)** | Один скрипт = одна задача |
| **Читаемость** | Docstring, типизация, понятные имена |
| **Обработка ошибок** | Явная, не голый `except:` |
| **Минимум зависимостей** | Не подключать библиотеку ради одной функции |

**Валидация:** [validation-principles.md](./validation-principles.md)

## Когда создавать скрипт

**Создавать скрипт когда:**

| Критерий | Порог |
|----------|-------|
| Шаг инструкции требует поиска/фильтрации/трансформации | Любой такой шаг |
| Повторяемость | > 2 раз |
| Количество шагов | > 3 шагов |
| Валидация данных | Требуется |

**НЕ создавать скрипт когда:**

- Разовая операция
- Простое действие (1-2 команды)
- Логика уже реализована в существующем скрипте
- Инструкция типа standard (только для create/modify/validation)

## Шаги

### Шаг 1: Проверить существующие скрипты

> **ОБЯЗАТЕЛЬНО перед созданием** — убедиться, что нет скрипта с аналогичным функционалом.

#### 1.1. Получить полный список скриптов

```bash
python .instructions/.scripts/parse-docstrings.py
```

**Вывод:** список всех скриптов с описаниями — передать LLM для анализа.

#### 1.2. LLM анализирует

1. Есть ли скрипт с аналогичным функционалом?
2. Есть ли похожий скрипт в другой области?
3. Можно ли доработать существующий?

#### 1.3. Если найден похожий скрипт

**Прочитать:**
- Код скрипта
- Связанную инструкцию (из секции "Скрипты")
- Понять контекст использования

**Анализ:**
- Можно ли переиспользовать как есть?
- Нужна ли доработка?
- Нужен ли новый скрипт в своей директории?

#### 1.4. Спросить пользователя

> **ОБЯЗАТЕЛЬНО** использовать `AskUserQuestion` tool с кликабельными опциями.

**Варианты:**
1. Переиспользовать `{name}` — уже делает нужное
2. Доработать `{name}` — добавить функционал → `/script-modify`
3. Создать новый в `{область}/.instructions/.scripts/`

#### 1.5. Выполнить действие

| Выбор пользователя | Действие |
|-------------------|----------|
| Переиспользовать | Добавить ссылку в секцию "Скрипты" инструкции |
| Доработать | Вызвать `/script-modify {путь}` |
| Создать новый | Продолжить с Шага 2 |

### Шаг 2: Определить назначение

```
Вопросы:
1. Что делает скрипт? (одно предложение)
2. Какой шаг инструкции автоматизирует?
3. Какие входные данные?
4. Какой выход?
5. К какой инструкции привязан? (validation/create/modify)
```

### Шаг 3: Выбрать имя и расположение

**Имя** — по паттерну из [standard-script.md § 3](./standard-script.md#3-расположение):

| Паттерн | Когда использовать |
|---------|-------------------|
| `validate-{object}.py` | Валидация |
| `{action}-{object}.py` | Действие над объектом |
| `{object}.py` | CRUD операции |
| `parse-{object}.py` | Парсинг/анализ |
| `find-{criteria}.py` | Поиск по критерию |

**Расположение:**

```
{область}/.instructions/.scripts/{script-name}.py
```

### Шаг 4: Создать файл по шаблону

```bash
python .instructions/.scripts/create-script-file.py {name} [--area {path}] [--description "{text}"]
```

Или вручную по шаблону:

```python
#!/usr/bin/env python3
"""
{script-name}.py — {Краткое описание}.

Использование:
    python {script-name}.py <input> [--option <value>]

Примеры:
    python {script-name}.py file.txt
    python {script-name}.py data.json --output result.json

Возвращает:
    0 — успех
    1 — ошибка
"""

import argparse
import sys
from pathlib import Path


def find_repo_root(start_path: Path) -> Path:
    """Найти корень репозитория (папка с .git)."""
    current = start_path.resolve()
    while current != current.parent:
        if (current / ".git").exists():
            return current
        current = current.parent
    return start_path.resolve()


def process(input_path: Path) -> bool:
    """Основная логика скрипта."""
    # Реализовать логику
    print(f"✅ Обработан: {input_path}")
    return True


def main():
    if sys.platform == "win32":
        sys.stdout.reconfigure(encoding="utf-8")
        sys.stderr.reconfigure(encoding="utf-8")

    parser = argparse.ArgumentParser(
        description="{Краткое описание}"
    )
    parser.add_argument("input", help="Входной файл")
    parser.add_argument(
        "--repo", default=".",
        help="Корень репозитория (по умолчанию: текущая папка)"
    )

    args = parser.parse_args()

    repo_root = find_repo_root(Path(args.repo))
    input_path = Path(args.input)

    success = process(input_path)
    sys.exit(0 if success else 1)


if __name__ == "__main__":
    main()
```

### Шаг 5: Реализовать логику

Соблюдать принципы из [standard-principles.md](./standard-principles.md):
- KISS — простой код
- DRY — без дублирования
- Обработка ошибок — явная
- Документация — docstring для функций

**Для скриптов валидации** (`validate-*.py`):
- Добавить константу `ERROR_CODES` — см. [standard-script.md § ERROR_CODES](./standard-script.md#error_codes-для-валидаторов)

### Шаг 6: Проверить принципы

> **Валидация:** [validation-principles.md](./validation-principles.md)

```bash
python .instructions/.scripts/validate-script.py --principles {path}
```

**При обнаружении нарушений:**
1. Исправить код
2. Повторить проверку

### Шаг 8: Протестировать


### Шаг 9: Обновить инструкцию

> **ОБЯЗАТЕЛЬНО:** вызвать скилл для обновления инструкции.

```

```

**Контекст для скилла:**
- Создан новый скрипт `{script}.py`
- Скрипт автоматизирует шаг `{номер и название шага}`
- Добавить скрипт в секцию "Скрипты" инструкции

**Формат строки в таблице "Скрипты":**
```markdown
| [{script}.py](./.scripts/{script}.py) | {описание} | Этот документ |
```

## Примеры

### Создание скрипта для шага инструкции

```bash
# Шаг инструкции: "Найти все файлы с standard: instruction-standard.md"
# → Нужен скрипт find-by-frontmatter.py

# 1. Проверить существующие
python .instructions/.scripts/parse-docstrings.py --search "frontmatter"
# Не найден → создаём

# 2. Назначение: поиск файлов по полю frontmatter
# 3. Имя: find-by-frontmatter.py
# 4. Путь: /.instructions/.scripts/find-by-frontmatter.py
# 5. Создать файл по шаблону, реализовать логику
# 6. Добавить в секцию "Скрипты" validation-instruction.md
# 7. Валидация
python .instructions/.scripts/validate-script.py .instructions/.scripts/find-by-frontmatter.py
```

### Переиспользование существующего скрипта

```bash
# Шаг инструкции: "Проверить все ссылки"
# → Проверяем существующие

python .instructions/.scripts/parse-docstrings.py --search "ссылки"
# Найден: validate-links.py — "Валидация ссылок в markdown"
# → Переиспользуем, не создаём новый
```
