## 1. KISS

**Keep It Simple, Stupid** — код должен быть максимально простым.

### Правило

Выбирать решение, которое легче понять будущему читателю кода, даже если оно содержит больше строк. Простота = минимум концепций для понимания, а не минимум символов.

Допустимо использовать более сложные конструкции, если они улучшают производительность на критичных участках (см. § 12).

### Пример

```python
# ❌ Плохо: переусложнённо
def process_files(files):
    return list(filter(lambda x: x.suffix == '.md',
                      map(lambda x: Path(x), files)))

# ✅ Хорошо: просто и понятно
def process_files(files):
    result = []
    for f in files:
        path = Path(f)
        if path.suffix == '.md':
            result.append(path)
    return result
```

### Признаки нарушения

- Вложенные lambda-выражения
- Цепочки map/filter/reduce из 3+ операций или с вложенными lambda
- Однострочники, которые сложно читать
- Преждевременные оптимизации

## 2. DRY

**Don't Repeat Yourself** — избегать дублирования кода.

### Правило

Каждая часть знания должна иметь единственное, непротиворечивое представление в системе.

### Пример

```python
# ❌ Плохо: дублирование
def validate_instruction(path):
    if not path.exists():
        print(f"❌ Файл не найден: {path}")
        return False
    # ...

def validate_script(path):
    if not path.exists():
        print(f"❌ Файл не найден: {path}")
        return False
    # ...

# ✅ Хорошо: общая функция
def check_file_exists(path, entity_type="Файл"):
    if not path.exists():
        print(f"❌ {entity_type} не найден: {path}")
        return False
    return True

def validate_instruction(path):
    if not check_file_exists(path, "Инструкция"):
        return False
    # ...

def validate_script(path):
    if not check_file_exists(path, "Скрипт"):
        return False
    # ...
```

### Признаки нарушения

- Копирование блоков кода (> 5 строк) между функциями
- Одинаковые сообщения об ошибках в разных местах
- Повторяющиеся паттерны валидации
- Функции с дублированием блоков кода (> 5 строк идентичного или похожего кода)

> **Критерий дублирования:** Повторение логики, даже если сообщения или параметры различаются.

## 3. YAGNI

**You Aren't Gonna Need It** — не добавлять функционал "на будущее".

### Правило

Реализовывать только то, что нужно сейчас. Не пытаться предугадать будущие требования.

### Пример

```python
# ❌ Плохо: функционал "про запас"
def validate(path, format='text', verbose=False,
             strict=False, cache=True, parallel=False,
             output_file=None, log_level='INFO'):
    # ...

# ✅ Хорошо: только то, что нужно сейчас
def validate(path):
    # ...
```

### Признаки нарушения

- Аргументы, которые не используются в теле функции (отсутствуют в коде после объявления)
- Пустые методы "для будущего расширения"
- Абстракции без конкретного применения
- Аргументы с дефолтными значениями, которые никогда не переопределяются при вызове

## 4. Принцип наименьшего удивления

**Principle of Least Astonishment** — код должен делать то, что ожидает пользователь.

### Правило

Поведение функции должно соответствовать её имени и документации. Никаких скрытых побочных эффектов.

### Пример

```python
# ❌ Плохо: неожиданное поведение
def delete_file(path):
    """Удаляет файл."""
    shutil.rmtree(path)  # Удаляет папку!

# ✅ Хорошо: предсказуемое поведение
def delete_file(path):
    """Удаляет файл."""
    Path(path).unlink()

def delete_directory(path):
    """Удаляет папку со всем содержимым."""
    shutil.rmtree(path)
```

### Правила

| Аспект | Правило |
|--------|---------|
| Имена функций | Точно отражают действие |
| Аргументы | Очевидные значения по умолчанию |
| Побочные эффекты | Документированы явно |
| Возвращаемые значения | Один тип или тип + None. Недопустимо: `return "ok"` в одной ветке и `return 0` в другой |

### Признаки нарушения

- Функция делает больше, чем говорит имя
- Неочевидные значения по умолчанию
- Скрытые изменения глобального состояния
- Разные типы возвращаемых значений в разных ветках

## 5. SOLID

**Применимые принципы:**

### S — Single Responsibility

**Один модуль/функция = одна задача.**

```python
# ❌ Плохо: функция делает всё
def main():
    validate_structure()
    validate_links()
    generate_report()
    send_email()

# ✅ Хорошо: разделение ответственности
# validate_structure() — валидация структуры
# validate_links() — валидация ссылок
# main() — координация, вызов других функций
```

### O — Open/Closed

**Расширение через дополнительные аргументы с дефолтными значениями.** Для добавления новой функциональности не переписывать тело функции, а добавить аргумент, который активирует новое поведение.

```python
# ❌ Плохо: изменение кода для нового формата
def output_result(data):
    print(json.dumps(data))

# ✅ Хорошо: расширение через аргументы
def output_result(data, format='text'):
    if format == 'json':
        print(json.dumps(data))
    else:
        print(data)
```

### D — Dependency Inversion

**Зависимость от абстракций.**

```python
# ❌ Плохо: зависимость от конкретики
def process(file_path: str):
    with open(file_path) as f:
        return f.read()

# ✅ Хорошо: зависимость от абстракции
def process(file_path: Path):
    return file_path.read_text()
```

### L — Liskov Substitution

**В Python:** Подклассы не должны нарушать контракт родительского класса.

```python
# ❌ Плохо: подкласс изменяет поведение
class File:
    def read(self): return "data"

class EmptyFile(File):
    def read(self): raise NotImplementedError  # Нарушение контракта

# ✅ Хорошо: подкласс расширяет поведение
class EmptyFile(File):
    def read(self): return ""  # Корректное расширение
```

### I — Interface Segregation

**В Python:** Использовать Protocol (PEP 544) для узких интерфейсов.

```python
# ❌ Плохо: один большой протокол
class Animal(Protocol):
    def fly(self): ...
    def swim(self): ...
    def walk(self): ...

# ✅ Хорошо: отдельные протоколы
class Flyable(Protocol):
    def fly(self): ...

class Swimmable(Protocol):
    def swim(self): ...
```

## 6. Читаемость и документация

### Правило

Код читается чаще, чем пишется. Приоритет — понятность для будущего читателя.

### Docstring для функций

```python
# ❌ Плохо: нет документации
def f(x, y):
    return [i for i in x if i.n in y]

# ✅ Хорошо: понятный код с документацией
def filter_files_by_names(files: list[Path], names: set[str]) -> list[Path]:
    """
    Фильтрует файлы по именам.

    Args:
        files: Список путей к файлам
        names: Множество допустимых имён

    Returns:
        Файлы, имена которых есть в names
    """
    return [f for f in files if f.name in names]
```

### Правила

| Элемент | Требование |
|---------|------------|
| Функции | Docstring обязателен |
| Аргументы | Типизация (type hints) |
| Переменные | Осмысленные имена |
| Константы | UPPER_CASE с комментарием |

### Признаки нарушения

- Однобуквенные имена переменных (кроме `i`, `j`, `k` в циклах for и `f` для file handles)
- Функции без docstring
- Магические числа без объяснения (выносить в константы, если используются > 1 раза или смысл неочевиден)
- Аббревиатуры без расшифровки

## 7. Обработка ошибок

### Правило

Ошибки должны обрабатываться явно. Никогда не игнорировать исключения.

### Пример

```python
# ❌ Плохо: игнорирование ошибок
def read_frontmatter(path):
    try:
        content = path.read_text()
        return parse_yaml(content)
    except:
        return {}

# ✅ Хорошо: явная обработка
def read_frontmatter(path: Path) -> dict | None:
    """Читает frontmatter из файла."""
    if not path.exists():
        print(f"⚠️ Файл не найден: {path}", file=sys.stderr)
        return None

    try:
        content = path.read_text(encoding='utf-8')
    except UnicodeDecodeError as e:
        print(f"⚠️ Ошибка кодировки {path}: {e}", file=sys.stderr)
        return None

    return parse_yaml(content)
```

### Правила

| Правило | Описание |
|---------|----------|
| Нет голого `except:` | Указывать конкретный тип исключения |
| Логирование | Ошибки выводить в stderr |
| Возвращаемые значения | None или пустая структура при ошибке |
| Ранний выход | Проверять критические условия (существование файла, валидность аргументов) в начале функции и выходить с ошибкой до выполнения основной логики |

### Признаки нарушения

- `except:` без указания типа исключения
- `pass` в блоке except
- Ошибки в stdout вместо stderr
- Возврат разных типов при ошибке и успехе

## 8. Минимизация зависимостей

### Правило

> **Не подключать библиотеку ради одной функции.**

### Пример

```python
# ❌ Плохо: тяжёлая зависимость для простой задачи
import pandas as pd

def count_lines(path):
    return len(pd.read_csv(path))

# ✅ Хорошо: стандартная библиотека
def count_lines(path):
    with open(path) as f:
        return sum(1 for _ in f)
```

### Допустимые зависимости

| Зависимость | Когда использовать |
|-------------|-------------------|
| Стандартная библиотека | Всегда |
| PyYAML | Работа с frontmatter |
| argparse | CLI (встроен) |

### Недопустимые зависимости

| Библиотека | Замена |
|------------|--------|
| requests | urllib.request |
| pandas | csv модуль |
| click | argparse |
| colorama | ANSI-коды напрямую |

### Признаки нарушения

- import тяжёлой библиотеки для одной функции
- Зависимость, которую нужно устанавливать через pip
- Использование библиотеки вместо стандартной функции

## 9. Приоритет принципов

При конфликте принципов применять в следующем порядке:

1. **Обработка ошибок** — корректность важнее всего
2. **DRY** — избегать дублирования
3. **KISS** — простота понимания
4. **YAGNI** — не добавлять лишнее
5. **Читаемость** — код читается чаще, чем пишется

**Пример:** Если DRY требует создать общую функцию, но она усложняет код (KISS) — выбрать DRY, если дублирование > 5 строк.

**Open/Closed vs KISS:** Расширение через аргументы допустимо, если это избегает дублирования кода.

## 10. Граничные случаи

### Дублирование в тестах

**Ситуация:** Тестовые функции дублируют setup-код.

**Применение DRY:** Вынести в фикстуру, если дублирование > 5 строк. Меньше — допустимо (читаемость теста важнее).

### Однобуквенные имена в математическом коде

**Ситуация:** Формула с переменными `x`, `y`, `z`.

**Применение читаемости:** Допустимо, если это соответствует математической нотации.

## 12. Производительность

> **Правило:** Оптимизация допустима, если профилирование показало узкое место.

**Принцип:** Сначала простой код (KISS), затем профилирование, затем оптимизация только горячих участков.

```python
# ✅ Хорошо: простой код для редких операций
def process_files(files):
    result = []
    for f in files:
        if f.suffix == '.md':
            result.append(f)
    return result

# ✅ Хорошо: оптимизация для частых операций (подтверждено профилированием)
def process_million_files(files):
    return [f for f in files if f.suffix == '.md']  # Быстрее на больших данных
```

**Когда оптимизировать:**
- Операция выполняется > 1000 раз в секунду
- Профилирование показало > 10% времени на этом участке

## 13. Тестируемость

> **Правило:** Код, соответствующий принципам, легко тестируется.

**Связь с принципами:**

| Принцип | Как помогает тестам |
|---------|---------------------|
| KISS | Меньше путей выполнения → меньше тестов |
| DRY | Изменение в одном месте → изменение теста в одном месте |
| Наименьшее удивление | Предсказуемое поведение → простые тесты |
| SOLID (S) | Один тест на одну функцию |
| SOLID (D) | Мокирование зависимостей |

**Пример:** Функция с глобальным состоянием (нарушение S) требует сложного setup/teardown в тестах.

## 14. Рефакторинг

> **Правило:** Рефакторить только если изменяешь код рядом.

**Принцип:** Не трогать работающий код ради соблюдения принципов.

**Когда рефакторить:**
- Добавляешь функциональность в файл → исправить нарушения в этом файле
- Code review показал критические проблемы (голый except)
- Код изменяется чаще 1 раза в месяц

**Когда НЕ рефакторить:**
- Код не менялся 3+ месяца
- Нет тестов для этого кода
- Проблема не блокирует разработку
