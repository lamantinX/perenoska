## 1. Типы ссылок

| Тип | Описание | Пример |
|-----|----------|--------|
| Внутренние | На файлы/папки проекта | `[README](/src/README.md)` |
| Внешние | На внешние ресурсы | `[Docs](https://example.com)` |
| Якорные | На секцию документа | `[Шаги](#шаги)` |

### Внутренние ссылки

```markdown
[Название](путь/к/файлу.md)
[Название](путь/к/папке/)
```

### Внешние ссылки

```markdown
[Название](https://example.com)
```

**Правила:**
- Предпочитать HTTPS
- Без trailing slash в конце URL (`https://example.com/docs`, НЕ `https://example.com/docs/`)
- Ссылаться на конкретную страницу (не корень домена)

## 2. Абсолютные vs относительные

| Тип | Формат | Когда использовать |
|-----|--------|-------------------|
| Абсолютные | `/путь/от/корня` | Между разными папками проекта |
| Относительные | `./файл` или `../папка` | Внутри одной папки или рядом |

### Правила выбора

```
Та же папка              → ./файл.md
Родитель (1 уровень)     → ../папка/файл.md
2+ уровня вверх          → /абсолютный/путь.md
Другая ветка структуры   → /абсолютный/путь.md
```

### Неправильно (цепочка > 2 уровней)

```markdown
[файл](../../../другая-папка/файл.md)  <!-- НЕПРАВИЛЬНО -->
```

### Правильно

```markdown
[файл](/другая-папка/файл.md)  <!-- Абсолютный путь -->
```

### Примеры

```markdown
<!-- Относительные -->
[format.md](./format.md)
[README родителя](../README.md)

<!-- Абсолютные -->
[CLAUDE.md](/CLAUDE.md)
[src/README.md](/src/README.md)
```

## 3. Ссылки на папки

### Формат (предпочтительно)

```markdown
[Название](путь/к/папке/README.md)
```

**Предпочтительно:** ссылка на README.md папки.

### Альтернативный формат

```markdown
[Название](путь/к/папке/)
```

Допустим trailing slash `/` в конце пути, если README.md отсутствует или недоступен.

### Примеры

```markdown
<!-- Предпочтительно -->
[skills/README.md](./skills/README.md)
[/src/](/src/README.md)

<!-- Допустимо (если README.md недоступен) -->
[skills/](./skills/)
[/src/](/src/)
```

## 4. Ссылки на файлы

### Формат

```markdown
[Название](путь/к/файлу.ext)
```

### Расширения

| Тип файла | Расширение |
|-----------|------------|
| Markdown | `.md` |
| YAML | `.yaml`, `.yml` |
| JSON | `.json` |
| Код | `.py`, `.ts`, `.go` |

## 5. Якорные ссылки

### Формат

```markdown
<!-- В том же файле -->
[Название секции](#название-секции)

<!-- В другом файле -->
[Название](путь/к/файлу.md#якорь)
```

### Правила формирования якоря

| Действие | Пример |
|----------|--------|
| Lowercase | `Шаги` → `шаги` |
| Пробелы → дефисы | `Формат описания` → `формат-описания` |
| Спецсимволы → удалить ИЛИ заменить на дефис | `1. Введение` → `1-введение`, `API (v2)` → `api-v2` |
| Сохранить цифры | `API v2` → `api-v2` |
| Кириллица остаётся | `Шаги работы` → `шаги-работы` |

### Примеры

```markdown
[См. Шаги](#шаги)
[Формат секции](./format.md#формат-секции)
[Скиллы в CLAUDE.md](/CLAUDE.md#скиллы-14)
```

## 6. Ссылки в SSOT структуры

В `/.structure/README.md` используется специальный формат для ссылок на папки.

### Формат

```markdown
### 🔗 [папка/](../папка/README.md)

**Краткое назначение.**

Описание содержимого.
```

### Правила

| Элемент | Правило |
|---------|---------|
| Иконка | `🔗` — папка имеет README |
| Название | `папка/` с trailing slash |
| Ссылка | На `README.md` папки |
| Заголовок | `###` (h3) |

### Исключение

Если папка — текущий документ:

```markdown
### 🔗 .structure/ (этот файл)
```

**Важно:** Все папки в SSOT структуры ДОЛЖНЫ иметь README.md. Ссылка на папку без README.md — ошибка структуры (требуется создать README.md).

## 7. Ссылки в frontmatter

```yaml
standard: .instructions/standard-instruction.md
index: .structure/.instructions/README.md
```

**Правила:**
- БЕЗ квадратных скобок
- БЕЗ ведущего `/`
- Относительный путь от корня репозитория

---

## 8. Ссылки на строки кода

Для файлов с кодом допустимы якоря на строки (GitHub/GitLab формат):

```markdown
[Функция validate](./validator.py#L42)
[Блок обработки ошибок](./validator.py#L100-L115)
```

**Правила:**
- Якорь формата `#LN` (одна строка) или `#LN-LM` (диапазон)
- Применимо только для файлов кода (не для .md)

---

## 9. Ссылки в блоках кода

Ссылки внутри блоков кода (` ``` `) НЕ валидируются — это примеры.

Ссылки в таблицах, инлайн-коде (`` ` ``) и цитатах — валидируются как обычные ссылки.

## 10. Ссылки между репозиториями

### В одном monorepo

Использовать абсолютные пути от корня monorepo:

```markdown
[API Docs](/packages/api/README.md)
```

### Между репозиториями

Использовать GitHub URL с привязкой к версии:

```markdown
[API Docs](https://github.com/org/repo/blob/v1.2.3/docs/api.md)
```

**Правила:**
- НЕ использовать `main`/`master` (ссылка может стать битой при изменениях)
- Привязывать к тегу версии или коммиту

## 11. Lifecycle ссылок

### Переименование/перемещение файлов

При переименовании файла:
1. Обновить ВСЕ ссылки на файл (использовать скрипт валидации ссылок)
2. Если файл — SSOT для скилла → обновить frontmatter скилла

### Удаление файлов

Перед удалением файла:
1. Проверить входящие ссылки: `grep -r "путь/к/файлу.md" .`
2. Удалить ИЛИ перенаправить ссылки на альтернативный документ
3. Если перенаправление невозможно — указать в commit message список битых ссылок

### Устаревшие ссылки

Если документ устарел, но не удалён:

```markdown
> ⚠️ **DEPRECATED:** Документ устарел. См. [новый документ](путь).
```

## 12. Ссылки в автогенерируемых файлах

Файлы с пометкой в frontmatter:

```yaml
autogenerated: true
generator: {script/tool}
```

**Правила:**
- Применяются ВСЕ правила стандарта ссылок
- Генератор ДОЛЖЕН создавать корректные ссылки
- Валидация выполняется после генерации
- Если ссылки битые — исправить генератор, не файл

## 13. Граф зависимостей ссылок

### Обязательные ссылки

| Тип документа | Обязательные ссылки |
|---------------|---------------------|
| Инструкция (standard-) | → standard-instruction.md (frontmatter) |
| Инструкция (create-/modify-/validation-) | → связанный standard-*.md (frontmatter) |
| Скилл | → SSOT-инструкция (frontmatter и текст) |
| README папки | → README родителя (секция "Полезные ссылки") |

### Проверка графа

- Каждый документ имеет минимум 1 входящую ссылку (кроме корневых)
- Нет циклических зависимостей в SSOT-цепочках
- Все обязательные ссылки присутствуют
