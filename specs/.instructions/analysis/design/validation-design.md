## Когда валидировать

После любого изменения в документе проектирования:

- Создание → [create-design.md](./create-design.md)
- Изменение → [modify-design.md](./modify-design.md)
- Перед переводом DRAFT → WAITING (включая финал create-воркфлоу) — проверить: 0 маркеров, 0 Dependency Barrier
- При code review

## Шаги

### Шаг 0: Автоматическая валидация

```bash
python specs/.instructions/.scripts/validate-analysis-design.py {путь}
```

Скрипт проверяет все правила DES001-DES035. Если валидация пройдена — **готово**, шаги 1-12 не нужны.

**Если скрипт недоступен** — выполнить шаги 1-12 вручную.

### Шаг 1: Расположение и именование

**SSOT:** [standard-design.md § 2](./standard-design.md#2-расположение-и-именование)

- Файл расположен в `specs/analysis/NNNN-{topic}/design.md`
- Папка соответствует regex: `^\d{4}-.+$`
- `NNNN` — четырёхзначный номер (0001, 0002, ...)
- `{topic}` — kebab-case, латиница
- Имя файла — `design.md` (фиксированное)

### Шаг 2: Frontmatter

**SSOT:** [standard-design.md § 3](./standard-design.md#3-frontmatter)

**Проверить обязательные поля:**

| Поле | Проверка |
|------|----------|
| `description` | Присутствует, до 1024 символов |
| `standard` | = `specs/.instructions/analysis/design/standard-design.md` |
| `standard-version` | Присутствует, формат `vX.Y` |
| `index` | = `specs/analysis/README.md` |
| `parent` | Присутствует, путь к `discussion.md` в той же папке |
| `children` | Присутствует, список путей или `[]` |
| `status` | Валидный статус: DRAFT, WAITING, RUNNING, REVIEW, DONE, CONFLICT, ROLLING_BACK, REJECTED |
| `milestone` | Присутствует, формат `vX.Y.Z`, совпадает с parent Discussion |
| `docs-synced` | Опционально. Если присутствует — значение `true` (boolean) |

**Управление статусами:** [`chain_status.py`](../../.scripts/chain_status.py) — SSOT-модуль для переходов статусов analysis chain.

**Проверить parent:**
- Файл `discussion.md` существует по указанному пути
- Milestone в Design совпадает с milestone в parent Discussion

### Шаг 3: Заголовок документа

**SSOT:** [standard-design.md § 2](./standard-design.md#2-расположение-и-именование)

- Формат: `# NNNN: {Тема} — Design`
- `NNNN` в заголовке совпадает с `NNNN` в имени папки
- `{Тема}` — до 80 символов, первое слово с заглавной буквы, без точки в конце
- Суффикс `— Design` обязателен

### Шаг 4: Обязательные разделы

**SSOT:** [standard-design.md § 5](./standard-design.md#5-разделы-документа)

| Раздел | Присутствие | Проверка |
|--------|------------|----------|
| `## Резюме` | Всегда (контент обязателен) | Присутствует и не пуст |
| `## Выбор технологий` | Всегда (мин. 1 категория с "Выбрано") | Присутствует между Резюме и SVC-1 |
| `## SVC-N: {Сервис}` | Минимум 1 | Хотя бы одна секция `## SVC-\d+:` |
| `## INT-N: {Взаимодействие}` | Условный (при 2+ сервисах) | Если SVC-N > 1 → хотя бы одна секция `## INT-\d+:` |
| `## Системные тест-сценарии` | Всегда | Присутствует. Контент или заглушка |
| `## Предложения` | Опциональный | Если есть — формат таблицы PROP-N |
| `## Отвергнутые предложения` | Опциональный | Если есть — таблица с колонкой "Причина отклонения" |

**Порядок секций:** Резюме → Выбор технологий → SVC-N → INT-N → Системные тест-сценарии → Предложения → Отвергнутые предложения

### Шаг 5: SVC-N — 9 подсекций и 8:8 маппинг

**SSOT:** [standard-design.md § 5 → SVC-N](./standard-design.md#svc-n-сервис)

Для **каждой** секции SVC-N:

1. **Описание (1-2 абзаца)** — присутствует после заголовка, содержит `**Решение:**`
2. **9 подсекций** — все присутствуют, в правильном порядке:

| # | Подсекция | Проверка |
|---|-----------|----------|
| 1 | `### 1. Назначение` | Контент обязателен (не заглушка) |
| 2 | `### 2. API контракты` | Контент или заглушка |
| 3 | `### 3. Data Model` | Контент или заглушка |
| 4 | `### 4. Потоки` | Контент или заглушка |
| 5 | `### 5. Code Map` | Контент или заглушка. Пути по конвенции монорепо: `src/{svc}/`, без вложенного `src/` ([standard-design.md § 5](./standard-design.md#5-разделы-документа)) |
| 6 | `### 6. Зависимости` | Контент или заглушка |
| 7 | `### 7. Доменная модель` | Контент или заглушка |
| 8 | `### 8. Границы автономии LLM` | Контент или заглушка |
| 9 | `### 9. Решения по реализации` | Контент обязателен (не заглушка) |

3. **Конвенция путей монорепо (§ 5 Code Map)** — если Code Map содержит файловые пути, проверить что нет двойной вложенности `src/{svc}/src/`. Правильно: `src/{svc}/components/`, `src/{svc}/routes/`. Неправильно: `src/{svc}/src/components/`
4. **8:8 маппинг** — названия §§ 1-8 совпадают с §§ 1-8 `specs/docs/{svc}.md` (идентичные строки заголовков)

### Шаг 6: INT-N — блоки взаимодействия

**SSOT:** [standard-design.md § 5 → INT-N](./standard-design.md#int-n-блоки-взаимодействия)

Для **каждой** секции INT-N:

| Элемент | Проверка |
|---------|----------|
| Заголовок | `## INT-N: {Описание}` |
| Метаданные | `**Участники:**` и `**Паттерн:**` присутствуют |
| `### Контракт` | Присутствует, контент обязателен |
| `### Sequence` | Присутствует, содержит блок `mermaid` |

**Перекрёстная проверка:**
- Каждый INT-N упомянут в § 6 «Зависимости» минимум одного SVC-N
- Каждый SVC-N, ссылающийся на INT-N в § 6, является участником этого INT-N

### Шаг 7: Системные тест-сценарии

**SSOT:** [standard-design.md § 5 → STS](./standard-design.md#системные-тест-сценарии)

Если секция содержит контент (не заглушку):
- Формат: таблица с колонками ID, Сценарий, Участники, Тип, Источник
- Каждый `STS-N` ссылается на существующие INT-N в колонке «Источник»
- Тип: одно из `e2e`, `integration`, `load`

### Шаг 8: Нумерация элементов

**SSOT:** [standard-design.md § 5](./standard-design.md#5-разделы-документа) → правила нумерации

- `SVC-N`: последовательная (пропуски допустимы, дублирование — нет), сквозная по документу
- `INT-N`: последовательная (пропуски допустимы, дублирование — нет), сквозная по документу
- `STS-N`: последовательная (пропуски допустимы, дублирование — нет), сквозная по документу

**Проверить:** нет дублирующихся номеров в пределах каждого типа.

### Шаг 9: Delta-формат

**SSOT:** [standard-design.md § 5 → Delta-формат](./standard-design.md#svc-n-сервис)

Для подсекций §§ 2-7 с контентом (не заглушкой):
- Каждый элемент имеет маркер: **ADDED**, **MODIFIED** или **REMOVED**
- ADDED — полное описание нового элемента
- MODIFIED — что именно меняется
- REMOVED — причина удаления

### Шаг 10: Зона ответственности

**SSOT:** [standard-design.md § 1](./standard-design.md#1-назначение)

Документ **НЕ должен** содержать:
- Бизнес-обоснование и требования (→ Discussion)
- Тестовые сценарии конкретного сервиса — unit, integration внутри сервиса (→ Plan Tests)
- Задачи на реализацию (→ Plan Dev)

### Шаг 11: Маркеры и статус

**SSOT:** [standard-design.md § 6](./standard-design.md#6-clarify), [standard-analysis.md § 8](../standard-analysis.md#8-clarify-и-блокирующие-правила)

Если `status` > DRAFT (WAITING, RUNNING, REVIEW, DONE, CONFLICT, ROLLING_BACK, REJECTED):
- Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров
- Нет `⛔ DEPENDENCY BARRIER`

### Шаг 12: Регистрация в README

**SSOT:** [standard-design.md § 2](./standard-design.md#2-расположение-и-именование)

- Запись существует в `specs/analysis/README.md`
- Колонки: `ID`, `Тема`, `Статус`, `Design`, `Milestone`, `Описание`
- Статус в README совпадает со статусом в frontmatter документа

## Чек-лист

### Именование
- [ ] Файл в `specs/analysis/NNNN-{topic}/design.md`
- [ ] Папка: `NNNN-{topic}` (regex `^\d{4}-.+$`)
- [ ] Заголовок: `# NNNN: {Тема} — Design`
- [ ] NNNN в папке = NNNN в заголовке

### Frontmatter
- [ ] `description` — до 1024 символов
- [ ] `standard` = `specs/.instructions/analysis/design/standard-design.md`
- [ ] `standard-version` — формат `vX.Y`
- [ ] `index` = `specs/analysis/README.md`
- [ ] `parent` — путь к Discussion-документу (обязателен)
- [ ] `children` — путь к Plan Tests (или `[]` до создания)
- [ ] `status` — валидный статус
- [ ] `milestone` — совпадает с parent Discussion

### Структура
- [ ] Секция «Резюме» заполнена контентом
- [ ] Секция «Выбор технологий» присутствует между Резюме и SVC-1 (мин. 1 категория с "Выбрано")
- [ ] Порядок секций: Резюме → Выбор технологий → SVC-N → INT-N → STS → Предложения → Отвергнутые предложения
- [ ] Минимум 1 секция SVC-N
- [ ] Каждая SVC-N: описание (1-2 абзаца, содержит `**Решение:**`) + 9 подсекций (§§ 1-9)
- [ ] § 1 «Назначение» — контент (не заглушка)
- [ ] § 9 «Решения по реализации» — контент (не заглушка), минимум 1 решение с WHY
- [ ] §§ 2-8 — контент или заглушка
- [ ] Delta-формат (ADDED/MODIFIED/REMOVED) в §§ 2-7
- [ ] Минимум 1 INT-N (если 2+ сервисов или взаимодействие с существующими)
- [ ] Каждый INT-N: метаданные + Контракт + Sequence (mermaid)
- [ ] «Системные тест-сценарии» — контент или заглушка
- [ ] Все SVC ссылаются на INT через § 6 «Зависимости»
- [ ] Каждый INT-N упомянут в § 6 минимум одного SVC-N
- [ ] Shared-компоненты — решение о размещении
- [ ] § 5 Code Map: пути по конвенции монорепо (`src/{svc}/`, без `src/{svc}/src/`)

### 8:8 маппинг
- [ ] Названия §§ 1-8 SVC-N совпадают с §§ 1-8 {svc}.md
- [ ] Delta корректна относительно текущего AS IS {svc}.md

### Нумерация
- [ ] SVC-N, INT-N, STS-N — сквозная по документу
- [ ] Пропуски допустимы, перенумерация запрещена
- [ ] Нет дублирующихся номеров

### Источники
- [ ] Unified Scan: Discussion + specs/docs/ (5 источников)
- [ ] Parent Discussion прочитан целиком

### Зона ответственности
- [ ] Нет бизнес-обоснований (→ Discussion)
- [ ] Нет тестов конкретного сервиса (→ Plan Tests)
- [ ] Нет задач (→ Plan Dev)

### Маркеры (если статус > DRAFT)
- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] Нет Dependency Barrier

### README
- [ ] Запись в `specs/analysis/README.md`
- [ ] Статус в README = статус в frontmatter

## Типичные ошибки

| Ошибка | Код | Причина | Решение |
|--------|-----|---------|---------|
| Неверное расположение файла | DES001 | Файл не в `specs/analysis/NNNN-{topic}/design.md` | Переместить файл |
| Отсутствует `description` | DES002 | Пустой или отсутствующий frontmatter | Добавить description |
| Неверный `standard` | DES003 | Путь не указывает на standard-design.md | Исправить путь |
| Невалидный `status` | DES004 | Статус не из допустимого списка | Использовать валидный статус |
| Отсутствует `parent` | DES005 | Design обязан иметь parent Discussion | Добавить поле parent |
| Parent Discussion не существует | DES006 | Файл по пути parent не найден | Исправить путь или создать Discussion |
| Milestone не совпадает с parent | DES007 | Milestone Design ≠ milestone Discussion | Синхронизировать milestone |
| Отсутствует «Резюме» | DES008 | Обязательный раздел пропущен | Добавить раздел |
| Нет секции SVC-N | DES009 | Минимум 1 SVC-N обязательна | Добавить SVC-N |
| SVC-N: отсутствует подсекция | DES010 | Не все 9 подсекций §§ 1-9 | Добавить недостающие подсекции |
| SVC-N: § 1 пуст | DES011 | Назначение — контент обязателен | Заполнить § 1 |
| SVC-N: § 9 пуст | DES012 | Решения по реализации — контент обязателен | Добавить минимум 1 решение с WHY |
| SVC-N: нет описания | DES013 | Описание (1-2 абзаца) перед подсекциями отсутствует | Добавить описание |
| SVC-N: нет `**Решение:**` | DES014 | Описание не содержит решение | Добавить решение |
| INT-N отсутствует при 2+ сервисах | DES015 | Условие: при 2+ сервисах мин. 1 INT-N | Добавить INT-N |
| INT-N: нет метаданных | DES016 | Отсутствует `**Участники:**` или `**Паттерн:**` | Добавить метаданные |
| INT-N: нет Контракта | DES017 | Подсекция Контракт пуста или отсутствует | Заполнить контракт |
| INT-N: нет Sequence | DES018 | Подсекция Sequence без mermaid-блока | Добавить sequence diagram |
| Нет «Системные тест-сценарии» | DES019 | Обязательный раздел пропущен | Добавить раздел (контент или заглушка) |
| Дублирование SVC-N | DES020 | Два SVC с одинаковым номером | Исправить нумерацию |
| Дублирование INT-N | DES021 | Два INT с одинаковым номером | Исправить нумерацию |
| Дублирование STS-N | DES022 | Два STS с одинаковым номером | Исправить нумерацию |
| Delta-формат: нет маркеров | DES023 | §§ 2-7 с контентом без ADDED/MODIFIED/REMOVED | Добавить маркеры |
| Маркер при статусе > DRAFT | DES024 | Неразрешённый `[ТРЕБУЕТ УТОЧНЕНИЯ]` | Разрешить маркер или вернуть в DRAFT |
| Dependency Barrier при статусе > DRAFT | DES025 | Неразрешённый барьер | Разрешить барьер или вернуть в DRAFT |
| Зона: бизнес-обоснование | DES026 | Бизнес-контент в Design (→ Discussion) | Перенести в Discussion |
| Зона: unit-тесты | DES027 | Тесты конкретного сервиса (→ Plan Tests) | Перенести в Plan Tests |
| Зона: задачи | DES028 | Задачи на реализацию (→ Plan Dev) | Перенести в Plan Dev |
| NNNN не совпадает | DES029 | Номер в папке ≠ номер в заголовке | Исправить заголовок или имя папки |
| Description слишком длинное | DES030 | Description > 1024 символов | Сократить description |
| INT-N «осиротевший» | DES031 | INT-N не упомянут в § 6 ни одного SVC-N | Добавить ссылку в § 6 или удалить INT-N |
| 8:8 маппинг: названия не совпадают | DES032 | Названия §§ 1-8 отличаются от {svc}.md | Исправить названия подсекций |
| Нет секции «Выбор технологий» | DES033 | Обязательная секция между Резюме и SVC-1 | Добавить секцию с мин. 1 категорией |
| «Выбор технологий» без обоснования | DES034 | Нет заполненного "Выбрано" ни в одной категории | Заполнить "Выбрано" через Clarify |
| Нарушен порядок секций | DES035 | STS не перед Предложениями или Выбор технологий не перед SVC-N | Исправить порядок секций |
| Code Map: двойная вложенность src/ | DES036 | Путь `src/{svc}/src/...` вместо `src/{svc}/...` | Убрать вложенный `src/` — конвенция монорепо |
