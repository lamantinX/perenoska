## 1. Назначение

**Зона ответственности:** ЗАЧЕМ и ЧТО ([Стандарт analysis/ § 2.2](../standard-analysis.md#22-таблица-объектов)).

Дискуссия отвечает на вопросы: **Зачем это нужно? Какие требования?**

**Содержит:**
- Проблему, контекст (текущее состояние и что не работает)
- Требования (естественные предложения)
- User Stories
- Варианты и предложения
- Критерии успеха
- Milestone (в frontmatter, [§ 3](./standard-discussion.md#3-frontmatter))

**НЕ содержит:**
- Технические детали реализации (→ Design)
- Распределение по сервисам (→ Design)
- Архитектурные решения (→ Design)
- Контракты API (→ Design)
- Тестовые сценарии (→ Plan Tests)
- Задачи на реализацию (→ Plan Dev)

**Граничные случаи (Discussion vs Design):**
- Допустимо: "Использовать OAuth2 вместо session-based" — упоминание технологии без привязки к сервисам
- Допустимо: "Рассмотреть варианты: OAuth2, SAML, JWT" — сравнение вариантов
- Недопустимо: "Реализовать OAuth2 в auth-сервисе" — привязка к сервису (→ Design)
- Недопустимо: "auth-сервис отвечает за токены, gateway — за rate limiting" — распределение ответственностей (→ Design)

Пользователь **не определяет** зоны ответственности по сервисам в Discussion. Это задача Design.

**Действие при нарушении зоны:** если в Discussion обнаружен контент из зоны Design (привязка к сервису, распределение ответственностей, контракты API) — LLM спрашивает пользователя через AskUserQuestion: "Найден контент зоны Design: [цитата]. Удалить из Discussion (будет учтён при создании Design) или оставить как контекст?"

**Когда создавать:**
- Новая функциональность (фича, интеграция, миграция)
- Группа связанных багфиксов ([решение #33](../standard-analysis.md#12-решения))
- Cross-cutting concern (NFR: производительность, безопасность — [решение #21](../standard-analysis.md#12-решения))

**Связи:** Discussion не имеет родителя. Дочерний объект — Design (1:1). ([Стандарт analysis/ § 3.1](../standard-analysis.md#31-frontmatter-и-навигация)). Переход Discussion → Design описан в [standard-design.md](/specs/.instructions/analysis/design/standard-design.md).

**После DONE:** Discussion становится архивной записью реализованного решения. Изменения запрещены (только исправление опечаток). При появлении новых требований — создать новую Discussion со ссылкой на DONE-дискуссию в секции "Проблема / Контекст".

**Отмена решения DONE-дискуссии:** если новая Discussion частично или полностью отменяет решение DONE-дискуссии — в секции "Проблема / Контекст" явно указать: "Решение NNNN (DONE) отменяется по причине [X]. Новый подход: [Y]."

## 2. Расположение и именование

**Расположение:** `specs/analysis/NNNN-{topic}/discussion.md`

**Формат имени папки:** `NNNN-{topic}`

| Элемент | Правило |
|---------|---------|
| `NNNN` | Четырёхзначный уникальный автоинкремент начиная с 0001 |
| `{topic}` | Kebab-case, латиница, описательное |

**Имя файла:** `discussion.md` (фиксированное)

**Regex папки:** `^\d{4}-.+$`

**Нумерация:** Уникальный автоинкремент в папке `specs/analysis/`. Следующий номер = максимальный существующий + 1. Все документы цепочки (Design, Plan Tests, Plan Dev) расположены в той же папке `specs/analysis/NNNN-{topic}/`.

**Заголовок документа:** `# NNNN: {Тема}`

**Формат `{Тема}`:** свободный текст, язык проекта (русский), первое слово с заглавной буквы, без точки в конце, до 80 символов. Пример: `# 0001: OAuth2 авторизация`.

**Формат README-таблицы** (`specs/analysis/README.md`):

| ID | Тема | Статус | Design | Milestone | Описание |
|----|------|--------|--------|-----------|----------|
| 0001 | oauth2-authorization | RUNNING | design.md | v1.2.0 | OAuth2 авторизация |
| 0002 | api-latency | DRAFT | — | v1.3.0 | Снижение latency API |

## 3. Frontmatter

**SSOT frontmatter:** [standard-frontmatter.md](/.structure/.instructions/standard-frontmatter.md) ([§ 1 — базовые поля](/.structure/.instructions/standard-frontmatter.md#1-обязательные-поля) + [§ 4 — поля specs](/.structure/.instructions/standard-frontmatter.md#4-дополнительные-поля-для-спецификаций-specs))

| Поле | Значение для Discussion | Обязательность |
|------|------------------------|----------------|
| `description` | Описание дискуссии (до 1024 символов) | Обязательно |
| `standard` | `specs/.instructions/analysis/discussion/standard-discussion.md` | Обязательно |
| `standard-version` | `v1.0` | Обязательно |
| `index` | `specs/analysis/README.md` | Обязательно |
| `parent` | **Нет** (Discussion — корневой объект цепочки) | Не указывается |
| `children` | Путь к Design-документу (1:1) | Обязательно (пустой список `[]` до создания Design; `[design.md]` — после) |
| `status` | Текущий статус ([Стандарт analysis/ § 5](../standard-analysis.md#5-статусы)) | Обязательно |
| `milestone` | Целевой Milestone (определяется при Clarify) | Обязательно |

```yaml
---
description: OAuth2 авторизация — переход с session-based на OAuth2 для масштабирования до 10k RPS.
standard: specs/.instructions/analysis/discussion/standard-discussion.md
standard-version: v1.1
index: specs/analysis/README.md
children:
  - design.md
status: DRAFT
milestone: v1.2.0
---
```

**Milestone:** Целевая версия релиза (строка, например `v1.2.0`). Определяется при Clarify на уровне Discussion и сохраняется в frontmatter. Все дочерние документы цепочки наследуют milestone от Discussion. Один Milestone (например, `v1.2.0`) может быть указан в нескольких Discussions — это означает, что эти дискуссии реализуются в одном релизе.

**Параллельные дискуссии в одном Milestone:** [Стандарт analysis/ § 7.4](../standard-analysis.md#74-параллельные-цепочки) — механизм Planned Changes для обнаружения конфликтов между цепочками.

## 4. Переходы статусов

Happy path — нормальный поток жизненного цикла Discussion ([Стандарт analysis/ § 6](../standard-analysis.md#6-последовательность-статусов)).

**Зона ответственности Discussion:** только переход DRAFT → WAITING (создание и модификация документа). Все последующие переходы управляются на уровне цепочки — дочерними документами, каскадами или обратной связью от кода.

| Текущий статус | Условие перехода | Следующий статус | Кто управляет |
|---|---|---|---|
| **DRAFT** | Пользователь явно одобрил документ через AskUserQuestion. Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]`. Нет Dependency Barrier | **WAITING** | Discussion (этот документ) |
| **WAITING** | Все документы цепочки (Discussion → ... → Plan Dev) в WAITING. Пользователь подтвердил запуск | **RUNNING** | Цепочка ([§ 6.2](../standard-analysis.md#62-waiting-to-running)) |
| **RUNNING** | Обратная связь от кода затронула уровень Discussion ([§ 6.3](../standard-analysis.md#63-running-to-conflict)) | **CONFLICT** | Цепочка (tree-level) |
| **CONFLICT** | LLM исправил документ, пользователь одобрил исправление ([§ 6.4](../standard-analysis.md#64-conflict-to-waiting)) | **WAITING** | Цепочка (top-down) |
| **RUNNING** | Все TASK-N выполнены → REVIEW. Каскад от review.md ([§ 6.5](../standard-analysis.md#65-running-to-review)) | **REVIEW** | Tree-level |
| **REVIEW** | review.md RESOLVED (вердикт READY). Каскад снизу вверх ([§ 6.6](../standard-analysis.md#66-review-to-done)) | **DONE** | Цепочка (bottom-up) |

**Механизм перехода DRAFT → WAITING:** определён в `create-discussion.md` и `modify-discussion.md`. Условие "пользователь одобрил" означает явное подтверждение через AskUserQuestion ("Перевести Discussion в WAITING?"). LLM **не переводит** статус без явного ответа пользователя, даже если все условия выполнены.

Откат и отклонение (ROLLING_BACK, REJECTED): [Стандарт analysis/ § 6.7–6.8](../standard-analysis.md#67-to-rolling_back).

**Управление статусами:** [`chain_status.py`](../../.scripts/chain_status.py) — SSOT-модуль для всех переходов. Вызовы `ChainManager.transition()` — в `create-discussion.md` и `modify-discussion.md`.

## 5. Разделы документа

Дискуссия — **структурированный документ с фиксированными разделами** (см. таблицу ниже). Порядок разделов неизменен. "Гибкость" означает: предложения могут менять фичи и user stories внутри документа до достижения консенсуса — итеративное уточнение контента, не структуры.

| Раздел | Присутствие | Когда заполнять контентом | Заглушка (если нет контента) |
|--------|------------|--------------------------|------------------------------|
| **Проблема / Контекст** | Всегда (контент обязателен) | Всегда | — |
| **Фичи** | Всегда (заглушка допустима) | Обсуждается конкретная функциональность | _Нет выделенных фич._ |
| **User Stories** | Всегда (заглушка допустима) | Фича ориентирована на конечного пользователя | _Нет User Stories._ |
| **Требования** | Всегда (заглушка допустима) | Нужна формализация требований | _Нет формализованных требований._ |
| **Предложения** | Всегда (заглушка допустима) | Есть альтернативные решения или изменения к фичам | _Предложений нет._ |
| **Критерии успеха** | Всегда (контент обязателен) | Всегда | — |

**Все 6 разделов присутствуют всегда.** Разделы "Проблема / Контекст" и "Критерии успеха" всегда заполняются контентом. Остальные разделы заполняются контентом или заглушкой (курсивом). Заглушка означает: раздел рассмотрен, контент не требуется.

**Правила нумерации (F-N, US-N, REQ-N, PROP-N):** Последовательная автоинкрементная. При добавлении — следующий номер = максимальный существующий + 1. При удалении — **не перенумеровывать** (пропуски допустимы, сохраняется ссылочная целостность).

**Механика итеративного уточнения:** Фичи (F-N), User Stories (US-N) и Требования (REQ-N) в статусе DRAFT редактируются **двумя способами**: (1) прямое редактирование пользователем — добавление, удаление, изменение строк; (2) через механизм Предложений (PROP-N) — LLM предлагает изменения, пользователь одобряет/отклоняет. Предложения (PROP-N) описывают изменения в секции "Предложения" с указанием затронутых элементов (→ Влияет на: F-1, US-2). После одобрения пользователем LLM вносит изменения в затронутые секции на месте (обновляется baseline). Старые версии не сохраняются в документе — история через git.

**Группировка (опционально).** Когда дискуссия охватывает несколько логически независимых процессов или объектов (например: авторизация, профиль, настройки), строки в табличных секциях (Фичи, User Stories, Требования, Предложения) можно группировать через `###` подзаголовки внутри секции. Каждая группа — отдельная таблица с тем же набором колонок. Нумерация элементов (F-N, US-N, REQ-N, PROP-N) сквозная по всей секции, не по группе. Группировка необязательна — при малом количестве элементов (до 5–7 строк) допустима одна плоская таблица без подзаголовков. Группировка независима по секциям — названия групп в Фичах и Требованиях могут отличаться, если это обосновано.

### Проблема и контекст

Описание текущей ситуации и причин для изменений. Формат: свободный текст.

Рекомендуется описать:
- Текущее состояние (AS IS): что сейчас работает и как, какие метрики
- Бизнес-контекст или техническое ограничение
- Масштаб проблемы (количественные метрики если есть)

```markdown
## Проблема / Контекст

**Текущее состояние (AS IS):**
Авторизация реализована через session-based подход с хранением в Redis.
Время авторизации: p99 = 800ms при 5k RPS.

**Проблема:**
При масштабировании до 10k RPS Redis становится узким местом.
Кроме того, session не подходит для mobile-клиентов и third-party интеграций.
```

### Фичи

Конкретная функциональность, которую нужно реализовать. **Формат: таблица.** Нумерация: `F-N`.

```markdown
## Фичи

| ID | Название | Описание |
|----|----------|----------|
| F-1 | OAuth2 авторизация для API | Авторизация через access + refresh токены вместо session |
| F-2 | Управление ролями (RBAC) | Назначение ролей пользователям, проверка разрешений на уровне endpoint |
| F-3 | Third-party интеграции | OAuth2 client credentials flow для внешних сервисов |
```

**С группировкой** (опционально, при множестве логических блоков):

```markdown
## Фичи

### Авторизация

| ID | Название | Описание |
|----|----------|----------|
| F-1 | OAuth2 авторизация для API | Авторизация через access + refresh токены вместо session |

### Управление доступом

| ID | Название | Описание |
|----|----------|----------|
| F-2 | Управление ролями (RBAC) | Назначение ролей пользователям, проверка разрешений |
| F-3 | Third-party интеграции | OAuth2 client credentials flow для внешних сервисов |
```

### User Stories

Описание функциональности с точки зрения пользователя. **Формат: таблица.** Нумерация: `US-N`.

```markdown
## User Stories

| ID | Роль | Действие | Цель |
|----|------|----------|------|
| US-1 | API-клиент | Получить OAuth2 токен через POST /auth/token | Авторизовывать запросы без хранения session |
| US-2 | Администратор | Назначать роли пользователям через UI | Контролировать доступ к функциям системы |
```

### Требования

Формализованные требования в виде естественных предложений. **Формат: таблица.** Нумерация: `REQ-N`. Колонка "Связь" — трассируемость к фичам и user stories. На следующих этапах (Design) одно требование может разделиться на несколько задач по разным сервисам.

```markdown
## Требования

| ID | Связь | Требование |
|----|-------|------------|
| REQ-1 | F-1, US-1 | Пользователь с валидными credentials отправляет POST /auth/token с grant_type=password — ответ 200 OK с access_token и refresh_token |
| REQ-2 | F-1 | Валидный refresh_token отправляется на POST /auth/token с grant_type=refresh_token — новый access_token, прежний refresh_token инвалидируется |
| REQ-3 | F-1 | Неавторизованный пользователь отправляет более 100 запросов за минуту — ответ 429 Too Many Requests |
```

### Предложения

Варианты решений, изменения к фичам и user stories. **Формат: таблица.** Нумерация: `PROP-N`. Предложения могут **менять** ранее описанные фичи и user stories — итеративное уточнение.

**Жизненный цикл предложений:** Триггер генерации PROP определён в `create-discussion.md`. Стандарт описывает только формат и жизненный цикл.

1. LLM генерирует предложения и записывает в секцию "Предложения"
2. Пользователь прорабатывает каждое (принять/отклонить)
3. Принятые — LLM применяет изменения к затронутым секциям
4. После того как пользователь принял/отклонил **ВСЕ** PROP в секции — LLM удаляет все записи PROP (независимо от решения) и заменяет секцию заглушкой

Финальное состояние секции:
- _Все предложения обработаны._ — были предложения, все обработаны
- _Предложений нет._ — LLM не выявил альтернатив

```markdown
## Предложения

| ID | Предложение | Влияет на |
|----|-------------|-----------|
| PROP-1 | ES256 вместо RS256 для JWT — быстрее на целевом железе, сопоставимая безопасность | F-1 |
| PROP-2 | Убрать F-3 из текущего Milestone — требует отдельного OAuth2-сервера, вынести в v1.3.0 | F-3, критерии |
```

### Отвергнутые предложения

Опциональная секция. Содержит PROP, отклонённые пользователем при обработке предложений. **Формат: таблица.** Назначение: передача контекста между уровнями цепочки (Discussion → Design).

PROP, отклонённые на уровне Discussion как "конкретика для Design", записываются с пометкой "→ Учесть в Design" — чтобы Design-агент не забыл их рассмотреть. Секция сохраняется в финальном документе (не удаляется при обработке PROP).

```markdown
## Отвергнутые предложения

| ID | Предложение | Причина отклонения |
|----|-------------|-------------------|
| PROP-3 | Установить rate limit 50 req/s вместо 100 | Конкретика для Design → Учесть в Design |
```

### Критерии успеха

Измеримые критерии завершения. Обязательный раздел. **Формат: bullet list.**

Допустимы два типа критериев: (1) числовые — с конкретной метрикой (p99 < 100ms); (2) бинарные (да/нет) — если факт проверяем (например: "Zero-downtime миграция: все сессии мигрированы без прерывания"). Недопустимы субъективные формулировки ("работает лучше", "стало быстрее").

```markdown
## Критерии успеха

- Время авторизации < 100ms (p99) при 10k RPS
- Zero-downtime миграция с session-based на OAuth2
- Все существующие API-клиенты продолжают работать через compatibility layer
- 100% покрытие acceptance-тестами (REQ-1 — REQ-N)
```

## 6. Clarify

**SSOT:** [Стандарт analysis/ § 8](../standard-analysis.md#8-clarify-и-блокирующие-правила) — полные правила Clarify, маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`, Dependency Barrier.

**Паттерн объекта:** [Стандарт analysis/ § 2.4 "Общий паттерн объекта"](../standard-analysis.md#24-общий-паттерн-объекта) — цикл CLARIFY → GENERATE → VALIDATE → REVIEW.

На уровне Discussion LLM уточняет через AskUserQuestion:

| Что уточняется | Пример вопроса |
|----------------|---------------|
| Проблема и scope | "Какую конкретно проблему решаем? Это только авторизация или также аутентификация?" |
| Требования | "Какой SLA по времени авторизации? Есть ли требования к offline-режиму?" |
| Целевая аудитория | "Кто пользователи? Только внутренние сервисы или также external API consumers?" |
| Критерии успеха | "Как измерим успешность? Какие метрики?" |
| Milestone | "В каком Milestone планируем? Есть ли дедлайн?" |
| Границы (scope) | "Что явно НЕ входит в эту дискуссию?" |

**Маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]`:** LLM ставит маркеры на оставшиеся неясности **в обоих случаях** — после проведения Clarify и при `--auto-clarify`. Пока в документе есть хотя бы один неразрешённый маркер, перевод в статус WAITING **невозможен**.

Воркфлоу создания (`create-discussion.md`) завершается переводом в WAITING. Поэтому все маркеры должны быть разрешены до завершения создания — через AskUserQuestion, не откладывая на modify.

**Пропуск Clarify:** Пользователь может указать `--auto-clarify` — LLM пропускает Clarify, генерирует на основе своего понимания и ставит маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` на все неясности. Шаг REVIEW остаётся обязательным.
