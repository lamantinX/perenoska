## Принципы

> **Discussion — точка входа.** Каждая цепочка SDD начинается с дискуссии. Без дискуссии — нет Design, Plan Tests, Plan Dev.

> **Зона: ЗАЧЕМ и ЧТО.** Discussion описывает проблему и требования. Технические детали, сервисы, API — на следующих уровнях.

> **Файл до Clarify.** Сначала создать файл из шаблона и заполнить базовые поля — затем уточнять содержание. Это обеспечивает resumability при прерывании LLM.

> **Clarify до генерации.** LLM уточняет неясности через AskUserQuestion перед генерацией контента разделов. Пропуск — только по `--auto-clarify`.

> **DRAFT → WAITING — единственный переход этого документа.** Все последующие переходы управляются на уровне цепочки.

> **Документ без маркеров.** Воркфлоу создания завершается переводом в WAITING. Все маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` должны быть разрешены до ревью. LLM разрешает маркеры через AskUserQuestion — не ставит маркер и не идёт дальше.

## Шаги

### Шаг 1: Определить тип дискуссии

**SSOT:** [standard-discussion.md § 1](./standard-discussion.md#1-назначение)

Определить тип на основе запроса пользователя:

| Тип | Когда | Особенности |
|-----|-------|------------|
| Новая функциональность | Фича, интеграция, миграция | Все разделы (Фичи, User Stories, Требования) |
| Группа багфиксов | Связанные баги | Минимум: Проблема + Критерии. Фичи/User Stories обычно не нужны |
| Cross-cutting concern (NFR) | Производительность, безопасность, масштабируемость | Требования + Критерии, фичи опционально |

### Шаг 2–3: Определить номер и создать папку с файлом

**SSOT:** [standard-discussion.md § 2](./standard-discussion.md#2-расположение-и-именование), [§ 7](./standard-discussion.md#7-шаблон)

1. Проверить существующие папки в `specs/analysis/`
2. Следующий номер = максимальный существующий NNNN + 1
3. Сформировать имя папки: `NNNN-{topic}` (topic — kebab-case, латиница)
4. Создать папку `specs/analysis/NNNN-{topic}/`
5. Скопировать шаблон из [standard-discussion.md § 7](./standard-discussion.md#7-шаблон) и создать файл `discussion.md`

**Если `specs/analysis/` не существует:** вызвать `/structure-create specs/analysis` (-> [core.md](/.claude/rules/core.md) правило скиллов).

### Шаг 4: Заполнить frontmatter

**SSOT:** [standard-discussion.md § 3](./standard-discussion.md#3-frontmatter)

Заполнить поля, известные до Clarify:

| Поле | Значение |
|------|----------|
| `description` | Краткое описание из запроса пользователя (до 1024 символов) |
| `standard` | `specs/.instructions/analysis/discussion/standard-discussion.md` |
| `standard-version` | `v1.1` |
| `index` | `specs/analysis/README.md` |
| `children` | `[]` (Design ещё не создан) |
| `status` | `DRAFT` |
| `milestone` | `[ТРЕБУЕТ УТОЧНЕНИЯ: milestone?]` (уточняется на шаге 5) |

### Шаг 5: Clarify

**SSOT:** [standard-discussion.md § 6](./standard-discussion.md#6-clarify), [Стандарт analysis/ § 8](../standard-analysis.md#8-clarify-и-блокирующие-правила)

**Если `--auto-clarify` НЕ указан:**

LLM уточняет через AskUserQuestion по таблице из [standard-discussion.md § 6](./standard-discussion.md#6-clarify):

1. Проблема и scope
2. Требования
3. Целевая аудитория
4. Критерии успеха
5. Milestone
6. Границы (scope)

**Если `--auto-clarify` указан:**

LLM пропускает Clarify, генерирует документ на основе своего понимания и ставит маркеры `[ТРЕБУЕТ УТОЧНЕНИЯ]` на все неясности.

**После Clarify:** Обновить frontmatter — `description` (уточнённое) и `milestone` (из ответов).

### Шаг 6: Заполнить разделы

**SSOT:** [standard-discussion.md § 5](./standard-discussion.md#5-разделы-документа)

На основе результатов Clarify (шаг 5) и типа дискуссии (шаг 1) заполнить **все 6 разделов**:

1. **Проблема / Контекст** — заполнить AS IS и Проблема (контент обязателен)
2. **Фичи** — контент или заглушка: _Нет выделенных фич._
3. **User Stories** — контент или заглушка: _Нет User Stories._
4. **Требования** — контент (естественные предложения в таблице) или заглушка: _Нет формализованных требований._
5. **Предложения** — контент или заглушка: _Предложений нет._
6. **Критерии успеха** — измеримые метрики (контент обязателен)

**Маркеры:** Если информации недостаточно — ставить `[ТРЕБУЕТ УТОЧНЕНИЯ: вопрос]`. Не угадывать.

**Dependency Barrier:** Если генерация следующей секции зависит от неразрешённого маркера — остановить генерацию контента и перейти к перечислению оставшихся секций ([Стандарт analysis/ § 8.3](../standard-analysis.md#83-dependency-barrier)).

**Разрешение маркеров (обязательно перед продолжением):**

После заполнения всех разделов:

1. Проверить документ на наличие `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров
2. Если маркеры есть — для каждого маркера уточнить через AskUserQuestion
3. Заменить маркеры на ответы пользователя
4. Если был Dependency Barrier — продолжить генерацию оставшихся секций
5. Повторять пока маркеров = 0

**Запрещено:** переходить к генерации предложений с неразрешёнными маркерами.

**Генерация и проработка предложений:**

После разрешения всех маркеров:

1. LLM анализирует готовый документ и формулирует предложения (PROP-N) — альтернативы, улучшения, уточнения к фичам/требованиям/критериям
2. Записать предложения в секцию "Предложения" с указанием `→ Влияет на:`
3. **Классификация предложений** (перед обработкой):
   Для каждого PROP определить тип:
   | Тип | Критерий | Действие |
   |-----|----------|----------|
   | **Формальный** | Очевидное уточнение, идемпотентность, safety, единственный разумный вариант | Принять автоматически |
   | **Решение** | Есть несколько разумных вариантов, требуется выбор пользователя | AskUserQuestion с рекомендацией |
   - Формальные PROP: применить все, показать пользователю список "Автоматически приняты: PROP-X, PROP-Y, PROP-Z"
   - Решения: обрабатывать пачками по 4 через AskUserQuestion, рекомендованный вариант первым + "(рекомендуется)"
4. Обработка предложений:
   - PROP-N типа "Решение" обрабатываются пачками по 4 через AskUserQuestion (лимит инструмента — 4 вопроса)
   - Каждый AskUserQuestion содержит до 4 PROP с вариантами (принять / отклонить), рекомендованный вариант первым + "(рекомендуется)"
   - Если PROP больше 4 — следующая пачка после ответа на предыдущую
   - **ВАЖНО: обрабатывать PROP по одному через Edit, НЕ заменять секцию «Предложения» целиком.**
     Для каждого PROP после получения ответа:
     - Принятый PROP:
       (a) применить основную правку к содержимому документа (F-N, US-N, REQ-N и т.д.)
       (b) Grep по документу на все связанные термины/секции из правки
       (c) распространить изменение на все затронутые места (Фичи, User Stories, Требования, Критерии успеха, Предложения)
       (d) удалить строку PROP из секции «Предложения»
     - Отклонённый PROP: (a) удалить строку PROP из секции «Предложения», (b) добавить в секцию «Отвергнутые предложения» (создать секцию, если не существует). Если PROP содержит конкретику уровня Design (интервалы, лимиты, значения) — пометить «→ Учесть в Design»
   - После обработки всех PROP: если секция «Предложения» пуста — оставить секцию с текстом `_Все предложения обработаны._`. Секция «Отвергнутые предложения» сохраняется.
7. Если предложений нет — оставить: _Предложений нет._

**Запрещено:** переходить к шагу 7 с необработанными предложениями.

### Шаг 7: Регистрация в README

Добавить запись в `specs/analysis/README.md`:

**7.1. Таблица Цепочки:**

```markdown
| NNNN | {topic} | DRAFT | — | vX.Y.Z | {Описание} |
```

**7.2. Дерево** — добавить папку цепочки:

```markdown
├── NNNN-{topic}/               # {Описание} (vX.Y.Z)
```

### Шаг 8: Валидация

```bash
python specs/.instructions/.scripts/validate-analysis-discussion.py specs/analysis/NNNN-{topic}/discussion.md
```

**Если скрипт недоступен:** пройти чек-лист из [validation-discussion.md](./validation-discussion.md).

Исправить ошибки до продолжения.

### Шаг 9: Ревью пользователем

**Перед вопросом:** проверить что маркеров = 0 и валидация пройдена. Если нет — вернуться к шагу 6.

**БЛОКИРУЮЩЕЕ.** AskUserQuestion: "Документ дискуссии готов. Всё корректно?"

| Ответ | Действие |
|-------|----------|
| Запустить ревью агентом (Recommended) | Запустить агента [discussion-reviewer](/.claude/agents/discussion-reviewer/AGENT.md) с путём к документу. Агент запишет PROP-N в секцию "Предложения". После возврата агента — **классифицировать PROP-N** (см. ниже), затем обработать (шаг 6, подраздел "Генерация и проработка предложений") → вернуться к шагу 8 |
| Да, всё корректно | Перевести DRAFT → WAITING через `chain_status.py` (см. ниже) → шаг 10 |
| Нет, нужны правки | Внести изменения → продолжить с шага 8 |

**Переход DRAFT → WAITING** — через модуль `chain_status.py` (SSOT статусов):

```python
from chain_status import ChainManager
mgr = ChainManager("NNNN")
result = mgr.transition(to="WAITING", document="discussion")
# Модуль автоматически: обновляет frontmatter + README dashboard
```

- `result.side_effects` — список побочных действий (для discussion обычно пуст)
- `result.auto_propose` — предложение следующего шага (`/design-create NNNN`)

### Шаг 10: Отчёт о выполнении

Вывести отчёт:

```
## Отчёт о создании дискуссии

Создана дискуссия: `specs/analysis/NNNN-{topic}/discussion.md`

Описание: {description}

Milestone: {vX.Y.Z}

Разделы:
- {список заполненных разделов}

Статус: DRAFT → WAITING

Следующий шаг: Создать Design

Валидация: пройдена
```

## Примеры

### Создание дискуссии для новой фичи

```
Пользователь: "Нужна OAuth2 авторизация вместо session-based"

1. Тип: новая функциональность
2. Папка: specs/analysis/0001-oauth2-authorization/discussion.md
3. Файл создан из шаблона
4. Frontmatter: status=DRAFT, milestone=[ТРЕБУЕТ УТОЧНЕНИЯ]
5. Clarify: scope, SLA, audience, milestone → v1.2.0
6. Разделы: Проблема, Фичи (F-1..F-3), User Stories, Требования (REQ-1..REQ-3), Критерии
   → Проверка маркеров: 0 → OK
   → Предложения: PROP-1 "ES256 вместо RS256" → принят → F-1 обновлён
   → Все предложения обработаны
7-8. README + Валидация
9. Ревью: "Да, всё корректно" → DRAFT → WAITING
10. Отчёт
```

### Создание дискуссии для группы багфиксов

```
Пользователь: "Исправить race conditions в кэше — 3 бага"

1. Тип: группа багфиксов
2. Папка: specs/analysis/0005-cache-race-conditions/discussion.md
3. Файл создан из шаблона
4. Frontmatter: status=DRAFT
5. Clarify: какие баги, метрики, приоритет → v1.1.1
6. Разделы: Проблема (BUG-101, BUG-115, BUG-122), Критерии
   + заглушки: Фичи, User Stories, Требования, Предложения
   → Проверка маркеров: 0 → OK
7-8. README + Валидация
9. Ревью: "Да, всё корректно" → DRAFT → WAITING
10. Отчёт
```

### Создание с --auto-clarify

```
Пользователь: "Создай дискуссию про снижение latency API, --auto-clarify"

1. Тип: NFR
2. Папка: specs/analysis/0008-api-latency-reduction/discussion.md
3. Файл создан из шаблона
4. Frontmatter: status=DRAFT, milestone=[ТРЕБУЕТ УТОЧНЕНИЯ]
5. Clarify пропущен — маркеры на все неясности
6. Разделы: Проблема, Требования, Критерии + маркеры [ТРЕБУЕТ УТОЧНЕНИЯ]
   → Разрешение маркеров: AskUserQuestion по каждому → замена → 0 маркеров
   → Предложения: нет → "Предложений нет"
7-8. README + Валидация
9. Ревью: "Да, всё корректно" → DRAFT → WAITING
10. Отчёт
```
