## Когда валидировать

После любого изменения в документе плана тестов:

- Создание → [create-plan-test.md](./create-plan-test.md)
- Изменение → [modify-plan-test.md](./modify-plan-test.md)
- Перед переводом DRAFT → WAITING — проверить: 0 маркеров, 0 Dependency Barrier
- При code review

## Шаги

### Шаг 0: Автоматическая валидация

```bash
python specs/.instructions/.scripts/validate-analysis-plan-test.py {путь}
```

Скрипт проверяет все правила PT001-PT031. Если валидация пройдена — **готово**, шаги 1-12 не нужны.

**Если скрипт недоступен** — выполнить шаги 1-12 вручную.

### Шаг 1: Расположение и именование

**SSOT:** [standard-plan-test.md § 2](./standard-plan-test.md#2-расположение-и-именование)

- Файл расположен в `specs/analysis/NNNN-{topic}/plan-test.md`
- Папка соответствует regex: `^\d{4}-.+$`
- `NNNN` — четырёхзначный номер (0001, 0002, ...)
- `{topic}` — kebab-case, латиница
- Имя файла — `plan-test.md` (фиксированное)

### Шаг 2: Frontmatter

**SSOT:** [standard-plan-test.md § 3](./standard-plan-test.md#3-frontmatter)

**Проверить обязательные поля:**

| Поле | Проверка |
|------|----------|
| `description` | Присутствует, до 1024 символов |
| `standard` | = `specs/.instructions/analysis/plan-test/standard-plan-test.md` |
| `standard-version` | Присутствует, формат `vX.Y` |
| `index` | = `specs/analysis/README.md` |
| `parent` | Присутствует, путь к `design.md` в той же папке |
| `children` | Присутствует, список путей или `[]` |
| `status` | Валидный статус: DRAFT, WAITING, RUNNING, REVIEW, DONE, CONFLICT, ROLLING_BACK, REJECTED |
| `milestone` | Присутствует, формат `vX.Y.Z`, совпадает с parent Discussion |

**Управление статусами:** [`chain_status.py`](../../.scripts/chain_status.py) — SSOT-модуль для переходов статусов analysis chain.

**Проверить parent:**
- Файл `design.md` существует по указанному пути
- Design в статусе WAITING (или выше)
- Milestone в Plan Tests совпадает с milestone в parent Discussion

### Шаг 3: Заголовок документа

**SSOT:** [standard-plan-test.md § 2](./standard-plan-test.md#2-расположение-и-именование)

- Формат: `# NNNN: {Тема} — Plan Tests`
- `NNNN` в заголовке совпадает с `NNNN` в имени папки
- `{Тема}` — до 80 символов, первое слово с заглавной буквы, без точки в конце
- Суффикс `— Plan Tests` обязателен

### Шаг 4: Обязательные разделы

**SSOT:** [standard-plan-test.md § 5](./standard-plan-test.md#5-разделы-документа)

| Раздел | Присутствие | Проверка |
|--------|------------|----------|
| `## Резюме` | Всегда (контент обязателен) | Присутствует и не пуст |
| `## SVC-N: {сервис}` (per-service) | Минимум 1 | Хотя бы одна секция per-service |
| ├── `### Acceptance-сценарии` | Всегда (контент обязателен) | Таблица TC-N с 5 колонками |
| └── `### Тестовые данные` | Всегда | Контент или заглушка |
| `## Системные тест-сценарии` | Всегда | Контент или заглушка |
| `## Матрица покрытия` | Всегда (контент обязателен) | Таблица Источник → TC |
| `## Блоки тестирования` | Всегда | Таблица BLOCK-N или заглушка |
| `## Предложения` | Всегда | PROP-N таблица или заглушка |
| `## Отвергнутые предложения` | Всегда | PROP-N таблица или заглушка |

### Шаг 5: Per-service разделы и TC-N

**SSOT:** [standard-plan-test.md § 5](./standard-plan-test.md#5-разделы-документа)

Для **каждого** per-service раздела:

1. **Acceptance-сценарии** — таблица TC-N:

| Колонка | Проверка |
|---------|----------|
| ID | Формат `TC-N` (число) |
| Описание | Естественное предложение (НЕ Given/When/Then), без «должен»/«следует» |
| Тип | Одно из: `unit`, `integration`, `e2e`, `load`, `smoke` |
| Источник | Обязательно ≥ 1 из: REQ-N, STS-N, SVC-N § K, INT-N |
| Данные | Ссылка на fixture из секции «Тестовые данные» или `—` |

2. **Тестовые данные** — таблица fixtures:

| Колонка | Проверка |
|---------|----------|
| Fixture | Уникальное имя (snake_case) |
| Описание | Краткое описание |
| Поля | Ключевые поля |

**Или заглушка:** `*Использует общие тестовые данные.*`

3. **Перекрёстная проверка:** каждый fixture, упомянутый в колонке «Данные» TC-N, существует в секции «Тестовые данные» этого или системного раздела.

### Шаг 6: Системные тест-сценарии

**SSOT:** [standard-plan-test.md § 5 → Системные тест-сценарии](./standard-plan-test.md#системные-тест-сценарии)

Если секция содержит контент (не заглушку):
- Формат: таблица TC-N с 5 колонками (ID, Описание, Тип, Источник, Данные)
- Каждый TC-N ссылается на STS-N в колонке «Источник»
- Каждый STS-N из parent Design покрыт хотя бы одним TC-N

**Заглушка допустима** если parent Design не содержит STS-N.

### Шаг 7: Матрица покрытия

**SSOT:** [standard-plan-test.md § 5 → Матрица покрытия](./standard-plan-test.md#матрица-покрытия)

| Проверка | Описание |
|----------|----------|
| Каждый REQ-N из Discussion покрыт ≥ 1 TC-N | Иначе маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]` |
| Каждый STS-N из Design покрыт ≥ 1 TC-N | Иначе маркер `[ТРЕБУЕТ УТОЧНЕНИЯ]` |
| Все TC-N из таблиц присутствуют в матрице | Нет «потерянных» TC-N |
| TC-N, покрывающий несколько источников | Присутствует в каждой строке источника |

### Шаг 8: Нумерация TC-N

**SSOT:** [standard-plan-test.md § 5](./standard-plan-test.md#5-разделы-документа) → правила нумерации

- `TC-N`: сквозная по всему документу (не по сервису)
- Порядок: per-service (в порядке Design) → системные
- Пропуски допустимы, перенумерация запрещена
- Нет дублирующихся номеров

### Шаг 9: Блоки тестирования (BLOCK-N)

**SSOT:** [standard-plan-test.md § 5 → Блоки тестирования](./standard-plan-test.md#блоки-тестирования)

Если секция `## Блоки тестирования` содержит таблицу (не заглушку):

| Проверка | Описание |
|----------|----------|
| Формат таблицы | 4 колонки: BLOCK, TC, Сервисы, Dev BLOCK |
| Покрытие | Каждый TC-N принадлежит ровно одному BLOCK-N |
| Нумерация | Сквозная: BLOCK-1, BLOCK-2, ... Совпадает с plan-dev |
| Dev BLOCK | Каждый BLOCK-N ссылается на соответствующий блок в plan-dev |
| Системные TC | TC с типом e2e/load выделены в отдельный BLOCK последней волны |

### Шаг 10: Зона ответственности

**SSOT:** [standard-plan-test.md § 1](./standard-plan-test.md#1-назначение)

Документ **НЕ должен** содержать:
- Бизнес-обоснование и требования (→ Discussion)
- Распределение ответственностей, контракты API (→ Design)
- Реализацию тестов — конкретный код, фреймворки (→ Plan Dev / разработка)
- Задачи на реализацию (→ Plan Dev)

### Шаг 11: Маркеры и статус

**SSOT:** [standard-plan-test.md § 6](./standard-plan-test.md#6-clarify), [standard-analysis.md § 8](../standard-analysis.md#8-clarify-и-блокирующие-правила)

Если `status` > DRAFT (WAITING, RUNNING, REVIEW, DONE, CONFLICT, ROLLING_BACK, REJECTED):
- Нет неразрешённых `[ТРЕБУЕТ УТОЧНЕНИЯ]` маркеров
- Нет `⛔ DEPENDENCY BARRIER`

### Шаг 12: Регистрация в README

**SSOT:** [standard-plan-test.md § 2](./standard-plan-test.md#2-расположение-и-именование)

- Запись существует в `specs/analysis/README.md`
- Статус в README совпадает со статусом в frontmatter документа

## Чек-лист

### Именование
- [ ] Файл в `specs/analysis/NNNN-{topic}/plan-test.md`
- [ ] Папка: `NNNN-{topic}` (regex `^\d{4}-.+$`)
- [ ] Заголовок: `# NNNN: {Тема} — Plan Tests`
- [ ] NNNN в папке = NNNN в заголовке

### Frontmatter
- [ ] `description` — до 1024 символов
- [ ] `standard` = `specs/.instructions/analysis/plan-test/standard-plan-test.md`
- [ ] `standard-version` — формат `vX.Y`
- [ ] `index` = `specs/analysis/README.md`
- [ ] `parent` — путь к Design-документу (обязателен)
- [ ] `children` — путь к Plan Dev (или `[]` до создания)
- [ ] `status` — валидный статус
- [ ] `milestone` — совпадает с parent Discussion

### Структура
- [ ] Секция «Резюме» заполнена контентом
- [ ] Минимум 1 per-service раздел (h2)
- [ ] Каждый per-service раздел: Acceptance-сценарии (h3) + Тестовые данные (h3)
- [ ] Acceptance-сценарии: таблица TC-N с 5 колонками (ID, Описание, Тип, Источник, Данные)
- [ ] Секция «Системные тест-сценарии» — контент или заглушка
- [ ] Секция «Матрица покрытия» заполнена контентом
- [ ] Секция «Блоки тестирования» — таблица или заглушка
- [ ] Секция «Предложения» — PROP-N таблица или заглушка
- [ ] Секция «Отвергнутые предложения» — PROP-N таблица или заглушка

### Формат TC-N
- [ ] Описание — естественное предложение (НЕ Given/When/Then)
- [ ] В колонке Описание — без «должен», «следует»
- [ ] Тип — один из: unit, integration, e2e, load, smoke
- [ ] Источник — обязательно ≥ 1 из: REQ-N, STS-N, SVC-N § K, INT-N
- [ ] Данные — ссылка на fixture или `—`

### Нумерация
- [ ] TC-N — сквозная по всему документу (не по сервису)
- [ ] Пропуски допустимы, перенумерация запрещена
- [ ] Нет дублирующихся номеров

### Покрытие
- [ ] Каждый REQ-N из Discussion покрыт ≥ 1 TC-N в матрице
- [ ] Каждый STS-N из Design покрыт ≥ 1 TC-N в матрице
- [ ] Нет REQ-N/STS-N без покрытия (иначе маркер)
- [ ] TC-N покрывающий несколько источников — в каждой строке

### Per-service разделы
- [ ] Каждый SVC-N из Design имеет per-service раздел
- [ ] Порядок разделов совпадает с Design (Основной → Вторичный → Новый)
- [ ] Fixtures в колонке «Данные» существуют в секции «Тестовые данные»

### Источники
- [ ] Parent Design прочитан целиком (SVC-N, INT-N, STS-N)
- [ ] Discussion прочитана (REQ-N)
- [ ] `specs/docs/{svc}.md` прочитаны (AS IS)
- [ ] `specs/docs/.system/testing.md` прочитан (стратегия)

### Блоки тестирования (BLOCK-N)
- [ ] Каждый TC-N принадлежит ровно одному BLOCK-N
- [ ] Нумерация BLOCK-N сквозная, совпадает с plan-dev
- [ ] Системные TC (e2e/load) в отдельном BLOCK последней волны
- [ ] Колонка «Dev BLOCK» указывает на соответствующий блок в plan-dev

### Зона ответственности
- [ ] Нет бизнес-обоснований (→ Discussion)
- [ ] Нет распределения ответственностей (→ Design)
- [ ] Нет контрактов API (→ Design)
- [ ] Нет задач на реализацию (→ Plan Dev)
- [ ] Нет реализации тестов — кода, фреймворков (→ Plan Dev / разработка)

### Маркеры (если статус > DRAFT)
- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]`
- [ ] Нет Dependency Barrier

### README
- [ ] Запись в `specs/analysis/README.md`
- [ ] Статус в README = статус в frontmatter

## Типичные ошибки

| Ошибка | Код | Причина | Решение |
|--------|-----|---------|---------|
| Неверное расположение файла | PT001 | Файл не в `specs/analysis/NNNN-{topic}/plan-test.md` | Переместить файл |
| Отсутствует `description` | PT002 | Пустой или отсутствующий frontmatter | Добавить description |
| Неверный `standard` | PT003 | Путь не указывает на standard-plan-test.md | Исправить путь |
| Невалидный `status` | PT004 | Статус не из допустимого списка | Использовать валидный статус |
| Отсутствует `parent` | PT005 | Plan Tests обязан иметь parent Design | Добавить поле parent |
| Parent Design не существует | PT006 | Файл по пути parent не найден | Исправить путь или создать Design |
| Milestone не совпадает с parent | PT007 | Milestone Plan Tests ≠ milestone Discussion | Синхронизировать milestone |
| Отсутствует «Резюме» | PT008 | Обязательный раздел пропущен | Добавить раздел |
| Нет per-service раздела | PT009 | Минимум 1 per-service раздел | Добавить per-service раздел |
| Нет «Acceptance-сценарии» | PT010 | Подсекция h3 пропущена в per-service | Добавить подсекцию |
| Нет «Тестовые данные» | PT011 | Подсекция h3 пропущена в per-service | Добавить подсекцию (контент или заглушку) |
| TC-N: формат Given/When/Then | PT012 | Описание в формате G/W/T вместо естественного предложения | Переписать как естественное предложение |
| TC-N: «должен»/«следует» | PT013 | В колонке Описание модальные глаголы | Переформулировать как констатацию факта |
| TC-N: невалидный тип | PT014 | Тип не из списка: unit, integration, e2e, load, smoke | Исправить тип |
| TC-N: нет источника | PT015 | Колонка Источник без обязательной ссылки (REQ-N, STS-N, SVC-N или INT-N) | Добавить ≥ 1 источник |
| TC-N: fixture не найден | PT016 | Ссылка на несуществующий fixture | Добавить fixture в «Тестовые данные» |
| Дублирование TC-N | PT017 | Два TC с одинаковым номером | Исправить нумерацию |
| Нет «Системные тест-сценарии» | PT018 | Обязательный раздел пропущен | Добавить раздел (контент или заглушку) |
| Нет «Матрица покрытия» | PT019 | Обязательный раздел пропущен | Добавить раздел |
| REQ-N без покрытия | PT020 | REQ-N из Discussion не покрыт TC-N | Добавить TC-N или маркер |
| STS-N без покрытия | PT021 | STS-N из Design не покрыт TC-N | Добавить TC-N или маркер |
| Маркер при статусе > DRAFT | PT022 | Неразрешённый `[ТРЕБУЕТ УТОЧНЕНИЯ]` | Разрешить маркер или вернуть в DRAFT |
| Dependency Barrier при статусе > DRAFT | PT023 | Неразрешённый барьер | Разрешить барьер или вернуть в DRAFT |
| NNNN не совпадает | PT024 | Номер в папке ≠ номер в заголовке | Исправить заголовок или имя папки |
| Description слишком длинное | PT025 | Description > 1024 символов | Сократить description |
| Нет «Блоки тестирования» | PT026 | Обязательный раздел пропущен | Добавить раздел (таблицу или заглушку) |
| TC-N без BLOCK-N | PT027 | TC не принадлежит ни одному блоку | Добавить в BLOCK-N |
| BLOCK-N не совпадает с plan-dev | PT028 | Нумерация BLOCK-N расходится с plan-dev | Синхронизировать нумерацию |
| Системные TC не в отдельном BLOCK | PT029 | e2e/load TC в том же блоке, что per-service | Выделить в отдельный BLOCK последней волны |
| Нет «Предложения» | PT030 | Обязательный раздел пропущен | Добавить раздел (PROP-N таблицу или заглушку) |
| Нет «Отвергнутые предложения» | PT031 | Обязательный раздел пропущен | Добавить раздел (PROP-N таблицу или заглушку) |
