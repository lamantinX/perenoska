## 1. Назначение

**Зона ответственности:** WHAT TASKS ([Стандарт analysis/ § 2.2](../standard-analysis.md#22-таблица-объектов)).

Plan Dev отвечает на вопрос: **Какие задачи реализовать?**

**Содержит:**
- Per-service разделы: задачи (TASK-N) с метаданными (сложность, приоритет, зависимости, ссылки на TC-N)
- Подзадачи (чек-лист с точечной нотацией внутри каждой задачи)
- Кросс-сервисные зависимости (между задачами разных сервисов)
- Маппинг TASK-N → GitHub Issues (создаётся через `/dev-create` при переходе WAITING → RUNNING)

**НЕ содержит:**
- Бизнес-обоснование и требования (→ Discussion)
- Распределение ответственностей, контракты API, data model (→ Design)
- Тестовые сценарии и тестовые данные (→ Plan Tests)
- Конкретный код реализации (→ разработка)

**Граничные случаи (Plan Dev vs Plan Tests / Design):**
- Допустимо: "Создать модуль auth.tokens с интерфейсом TokenGenerator" — задача на реализацию
- Допустимо: "Сложность: 7/10, Приоритет: high" — метаданные задачи
- Допустимо: "TASK-3 зависит от TASK-1 (создание схемы в shared/)" — зависимость между задачами
- Недопустимо: "Пользователь с валидными credentials получает 200" — тестовый сценарий (→ Plan Tests)
- Недопустимо: "auth-сервис отвечает за генерацию JWT" — распределение ответственностей (→ Design)
- Недопустимо: "POST /auth/token принимает grant_type" — контракт API (→ Design)

**Когда создавать:**
- После одобрения Plan Tests (статус WAITING)
- Один Plan Dev на один Plan Tests (1:1)

**Связи:** Parent — Plan Tests (обязателен, 1:1). Дочерних объектов нет (терминальный). ([Стандарт analysis/ § 3.1](../standard-analysis.md#31-frontmatter-и-навигация))

**Входные данные (фильтрация Plan Tests → Plan Dev):**

Для каждого per-service раздела Plan Dev читает:

| # | Источник | Что читается | Назначение |
|---|----------|-------------|------------|
| 1 | Plan Tests | TC-N сценарии для сервиса | Определить задачи по реализации тестов и кода |
| 2 | Design SVC-N | Подсекции §§ 1-8 (delta) + § 9 решения по реализации | Определить что менять в коде |
| 3 | Discussion REQ-N | Приоритеты требований | Расставить приоритеты задач |
| 4 | `specs/docs/{svc}.md` | Текущий AS IS (Code Map, зависимости) | Контекст для определения объёма работ |
| 5 | `specs/docs/.system/testing.md` | Стратегия тестирования (инструменты, фреймворки) | Контекст для задач по тестированию |

Plan Dev определяет **какие задачи** выполнить для реализации решения. **Как реализовать** конкретную задачу — решение агента-кодера при выполнении.

**Ограничение размера:** Если Plan Dev содержит более 30 TASK-N суммарно — рассмотреть разбиение Discussion на несколько отдельных цепочек. Plan Dev оптимален для 5-15 задач на сервис.

## 4. Переходы статусов

**SSOT:** [Стандарт analysis/ § 6](../standard-analysis.md#6-последовательность-статусов)

Plan Dev использует общую модель 8 статусов. Особенности:

| Переход | Особенность Plan Dev |
|---------|---------------------|
| DRAFT → WAITING | Пользователь одобрил план задач. При переходе в WAITING создаётся review.md (`/review-create`) |
| WAITING → RUNNING | Tree-level: все 4 документа цепочки в WAITING. **По команде пользователя** создаются GitHub Issues через `/issue-create` ([standard-issue.md](/.github/.instructions/issues/standard-issue.md)) и привязываются к Milestone |
| RUNNING → REVIEW | Все задачи TASK-N выполнены. Tree-level переход ([§ 6.5](../standard-analysis.md#65-running-to-review)) |
| REVIEW → DONE | review.md RESOLVED (вердикт READY). Триггер каскада DONE ([§ 6.6](../standard-analysis.md#66-review-to-done)): Plan Tests → Design → Discussion → DONE |
| RUNNING → CONFLICT | Tree-level. Обратная связь от кода. Issues остаются открытыми |
| CONFLICT → WAITING | LLM сравнивает TASK-N в обновлённом Plan Dev с существующими Issues. Критерий изменения: изменились поля Сложность, Приоритет, Зависимости, TC или подзадачи. Изменённые → `/issue-modify`, удалённые → close `--reason "not planned"`, новые → `/issue-create`. Не изменившиеся Issues не обновляются |
| → ROLLING_BACK | Tree-level. Issues закрываются `--reason "not planned"` с комментарием "rolled back". Feature-ветка удаляется |

**Управление статусами:** [`chain_status.py`](../../.scripts/chain_status.py) — SSOT-модуль для всех переходов. Вызовы `ChainManager.transition()` — в `create-plan-dev.md` и `modify-plan-dev.md`.

**Следующий шаг после WAITING:** Когда все 4 документа цепочки в WAITING — запустить `/dev-create {NNNN}` ([create-development.md](/.github/.instructions/development/create-development.md)) для создания Issues, Milestone, ветки и перехода в RUNNING.

**Порядок выполнения задач:** Агент-кодер получает ссылку на Plan Dev и идёт **сверху вниз**. Если текущая задача заблокирована зависимостью — пропустить, взять следующую незаблокированную. Расположение в документе и поле `Зависимости` должны быть согласованы: задача с зависимостями размещается **после** задач, от которых зависит.

**Критерий выполненности TASK-N:** Задача считается выполненной (зависящие задачи разблокированы), когда все подзадачи в чек-листе отмечены `[x]` и соответствующий GitHub Issue закрыт.

**Рабочие правки (RUNNING, без CONFLICT):** При «рабочих правках» ([Стандарт analysis/ § 6.3](../standard-analysis.md#63-running-to-conflict) — уровень «Флаг») LLM автономно обновляет Plan Dev: может добавлять подзадачи, уточнять описания, менять порядок внутри задачи. Новые TASK-N добавляются **в конец per-service раздела** с номером = max(TASK-N) + 1. Переименование существующих TASK-N запрещено — ломает ссылки в GitHub Issues.

**Модель выполнения:** Plan Dev описывает блочную параллельную модель. Задачи (TASK-N) группируются в блоки (BLOCK-N) по 1-2 задачи (максимум 3 при тесной связности), блоки распределяются по волнам (waves) на основе зависимостей. Каждый блок выполняется отдельным dev-agent ([AGENT.md](/.claude/agents/dev-agent/AGENT.md)) через Task tool. Блоки одной волны запускаются параллельно. Dev-agent работает в изолированном контексте: получает BLOCK-N с 1-2 Issues и контекст (plan-dev, plan-test, specs/docs/{svc}.md, design, conventions), итерирует по Issues — код → тесты → линт → коммит → CONFLICT-CHECK → close issue. После каждого коммита агент обязан проверить границы автономии (specs/docs/{svc}.md → "Границы автономии LLM") и при обнаружении CONFLICT — немедленно остановиться.

## 5. Разделы документа

### Резюме

Первый раздел после заголовка. Краткое описание scope: количество сервисов, общее количество задач, средняя сложность, ключевые кросс-сервисные зависимости.

**Формат:**

```markdown
## Резюме

Реализация {тема} затрагивает {N} сервис(ов): {список}. Всего {M} задач (средняя сложность: {S}/10). Ключевая зависимость: TASK-K ({сервис A}) блокирует TASK-L ({сервис B}).
```

### SVC-N: {Сервис} — per-service разделы

Каждый SVC-N из Design получает свой раздел h2 в Plan Dev. Формат заголовка совпадает с Design и Plan Tests. Порядок — как в Design ([standard-design.md § 5](../design/standard-design.md#5-разделы-документа): Основной сервис → Вторичные сервисы → Новые сервисы).

**Структура per-service раздела:**

```markdown
## SVC-N: {Сервис}

### Задачи

#### TASK-N: {Название задачи}
- **Сложность:** {1-10}/10
- **Приоритет:** {high/medium/low}
- **Зависимости:** {TASK-M / —}
- **TC:** {TC-N, TC-M}
- **Источник:** {SVC-N § K}
- **Issue:** —
- **Type:** {feature / task / infra / test}

Подзадачи:
- [ ] N.1. {Подзадача}
- [ ] N.2. {Подзадача} (deps: N.1)
- [ ] N.3. {Подзадача} (deps: N.1, N.2)
```

**Правила per-service разделов:**
- Один h2 на сервис, формат `## SVC-N: {name}` — совпадает с заголовком в Design и Plan Tests
- Внутри — один h3 `### Задачи`
- Каждая задача — h4 `#### TASK-N: {Название}`
- Нумерация TASK-N сквозная по всему документу
- Порядок задач = порядок выполнения внутри сервиса

**Формат задачи (TASK-N):**

| Поле | Формат | Описание | Обязательность |
|------|--------|----------|----------------|
| **Сложность** | `{1-10}/10` | LLM оценивает на основе контекста из 4 уровней спецификаций | Обязательно |
| **Приоритет** | `high` / `medium` / `low` | Маппится на label приоритета ([standard-labels.md](/.github/.instructions/labels/standard-labels.md)) при создании Issue | Обязательно |
| **Зависимости** | `TASK-M` / `TASK-M, TASK-K` / `—` | Блокирующие: задача не берётся, пока зависимости не выполнены. Внутренние и кросс-сервисные | Обязательно |
| **TC** | `TC-N` / `TC-N, TC-M` | Ссылки на тест-сценарии из Plan Tests. Трассируемость задача → тесты | Обязательно |
| **Источник** | `SVC-N § K` / `SVC-N § K, SVC-N § L` | Подсекции Design, определяющие что менять в коде. Используется при `/dev-create` для генерации навигации в Issue body | Обязательно |
| **Issue** | `[#N](url)` / `—` | Ссылка на GitHub Issue. Заполняется при `/dev-create` (WAITING → RUNNING) | Опционально |
| **Type** | `feature` / `task` / `infra` / `test` | TYPE-метка для GitHub Issue. Используется при `/dev-create` для назначения метки | Опционально |

**Правила значения полей:**
- **Сложность:** целое число 1-10, через `/10`. 1 — тривиальная задача, 10 — максимально сложная
- **Приоритет:** ровно одно из трёх значений: `high`, `medium`, `low` (строчные)
- **Зависимости:** если нет зависимостей — символ `—` (em-dash). Зависимость на задачу другого сервиса = кросс-сервисная зависимость
- **TC:** хотя бы один TC-N из Plan Tests. Отсутствие TC-N — **блокирующая ошибка валидации** (документ не проходит валидацию в статусе > DRAFT). **Исключение:** инфраструктурные задачи (настройка окружения, CI/CD, Dockerfile, инициализация схем БД), не имеющие acceptance-сценариев в Plan Tests, используют маркер `TC: INFRA` вместо конкретного TC-N. Таких задач — не более 20% от общего числа TASK-N
- **Источник:** ссылка в формате `SVC-N § K` — N = порядковый номер SVC-N секции в `design.md` данной цепочки (один Design-документ на цепочку, нумерация сквозная), K = номер подсекции (1-9). **Точность обязательна:** указывать конкретные номера подсекций (`SVC-3 § 4, SVC-3 § 5`), а не обобщённые ссылки (`SVC-3`). Используется при `/dev-create` для автоматической генерации навигации в Issue body — колонка "Что искать" в таблице "Документы для изучения" ([create-development.md](/.github/.instructions/development/create-development.md))
  **Примеры Источник (хорошо / плохо):**

  | Хорошо | Плохо | Почему |
  |--------|-------|--------|
  | `SVC-3 § 4, SVC-3 § 5` | `SVC-3` | Агент не знает, какие подсекции читать — читает весь SVC-3 (избыточно) |
  | `SVC-1 § 2, SVC-1 § 9` | `SVC-1 § 2-9` | Диапазон неоднозначен — агент не знает, нужны ли §§ 3-8 |
  | `SVC-2 § 5` | `Design` | Нет указания на конкретный SVC-N и подсекцию |

- **Issue:** при создании Plan Dev — символ `—` (em-dash). При `/dev-create` заполняется полной ссылкой `[#N](https://github.com/{owner}/{repo}/issues/N)`. Поле опционально: при статусе DRAFT/WAITING может отсутствовать
- **Type:** TYPE-метка для GitHub Issue. Одно из: `feature` (новая функциональность, CRUD, UI, бизнес-логика), `task` (scaffold, middleware, схемы, boilerplate), `infra` (Docker, CI/CD, монорепо, конфиги), `test` (E2E, integration, load, smoke). Поле опционально: при отсутствии `/dev-create` определяет тип автоматически по критериям: `TC: INFRA` → `infra`; системные тесты (STS-N, E2E/load) → `test`; бизнес-логика (CRUD, UI, API) → `feature`; остальное → `task`

**Формат подзадач:**

```markdown
Подзадачи:
- [ ] N.1. {Описание}
- [ ] N.2. {Описание} (deps: N.1)
- [ ] N.3. {Описание} (deps: N.1, N.2)
```

**Правила подзадач:**
- Точечная нотация: `N.M` (N = номер задачи, M = номер подзадачи)
- Чек-лист формат: `- [ ]` (unchecked при создании, `- [x]` при выполнении)
- Зависимости внутри задачи: `(deps: N.M)` — на сиблингов-подзадачи
- Кросс-задачные зависимости отражаются в поле `Зависимости` самой TASK-N, а **не** в подзадачах. Подзадача начинается только после того как вся TASK-N (от которой зависит родительская задача) выполнена
- Подзадачи опциональны: если задача атомарная — блок `Подзадачи:` не указывается
- Если блок `Подзадачи:` присутствует — минимум 2 подзадачи. Один пункт в блоке — ошибка валидации: либо добавить вторую подзадачу, либо удалить блок (задача атомарная)
- При CONFLICT-обновлении: отметки `[x]` сохраняются — прогресс выполненных подзадач не сбрасывается

### Кросс-сервисные зависимости

Общий раздел после всех per-service разделов. Таблица связей между задачами разных сервисов.

**Формат:**

```markdown
## Кросс-сервисные зависимости

| Зависимость | Блокирует | Причина |
|-------------|-----------|---------|
| TASK-1 (auth) | TASK-5 (gateway) | gateway требует JWT-модуль из auth |
| TASK-1 (auth) | TASK-8 (users) | users требует shared-схему из auth |
```

**Правила:**
- Таблица — дубль зависимостей из полей TASK-N, собранная в одном месте для обзора
- Только кросс-сервисные (между задачами разных сервисов)
- Внутрисервисные зависимости — только в поле `Зависимости` задачи
- Если кросс-сервисных зависимостей нет — указать: `*Кросс-сервисных зависимостей нет.*`

### Блоки выполнения

Раздел после "Кросс-сервисные зависимости". Определяет группировку задач в блоки (BLOCK-N) и распределение блоков по волнам (waves) для параллельного выполнения dev-agent-ами.

**Формат:**

```markdown
## Блоки выполнения

| BLOCK | Задачи | Сервисы | Зависимости | Wave |
|-------|--------|---------|-------------|------|
| BLOCK-1 | TASK-1 | shared (INFRA) | — | 0 |
| BLOCK-2 | TASK-2, TASK-3 | auth | BLOCK-1 | 1 |
| BLOCK-3 | TASK-4 | auth | BLOCK-2 | 2 |
| BLOCK-4 | TASK-5 | gateway | BLOCK-1 | 1 |
| BLOCK-5 | TASK-6, TASK-7 | users | BLOCK-1 | 1 |
| BLOCK-6 | TASK-8 | users | BLOCK-5 | 2 |
| BLOCK-7 | TASK-9 | e2e (system) | BLOCK-3, BLOCK-6 | 3 |
```

**Правила блоков:**

| Правило | Описание |
|---------|----------|
| **No file overlap** | Блоки одной волны НЕ ДОЛЖНЫ затрагивать одни и те же файлы. Shared-файлы (`shared/`, `config/`, `platform/`) выделяются в отдельный INFRA-блок |
| **INFRA-блок** | Блок с TASK-N типа `TC: INFRA` (настройка окружения, shared-контракты). Помещается в wave 0 или первую волну, выполняется до service-блоков |
| **Покрытие** | Каждый TASK-N принадлежит ровно одному BLOCK-N. Нет TASK-N без блока |
| **Размер** | 1-2 TASK-N на блок (по умолчанию). Максимум 3 — только для тесно связанных задач (модель + миграция, page + route) |
| **Тесты** | Системные/e2e тестовые TASK-N по умолчанию 1 TC = 1 BLOCK (разные файлы). Объединять TC в одну TASK только при file overlap, специфическом shared setup или последовательной зависимости между TC |
| **Зависимости** | Между блоками — через колонку "Зависимости". Блок не запускается, пока все зависимости не завершены |
| **Соответствие TC** | Каждый dev-BLOCK имеет соответствующий test-BLOCK с TC-N для тех же сервисов (см. [standard-plan-test.md](../plan-test/standard-plan-test.md)) |
| **Нумерация** | Сквозная по документу: BLOCK-1, BLOCK-2, ... Совпадает между plan-dev и plan-test |

**Как определяются блоки:**

1. **Группировка по связности** — по умолчанию 1 TASK-N = 1 BLOCK. Группировать 2 TASK-N в один BLOCK только если они в одном сервисе И один без другого бессмысленен (модель + миграция, page + route). Максимум 3 TASK-N в исключительных случаях
2. **Shared → INFRA** — TASK-N с `TC: INFRA` или затрагивающие `shared/` → отдельный блок
3. **Зависимости** — если BLOCK-A зависит от BLOCK-B, они в разных волнах
4. **File overlap check** — если два блока одной волны затрагивают одни файлы → перенести один в следующую волну или объединить
5. **Тестовая декомпозиция** — системные/e2e тесты: каждый TC-N (или группа TC, пишущих в один файл) = отдельная TASK-N → отдельный BLOCK-N. Общая тестовая инфраструктура (helpers, conftest) → INFRA-блок в предшествующей волне

**Волна (wave)** — набор BLOCK-N без взаимных зависимостей. Все блоки волны запускаются параллельно.

**Формирование волн:**

1. Топологическая сортировка графа зависимостей BLOCK-N
2. Блоки без зависимостей → wave 1 (или wave 0, если INFRA)
3. Блоки, зависящие от wave N → wave N+1
4. Циклические зависимости → ошибка при создании plan-dev (валидация)

### Маппинг GitHub Issues

Последний раздел. Описание процесса маппинга задач в GitHub Issues.

**Формат:**

```markdown
## Маппинг GitHub Issues

Маппинг выполняется через `/dev-create` при переходе WAITING → RUNNING ([Стандарт analysis/ § 4.1](../standard-analysis.md#41-прямой-поток)). LLM создаёт каждый TASK-N → Issue через `/issue-create`. После создания Issue номер записывается в поле `Issue` соответствующей TASK-N.

| Элемент Plan Dev | GitHub Issue |
|------------------|-------------|
| TASK-N | Отдельный Issue через `/issue-create` |
| Подзадачи N.M | Чек-лист в body Issue |
| Приоритет | Label приоритета ([standard-labels.md](/.github/.instructions/labels/standard-labels.md)) |
| Type | TYPE-метка (`feature`, `task`, `infra`, `test`) |
| Зависимости | `**Зависит от:** #N` в body |
| TC | Ссылка в секции "Связанная документация" |
| Milestone | Из frontmatter Discussion ([standard-milestone.md § 6](/.github/.instructions/milestones/standard-milestone.md#6)) |
| Issue номер | Поле `- **Issue:** [#N](url)` в TASK-N |
```

**Правила:**
- Маппинг выполняется через `/dev-create` при переходе всей цепочки в RUNNING
- После создания Issue — номер записывается **inline в поле `Issue`** каждой TASK-N (формат: `[#N](url)`)
- Подзадачи → чек-лист в body Issue. Если подзадача требует отдельного Pull Request (самостоятельный deliverable, независимое ревью) — sub-issue через `/issue-create` с ссылкой на родительский Issue
- Milestone берётся из frontmatter Discussion, проверяется/создаётся при маппинге

### Предложения

Раздел для записи замечаний plandev-reviewer (PROP-N). Формат аналогичен Design.

**Формат:**

```markdown
## Предложения

_Предложений нет._
```

**Или с PROP-N:**

```markdown
## Предложения

### {Аспект}

| PROP | Приоритет | Категория | Описание |
|------|-----------|-----------|----------|
| PROP-1 | P1 | {категория} | {описание} |
```

**Правила:**
- Записывается plandev-reviewer (агент), не пользователем
- Приоритеты: P1 (критический), P2 (значительный), P3 (мелкий)
- При отсутствии замечаний — заглушка `_Предложений нет._`
- Максимум 15 PROP-N за один запуск ревьюера

### Отвергнутые предложения

Раздел для отклонённых PROP-N — предложений, которые пользователь решил не принимать.

**Формат:**

```markdown
## Отвергнутые предложения

_Отвергнутых предложений нет._
```

**Правила:**
- Перенос из "Предложения" выполняется оркестратором/пользователем
- Каждый перенесённый PROP-N сохраняет оригинальный номер
- При отсутствии — заглушка `_Отвергнутых предложений нет._`

## 8. Чек-лист качества

### Структура

- [ ] Заголовок: `# NNNN: {Тема} — Plan Dev`
- [ ] Резюме: scope, количество задач, средняя сложность (среднее арифметическое оценок всех TASK-N)
- [ ] Per-service разделы: каждый SVC-N из Design имеет свой h2 в формате `## SVC-N: {name}`
- [ ] Порядок per-service: как в Design (Основной → Вторичный → Новый)
- [ ] Кросс-сервисные зависимости: таблица или заглушка
- [ ] Маппинг GitHub Issues: описание процесса
- [ ] Секция «Предложения» — PROP-N таблица или заглушка
- [ ] Секция «Отвергнутые предложения» — PROP-N таблица или заглушка

### Задачи (TASK-N)

- [ ] Нумерация сквозная по всему документу (TASK-1, TASK-2, ...)
- [ ] Каждая задача — h4 `#### TASK-N: {Название}`
- [ ] Все 5 обязательных полей заполнены (Сложность, Приоритет, Зависимости, TC, Источник)
- [ ] Поле Issue: `—` при DRAFT/WAITING, `[#N](url)` при RUNNING+
- [ ] Поле Type (если указано): одно из `feature` / `task` / `infra` / `test`
- [ ] Сложность: целое число 1-10, формат `N/10`
- [ ] Приоритет: одно из `high` / `medium` / `low`
- [ ] Зависимости: `TASK-M` или `—`
- [ ] TC: хотя бы один TC-N из Plan Tests
- [ ] Источник: `SVC-N § K` из Design (конкретные номера подсекций, не обобщённые)

### Подзадачи

- [ ] Точечная нотация: `N.M`
- [ ] Чек-лист формат: `- [ ]`
- [ ] Зависимости на сиблингов: `(deps: N.M)` — только внутри одной задачи
- [ ] Минимум 2 подзадачи, если блок указан

### Блоки выполнения (BLOCK-N)

- [ ] Каждый TASK-N принадлежит ровно одному BLOCK-N
- [ ] Нет циклических зависимостей между BLOCK-N
- [ ] INFRA-блоки размещены в wave 0 или первой волне
- [ ] Блоки одной волны не имеют file overlap (no shared files)
- [ ] Размер блока: 1-2 TASK-N (по умолчанию), максимум 3 для тесно связанных
- [ ] Нумерация BLOCK-N сквозная, совпадает с plan-test

### Покрытие

- [ ] Каждый TC-N из Plan Tests покрыт хотя бы одной задачей TASK-N (обратная трассируемость)
- [ ] Каждый SVC-N из Design имеет задачи
- [ ] Нет задач без TC-N (каждая задача привязана к тест-сценарию или имеет `TC: INFRA`)
- [ ] Задачи с `TC: INFRA` составляют не более 20% от общего числа TASK-N
- [ ] Нет циклических зависимостей между TASK-N
- [ ] Задачи с зависимостями расположены **после** задач, от которых зависят

### Зона ответственности

- [ ] Нет тестовых сценариев (→ Plan Tests)
- [ ] Нет распределения ответственностей (→ Design)
- [ ] Нет бизнес-обоснований (→ Discussion)
- [ ] Нет конкретного кода реализации (→ разработка)

### Маркеры

- [ ] Нет `[ТРЕБУЕТ УТОЧНЕНИЯ]` при статусе > DRAFT

#### TASK-1: Создать модуль auth.tokens

#### TASK-5: Интеграция gateway с auth-токенами
- **Сложность:** 5/10
- **Приоритет:** high
- **Зависимости:** TASK-1
- **TC:** TC-8, TC-9
