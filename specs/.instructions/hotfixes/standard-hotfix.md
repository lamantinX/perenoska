# Стандарт хотфикса

Версия стандарта: 1.0

Формат и правила оформления файлов хотфиксов в `specs/hotfixes/`. Каждый файл — полная история инцидента: проблема, диагностика, корневая причина, решение, impact analysis, внесённые изменения.

hotfix.md создаётся и наполняется через скилл `/hotfix`. Ручное создание не рекомендуется.

> **GitHub Issues:** Для хотфиксов Issue не создаются. Трекинг — через hotfix.md (файл является единственным источником истории инцидента).

## 1. Frontmatter

Каждый файл hotfix.md начинается с YAML frontmatter:

```yaml
---
date: YYYY-MM-DD
topic: краткое-описание (kebab-case)
status: INVESTIGATING | IN_PROGRESS | RESOLVED
services: [svc1, svc2]
rollback_commit: — | <commit-hash>
rollback_command: — | git revert <commit-hash>
---
```

| Поле | Тип | Обязательное | По умолчанию | Описание |
|------|-----|--------------|--------------|----------|
| `date` | string | Да | — | Дата обнаружения проблемы (YYYY-MM-DD) |
| `topic` | string | Да | — | Краткое описание в kebab-case |
| `status` | enum | Да | `INVESTIGATING` | Текущий статус (см. [§ 3](#3-статусы)) |
| `services` | list | Да | — | Список затронутых сервисов. При инфраструктурном инциденте, затрагивающем все сервисы, перечислить конкретные сервисы явно |
| `rollback_commit` | string | Да | `—` | Обязательно. Значение `—` до шага 5 TaskList, после — хеш коммита HEAD до начала изменений |
| `rollback_command` | string | Да | `—` | Обязательно. Значение `—` до шага 5 TaskList, после — `git revert <hash>` |

> **Обновление `status`:** Поле `status` в frontmatter обновляется синхронно с переходами, описанными в [§ 3](#3-статусы). INVESTIGATING → IN_PROGRESS на шаге 4 TaskList, IN_PROGRESS → RESOLVED на шаге 7 TaskList.

> **Примечание:** "Шаг N TaskList" — шаги процесса из [create-hotfix.md](./create-hotfix.md), не номера секций этого документа.

## 2. Обязательные секции

### 2.1 Правило наполнения

> **hotfix.md наполняется строго последовательно по TaskList.** Содержимое уже заполненной секции не редактируется. Исключения:
> - Секция "Статус" (чек-лист) — обновляется на каждом шаге TaskList
> - Поле `status` в frontmatter — обновляется при смене статуса (см. [§ 3](#3-статусы))
> - При обратном переходе IN_PROGRESS → INVESTIGATING — см. [§ 3](#3-статусы)

### 2.2 Таблица секций

Каждый hotfix.md содержит 9 секций. Секции заполняются последовательно по мере продвижения по TaskList.

| # | Секция | Заполняется на шаге | Описание |
|---|--------|---------------------|----------|
| 1 | **Проблема** | Шаг 1 | Симптомы, что видит пользователь, как воспроизвести |
| 2 | **Диагностика** | Шаг 2 | Логи, метрики, воспроизведение, цепочка отказа |
| 3 | **Корневая причина** | Шаг 2 | Что сломано и почему (root cause, не симптом) |
| 4 | **Варианты решения** | Шаг 3 | Минимум 2 варианта. Для каждого: плюсы/минусы/риски + code impact. Если обоснованный вариант только один — добавить вариант "Без изменений" с явными рисками |
| 5 | **Выбранное решение** | Шаг 3 | Какой вариант утверждён, почему + таблица: файл → что изменить → риски |
| 6 | **Затронутые сервисы** | Шаг 3 | Таблица: сервис → что меняется |
| 7 | **Затронутые документы** | Шаг 4 | Impact analysis: таблицы по specs/analysis/ и specs/docs/ |
| 8 | **Внесённые изменения** | Шаг 5 | Таблица коммитов + описание изменений (см. [формат](#секция-8-внесённые-изменения--формат)) |
| 9 | **Статус** | Все шаги | Чек-лист из 7 пунктов |

### 2.3 Форматы секций

#### Секция 4: Варианты решения — формат

Для каждого варианта:

```markdown
### Вариант N: Название

**Описание:** Что делает решение.

**Code impact:**

| Файл | Что изменить | Риски |
|------|-------------|-------|
| `src/{svc}/...` | Описание изменения | Что может сломаться |

**Плюсы:** ...
**Минусы:** ...
```

#### Секция 5: Выбранное решение — формат

```markdown
**Решение:** Вариант N — Название.
**Почему:** Обоснование выбора.

**План изменений:**

| Файл | Что изменить | Риски |
|------|-------------|-------|
| `src/{svc}/...` | Описание | Риск |

### Архитектура решения

Полное описание архитектуры решения. Если есть драфт — скопировать
архитектуру из драфта без сокращений. Включает:
- Новые таблицы/модели (полная schema с полями)
- Правила и границы (конкретные значения, формулы, пороги)
- Потоки данных (кто → кому → когда → что передаёт)
- Граничные условия (переходные состояния, edge cases)
- UI изменения (макет, компоненты)

### Чеклист реализации

Каждое поведенческое требование из архитектуры = одна строка чеклиста.
LLM отмечает чеклист по мере реализации. Пользователь видит прогресс.

- [ ] Требование 1: конкретное описание поведения
- [ ] Требование 2: конкретное описание поведения
- [ ] ...
```

#### Секция 7: Затронутые документы — формат

```markdown
### specs/analysis/

| Документ | Статус | Секции | Что обновить |
|----------|--------|--------|-------------|
| `{NNNN}-{topic}/design.md` | DONE | SVC-1 § 8.3 | Обновить Data Model |

### specs/docs/

| Документ | Секции | Что обновить |
|----------|--------|-------------|
| `{svc}.md` | Data Model, API | Добавить новое поле |
```

#### Секция 8: Внесённые изменения — формат

```markdown
### Коммит

| Хеш | Описание |
|-----|---------|
| `abc1234` | fix: isolate Redis pools, add market_closed gate |

### Описание изменений

Краткое prose-описание: что изменено, почему, какие решения приняты в процессе.
```

> **Один коммит на хотфикс.** Все изменения (код + документация) коммитятся одним `fix:` коммитом после полного тестирования.

#### Секция 9: Статус — формат

```markdown
- [ ] Проблема описана
- [ ] Диагностика завершена, корневая причина найдена
- [ ] Решение выбрано, code impact определён
- [ ] Impact analysis на документацию завершён
- [ ] Код исправлен, тесты обновлены
- [ ] Документация обновлена
- [ ] Тестирование пройдено
```

## 3. Статусы

| Статус | Когда | Шаги TaskList |
|--------|-------|---------------|
| `INVESTIGATING` | Проблема описана, диагностика и выбор решения | Шаги 1-3 |
| `IN_PROGRESS` | Impact analysis завершён, идёт реализация и тестирование | Шаги 4-6 |
| `RESOLVED` | Исправление завершено, коммит в main | Шаг 7 |

**Переходы:**

```
INVESTIGATING → IN_PROGRESS → RESOLVED
```

**Обратный переход IN_PROGRESS → INVESTIGATING:**

Допустим только при неудаче тестирования (шаг 6 TaskList), если обнаружено, что корневая причина определена неверно. При откате статуса:

1. Обновить `frontmatter.status` на `INVESTIGATING`
2. Не удалять предыдущие секции 3-5 — добавить в конец каждой секции подзаголовок `### Итерация 2` с обновлённым содержимым
3. Вернуться к шагу 2 TaskList

## 4. Именование и расположение

**Файл:** `specs/hotfixes/{YYYY-MM-DD}-{topic}.md`

| Компонент | Формат | Пример |
|-----------|--------|--------|
| Дата | `YYYY-MM-DD` | `2026-03-23` |
| Topic | kebab-case | `ws-redis-pool-exhaustion` |

**Примеры:**
- `specs/hotfixes/2026-03-23-ws-redis-pool-exhaustion.md`
- `specs/hotfixes/2026-03-15-oauth-callback-redirect.md`

**Обновление индекса:** После завершения хотфикса (статус RESOLVED) — обновить таблицу в [specs/hotfixes/README.md](/specs/hotfixes/README.md). Подробнее — см. [create-hotfix.md](./create-hotfix.md).

---

## 5. Коммиты и откат

**Один коммит на хотфикс:** Тип `fix:` (Conventional Commits) напрямую в main. Все изменения (код + тесты + документация) в одном коммите после полного тестирования.

```
fix: isolate Redis pools, add market_closed gate, update docs
```

**Откат:** Перед реализацией (шаг 5a TaskList) фиксируется `rollback_commit` (текущий HEAD) и `rollback_command` (`git revert <hash>`) в frontmatter.

> **Откат выполняется ТОЛЬКО по явной команде пользователя.** LLM не инициирует откат самостоятельно.
