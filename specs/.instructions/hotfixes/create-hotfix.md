## Принципы

> **Правильно, а не быстро.** Цель — устранить корневую причину, а не замаскировать симптомы. Быстрый workaround допустим только как временная мера с обязательным TODO на правильное решение. Если "быстрый фикс" и "правильный фикс" конфликтуют — выбирать правильный.

> **Чинить причину, а не следствие.** Диагностика (шаги 1-3) продолжается до тех пор, пока не найдена корневая причина. "Перезапустить сервис", "увеличить таймаут", "добавить retry" — это не решение, если причина в другом.

> **`/hotfix` — точка входа для ВСЕХ багов.** Критический production-инцидент или bundle мелких фиксов — всё через `/hotfix`.

> **Диагностика ДО кода.** Сначала понять проблему (шаги 1-3), потом исправлять (шаги 5-7).

> **Impact analysis на ВСЮ документацию.** Хотфикс может затрагивать docs в любом статусе (DONE, RUNNING, WAITING). Поиск — параллельными агентами.

> **Параллельное исправление.** Код (основной LLM) и документация (hotfix-docs-agent × N) — одновременно.

> **Один файл hotfix.md — полная история.** Проблема, диагностика, решение, impact, изменения — всё в одном документе.

## Шаги

### Шаг 1: Создать TaskList

Создать 7 задач через TaskCreate (шаблон — см. [TaskList: шаблон](#tasklist-шаблон)).

AskUserQuestion: "Создаю хотфикс-процесс из 7 задач. Начинаем?"

### Шаг 2: Начать выполнение

TaskUpdate первой задачи → `in_progress`. Далее по шагам TaskList.

### Шаг 3: Возобновление (при /hotfix --resume)

Найти hotfix-файлы в `specs/hotfixes/` со статусом `INVESTIGATING` или `IN_PROGRESS`. Восстановить TaskList по чек-листу в hotfix.md (секция "Статус").

## TaskList: шаблон

```
Шаг 1: Определить проблематику
  description: >
    Основной LLM + пользователь. AskUserQuestion для уточнения контекста:
    сервисы, логи.
    Создать specs/hotfixes/{date}-{topic}.md с секциями: Проблема, Контекст.
    Статус hotfix.md: INVESTIGATING.
    SSOT: standard-hotfix.md
  activeForm: Определяю проблематику
  blockedBy: —

Шаг 2: Диагностика
  description: >
    Основной LLM + пользователь. Изучить логи (docker logs), код затронутых
    сервисов (specs/docs/{svc}.md + src/), воспроизвести если возможно.
    Обновить hotfix.md: Диагностика, Корневая причина, Цепочка отказа.
    SSOT: standard-hotfix.md
  activeForm: Диагностирую проблему
  blockedBy: [1]

Шаг 3: Выработать решение
  description: >
    Основной LLM + пользователь.
    3a. Code impact analysis: прочитать specs/docs/{svc}.md затронутых сервисов
    (Code Map, Data Model, зависимости), grep по src/{svc}/ — найти конкретные
    файлы и функции для изменения, определить зависимые модули и тесты.
    3b. Определить минимум 2 варианта решения. Для каждого варианта:
    какие файлы/функции менять, что может сломаться (зависимости, контракты, тесты).
    Если обоснованный вариант только один — добавить вариант "Без изменений" с рисками.
    3c. AskUserQuestion: выбор варианта.
    Обновить hotfix.md: Варианты решения (с code impact), Выбранное решение,
    Затронутые сервисы, таблица: файл → что изменить → риски.
    SSOT: standard-hotfix.md
  activeForm: Вырабатываю решение
  blockedBy: [2]

Шаг 4: Impact analysis на документацию
  description: >
    Агент: hotfix-docs-search-agent × 5 (параллельно по scope:
    analysis, docs, technologies, system, pipelines).
    Поиск влияния хотфикса на ВСЮ документацию.
    Основной LLM сводит результаты.
    Обновить hotfix.md: Затронутые документы (impact analysis).
    Обновить frontmatter: status → IN_PROGRESS.
    SSOT: standard-hotfix.md
  activeForm: Анализирую влияние на документацию
  blockedBy: [3]

Шаг 5: Исправление (код + документация)
  description: >
    5a. Зафиксировать rollback: записать текущий HEAD коммит в frontmatter
    hotfix.md (rollback_commit, rollback_command) — точка отката ДО изменений.
    5b. Основной LLM: исправить код, написать/обновить тесты. НЕ коммитить.
    После каждого TASK: пересобрать затронутый сервис, проверить логи,
    сверить поведение с чеклистом реализации из hotfix.md. Отметить чеклист.
    5c. Агент: hotfix-docs-agent × N (параллельно, по одному на документ
    из impact analysis). Обновление конкретных секций.
    5d. Финальная проверка: пересобрать ВСЕ затронутые сервисы, проверить
    логи и UI. Сверить git diff с чеклистом — каждый пункт покрыт?
    5e. ОДИН коммит fix: со всеми изменениями через /commit. Push в remote.
    Обновить hotfix.md: Внесённые изменения (коммит + описание).
    SSOT: standard-hotfix.md
  activeForm: Исправляю код и документацию
  blockedBy: [4]

Шаг 6: Тестирование
  description: >
    /docker-up → make test (per-service для затронутых сервисов) →
    пользователь проверяет вручную (если нужно).
    При NOT READY → вернуться к шагу 5 (или к шагу 2 если корневая причина неверна).
    SSOT: standard-hotfix.md
  activeForm: Тестирую исправление
  blockedBy: [5]

Шаг 7: Завершение
  description: >
    Проверить что rollback_commit в frontmatter заполнен (из шага 5a).
    Обновить hotfix.md: frontmatter status → RESOLVED, дата завершения.
    Отметить все чек-боксы в секции "Статус".
    Обновить specs/hotfixes/README.md (добавить строку в индекс).
    SSOT: standard-hotfix.md
  activeForm: Завершаю хотфикс
  blockedBy: [6]
```

**7 шагов** (vs 15 у /chain). Нет analysis chain, нет docs sync — вместо них диагностика + impact analysis + параллельное исправление.

**Динамическое поведение:**
- Шаг 6 → Шаг 5 (цикл при NOT READY)
- Шаг 6 → Шаг 2 (если корневая причина неверна, см. [standard-hotfix.md § 3](./standard-hotfix.md#3-статусы))
- `{date}` и `{topic}` подставляются из шага 1

## TaskList: описание шагов

### Шаг 1: Определить проблематику

**Кто:** Основной LLM + пользователь.

**Действия:**
1. AskUserQuestion: уточнить сервисы, симптомы, логи
2. Создать `specs/hotfixes/{date}-{topic}.md` по шаблону из [standard-hotfix.md § 6](./standard-hotfix.md#6-шаблон)
3. Заполнить секцию "Проблема"
4. Отметить чек-бокс "Проблема описана" в секции "Статус"

### Шаг 2: Диагностика

**Кто:** Основной LLM + пользователь.

**Действия:**
1. Прочитать `specs/docs/{svc}.md` затронутых сервисов (Code Map, зависимости)
2. Изучить логи (`docker logs`), код (`src/{svc}/`)
3. Воспроизвести проблему если возможно
4. Заполнить секции "Диагностика" и "Корневая причина"
5. Отметить чек-бокс "Диагностика завершена, корневая причина найдена"

### Шаг 3: Выработать решение

**Кто:** Основной LLM + пользователь.

**Действия:**
1. **Code impact analysis:** grep по `src/{svc}/`, прочитать Code Map и Data Model из `specs/docs/{svc}.md`, определить файлы/функции для изменения и зависимые модули
2. Определить минимум 2 варианта решения с code impact для каждого
3. AskUserQuestion: выбор варианта
4. Заполнить секции "Варианты решения", "Выбранное решение", "Затронутые сервисы"
5. Отметить чек-бокс "Решение выбрано, code impact определён"

### Шаг 4: Impact analysis на документацию

**Кто:** hotfix-docs-search-agent × 5 (параллельно).

**Действия:**
1. Запустить 5 агентов параллельно по scope: `analysis`, `docs`, `technologies`, `system`, `pipelines`
2. Каждый агент: grep по keywords + service names, вернуть список затронутых документов
3. Основной LLM сводит результаты
4. Заполнить секцию "Затронутые документы"
5. Обновить `frontmatter.status` → `IN_PROGRESS`
6. Отметить чек-бокс "Impact analysis на документацию завершён"

### Шаг 5: Исправление (код + документация)

**Кто:** Основной LLM (код) + hotfix-docs-agent × N (документация, параллельно).

**Действия:**
1. **5a.** Зафиксировать rollback: записать текущий `git rev-parse HEAD` в `frontmatter.rollback_commit`, сформировать `rollback_command`
2. **5b.** Исправить код, написать/обновить тесты. **НЕ коммитить.** После каждого TASK: пересобрать затронутый сервис, проверить логи, сверить поведение с чеклистом реализации. Отметить чеклист в hotfix.md.
3. **5c.** Запустить hotfix-docs-agent × N параллельно для обновления затронутых документов
4. **5d.** Финальная проверка: пересобрать ВСЕ затронутые сервисы, проверить логи и UI. Сверить `git diff` с чеклистом — каждый пункт покрыт?
5. **5e.** ОДИН коммит `fix:` со всеми изменениями через `/commit`. Push в remote.
6. Заполнить секцию "Внесённые изменения" (коммит + описание)
7. Отметить чек-боксы "Код исправлен, тесты обновлены" и "Документация обновлена"

### Шаг 6: Тестирование

**Кто:** Основной LLM + пользователь.

**Действия:**
1. `/docker-up` → `make test` (per-service для затронутых сервисов)
2. Пользователь проверяет вручную (если нужно)
3. При READY → отметить чек-бокс "Тестирование пройдено", перейти к шагу 7
4. При NOT READY → вернуться к шагу 5 (исправить код)
5. Если обнаружено, что корневая причина неверна → вернуться к шагу 2 (см. [standard-hotfix.md § 3](./standard-hotfix.md#3-статусы))

### Шаг 7: Завершение

**Кто:** Основной LLM.

**Действия:**
1. Проверить что `rollback_commit` в frontmatter заполнен
2. Обновить `frontmatter.status` → `RESOLVED`
3. Убедиться что все чек-боксы в секции "Статус" отмечены
4. Добавить строку в таблицу `specs/hotfixes/README.md`

## Примеры

### Пример: WS Redis Pool Exhaustion

| Шаг | Что было сделано |
|-----|-----------------|
| 1. Проблематика | Страница /charts: "Ошибка загрузки данных", бесконечный WS reconnect |
| 2. Диагностика | Redis pool exhaustion → каскадный отказ HTTP+WS. Логи: "Too many connections" |
| 3. Решение | Code impact: grep src/ → 5 файлов с Redis pool usage. v1: изоляция пулов (3 файла, риск: миграция конфига). v2: market-aware WS (утверждено, 5 файлов, риск: новый event) |
| 4. Impact analysis | 6 документов specs/analysis/, 6 документов specs/docs/, 1 новый pipeline |
| 5. Исправление | Код: 3 Redis pools, WS gate, market_closed event. Docs: параллельное обновление |
| 6. Тестирование | Docker up, per-service тесты, ручная проверка /charts |
| 7. Завершение | hotfix.md → RESOLVED, rollback frontmatter заполнен |
