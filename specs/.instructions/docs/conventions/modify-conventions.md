## Принципы

> **Каскадный эффект.** Изменение кросс-сервисной конвенции затрагивает все сервисы. Обновить conventions.md — недостаточно. Нужно обновить все затронутые `{svc}.md`.

> **Синхронизация с overview.md.** При добавлении/удалении shared-пакета — обновить таблицу в overview.md (секция Shared-код).

> **Валидация в конце.** После любого изменения — запустить валидацию. Не считать обновление завершённым без прохождения.

## Шаги

### Шаг 1: Определить триггер

Определить, какое изменение произошло. Найти соответствующий триггер в [таблице](#триггеры-обновления) и перейти к указанному сценарию.

### Шаг 2: Выполнить обновление по сценарию

Выполнить шаги из соответствующего [сценария](#сценарии).

### Шаг 3: Валидировать результат

```bash
python specs/.instructions/.scripts/validate-docs-conventions.py
```

## Сценарии

### Сценарий 1: Изменена кросс-сервисная конвенция

**Когда:** Изменён формат ответов, ошибок, пагинации, JWT claims, схема версионирования или конвенция логирования.

**Шаги:**

1. **Обновить секцию** — внести изменения в соответствующую секцию conventions.md.
2. **Обновить code-блоки** — если изменение затрагивает реализацию (новое поле, изменённая сигнатура) — обновить code-блок.
3. **Проверить {svc}.md** — если какой-либо `{svc}.md` ссылается на изменённую конвенцию — убедиться, что ссылка актуальна.
4. **Валидация.**

### Сценарий 2: Добавлен shared-пакет

**Когда:** Создан новый пакет в `/shared/`.

**Шаги:**

1. **Добавить h3-подсекцию** в секцию Shared-пакеты:
   ```
   ### shared/{package} — {назначение}
   **Владелец:** {svc}
   {интерфейс, таблица, пример}
   ```
2. **Обновить overview.md** — добавить строку в таблицу Shared-код (см. [modify-overview.md](../overview/modify-overview.md)).
3. **Удалить stub** — если секция содержала stub-текст (`*Shared-пакеты будут добавлены...*`) — удалить его.
4. **Валидация.**

### Сценарий 3: Изменён интерфейс shared-пакета

**Когда:** Изменены сигнатуры, параметры или поведение существующего shared-пакета.

**Шаги:**

1. **Обновить code-блок интерфейса** — новые сигнатуры.
2. **Обновить таблицу параметров/полей** — добавить/удалить/изменить строки.
3. **Обновить пример использования** — если пример устарел.
4. **Валидация.**

### Сценарий 4: Удалён shared-пакет

**Когда:** Пакет в `/shared/` удалён или выведен из эксплуатации.

**Шаги:**

1. **Удалить h3-подсекцию** из секции Shared-пакеты.
2. **Обновить overview.md** — удалить строку из таблицы Shared-код.
3. **Если не осталось пакетов** — добавить stub-текст: `*Shared-пакеты будут добавлены при появлении переиспользуемого кода.*`
4. **Валидация.**

### Сценарий 5: Добавлена новая конвенция (новая секция)

**Когда:** В проекте появилась новая кросс-сервисная конвенция, не покрытая 8 обязательными секциями (например, Rate Limiting, CORS, Health Checks).

**Шаги:**

1. **Добавить h2-секцию после «Требования по уровням критичности»** — новые секции добавляются после последней обязательной (секция 8).
2. **Следовать принципу стандарта** — code-блок с реализацией (если применимо), таблица (если применимо).
3. **Валидация.**

### Сценарий 6: Первичное заполнение (из визуализации)

**Когда:** conventions.md содержит визуализацию — пример из стандарта (MyApp, FastAPI/Python) с пометкой «Это визуализация». Файл уже соответствует формату стандарта, но содержание не отражает реальный проект.

**Шаги:**

1. **Прочитать стандарт:** [standard-conventions.md](./standard-conventions.md) — формат, секции, шаблон.
2. **Удалить blockquote-пометку** `> **Это визуализация.**...` в начале файла.
3. **Заменить содержание секций на актуальные данные проекта** — сохранить структуру (frontmatter, порядок секций, формат таблиц), заменить содержание:
   - Формат ответов — реальный JSON-формат проекта
   - Формат ошибок — реальные коды и реализация
   - Пагинация — реальные параметры и реализация
   - Аутентификация — реальная схема JWT
   - Версионирование API — реальная схема
   - Shared-пакеты — реальные пакеты из `/shared/` (или stub, если ещё нет)
   - Логирование — реальная библиотека и паттерн
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-conventions.py
   ```

### Сценарий 7: Изменены требования по уровням критичности

**Когда:** Изменены матрицы отказоустойчивости или логирования по уровням критичности, добавлены новые критерии в таблицы.

**Шаги:**

1. **Обновить таблицу(ы)** — внести изменения в таблицу отказоустойчивости и/или таблицу логирования в секции «Требования по уровням критичности».
2. **Проверить вводный абзац** — убедиться, что пояснение и ссылка на `{svc}.md` актуальны.
3. **Валидация.**
