## 1. Назначение

`docs/.system/infrastructure.md` — единый документ с платформенной информацией. Один файл — весь инфраструктурный контекст: как поднять систему, где что слушает, как сервисы находят друг друга, куда смотреть при ошибках.

**Содержит:** платформу — локальный запуск, сервисы и порты, хранилища, брокеры, service discovery, окружения, мониторинг, секреты.

**НЕ содержит:** архитектуру системы (карта сервисов, связи, потоки) — это в `overview.md`. Конвенции API и shared-интерфейсы — это в `conventions.md`. Per-service детали — это в `{svc}.md`.

**Целевая аудитория:** LLM-разработчик, которому нужно запустить систему для разработки и тестирования, подключиться к хранилищам, настроить service-to-service вызовы, найти ошибку в логах.

**Триггеры обновления:** при добавлении/удалении сервиса — обновить секции Сервисы и порты, Хранилища (если у сервиса своя БД), Service Discovery (если сервис вызывает другие); при добавлении/удалении хранилища — обновить секцию Хранилища; при смене окружения — обновить секцию Окружения.

## 2. Секции

8 обязательных секций в фиксированном порядке. Имена секций фиксированы — использовать точные названия из таблицы. Дополнительные секции добавляются после Секретов, если содержание не покрыто ни одной из 8 обязательных секций (примеры: Rate Limiting, CDN, Feature Flags).

| # | Секция | Обязательность | Содержание |
|---|--------|---------------|-----------|
| 1 | Локальный запуск | Обязательна | docker-compose, make-команды, порядок запуска, .env.example |
| 2 | Сервисы и порты | Обязательна | Таблица: сервис, хост, порт, URL, healthcheck |
| 3 | Хранилища | Обязательна | Для каждого хранилища: хост, порт, база, пользователь, env-переменная |
| 4 | Message Broker | Обязательна | URL, каналы (exchange/queue/topic), формат сообщений |
| 5 | Service Discovery | Обязательна | Как сервисы находят друг друга: DNS, env vars, URL |
| 6 | Окружения | Обязательна | Таблица dev/staging/prod: отличия, ресурсы, ограничения |
| 7 | Мониторинг и логи | Обязательна | Где логи, как искать, формат, метрики |
| 8 | Секреты | Обязательна | Откуда берутся, что не коммитить, env-переменные |

**Пустые секции:** если секция не применима — stub-текст в курсиве по шаблону: `*{Название раздела} не используется.*`. Правило применяется ко всем секциям документа, включая дополнительные. Удалять секцию запрещено.

### Локальный запуск

**Обязательные элементы:**
- Ссылка на Makefile как SSOT команд
- Порядок запуска — описать явно, даже если произвольный. Если зависимостей нет — написать: "Порядок запуска не имеет значения, сервисы запускаются параллельно."
- Инструкция по `.env.example` → `.env`

**Запрещено:** дублировать полный список make-команд — ссылаться на `make help`.

### Сервисы и порты

**Обязательная таблица:**

| Колонка | Описание |
|---------|----------|
| Сервис | Имя сервиса |
| Хост (docker) | Имя контейнера в docker-compose |
| Хост (local) | localhost |
| Порт | Номер порта |
| URL | Полный URL для доступа |
| Healthcheck | Эндпоинт проверки здоровья |

Если используется gateway/reverse proxy — указать маршруты проксирования после таблицы.

### Хранилища

Для каждого хранилища — h3-подсекция с именем типа и назначением.

**Обязательные элементы каждого хранилища:**
1. Таблица параметров: Хост, Порт, База/Namespace, Пользователь, Пароль — с dev-значениями и env-переменными
2. Connection string — шаблон подключения с фактическими именами env-переменных из таблицы (например, `postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@...`)

**Если хранилище используется несколькими сервисами** — указать все базы в таблице (по одной строке на базу).

**Если хранилищ нет** (stateless-проект) — написать stub-текст: `*Внешние хранилища не используются.*`

### Message Broker

Для каждого брокера — h3-подсекция с типом и назначением. Если брокеров несколько — одна h3-подсекция на каждый брокер.

**Обязательные элементы:**
1. Таблица параметров: URL, Management UI — с dev-значениями и env-переменными. Если Management UI доступен только в dev — указать `*недоступно в staging/prod*`
2. Таблица каналов: Канал, Тип, Издатели, Подписчики, Описание
3. Формат сообщений: один code-блок JSON на каждый канал. Если все каналы используют одинаковую обёртку — один блок с пометкой "общий формат для всех каналов"

**Если брокера нет** — написать stub-текст: `*Брокер сообщений не используется.*`

### Service Discovery

**Обязательная таблица:**

| Колонка | Описание |
|---------|----------|
| Сервис-источник | Кто вызывает |
| Сервис-цель | Кого вызывают |
| Механизм | env var / DNS / hardcoded |
| Значение (dev) | URL для dev-окружения |
| Env-переменная | Имя переменной с URL |

Вводный абзац — краткое описание механизма discovery (env vars, DNS, service mesh).

**Если сервис один** (нет межсервисных вызовов) — написать stub-текст: `*Межсервисное взаимодействие отсутствует.*`

### Окружения

**Обязательная таблица dev/staging/prod:**

| Колонка | Описание |
|---------|----------|
| Параметр | Что сравнивается |
| dev | Значение для dev |
| staging | Значение для staging |
| prod | Значение для prod |

**Обязательные строки** (присутствуют всегда, даже если значения одинаковы — явная одинаковость лучше неявной): Реплики сервисов, БД, Секреты, Домен, Логирование.

**Если staging отсутствует** — колонку staging включить со значением `—` в каждой строке.

### Мониторинг и логи

**Обязательные элементы:**
1. Формат логов (JSON structured / plain)
2. Таблица "Где смотреть" — по окружениям: Логи, Метрики
3. Стандарт логирования: ссылка на `conventions.md` секция Логирование

**Опциональные элементы:**
- Alerting-правила (prod)

### Секреты

**Обязательные элементы:**
1. Правило: секреты НИКОГДА не коммитятся
2. Таблица секретов: Секрет, Env-переменная, Источник (dev), Источник (prod)

## 3. Принципы

> **Один файл — вся платформа.** Вся инфраструктурная информация в одном документе. Не разбивать по файлам. Если сервисов более 10 — рассмотреть разбивку по доменам.

> **Dev-first.** Значения по умолчанию — для dev-окружения. Staging/prod — отличия от dev.

> **Env-переменные — единый механизм конфигурации.** Каждый параметр, зависящий от окружения, привязан к env-переменной. Хардкод запрещён.

> **SSOT конфигурации.** `.env.example` и `docker-compose.yml` — источник правды значений. `infrastructure.md` синхронизируется с ними, не является источником конфигурации.

> **Не дублировать Makefile.** Команды запуска — через ссылку на `make help`. infrastructure.md описывает порядок и зависимости, а не повторяет список команд.

> **Не дублировать conventions.md.** Стандарт логирования (уровни, формат, поля) — в `conventions.md`. В infrastructure.md — только "где смотреть логи" и ссылка на conventions.md.

> **Расширяемость.** 8 обязательных секций в фиксированном порядке. Дополнительные секции — после Секретов. Новую секцию не создавать, если содержание можно разместить в существующей.

> **Пустые секции.** Если секция не применима — stub-текст в курсиве: `*{Название раздела} не используется.*`. Удалять секцию запрещено.

## 4. Требования по уровням критичности

Требования к инфраструктуре зависят от уровня критичности сервиса. Уровень критичности определяется в `{svc}.md` (поле `criticality`). Допустимые значения: `critical-high`, `critical-medium`, `critical-low`.

### Мониторинг и алертинг

Определяет требования к наблюдаемости сервиса: SLA, скорость реакции на инциденты, каналы оповещения и частоту проверок здоровья.

| Критерий | critical-high | critical-medium | critical-low |
|----------|--------------|-----------------|--------------|
| SLA (uptime) | 99.9% (≤8.7ч простоя/год) | 99.5% (≤43.8ч/год) | 99% (≤87.6ч/год) |
| Время реакции на инцидент | ≤15 мин | ≤1 час | ≤4 часа (рабочее время) |
| Алертинг | PagerDuty / SMS, 24/7 | Slack #critical + email | Slack #alerts (рабочие часы) |
| Дашборд | Real-time, выделенный | Общий с приоритетной панелью | Общий |
| Health-check интервал | ≤30 сек | ≤1 мин | ≤5 мин |

### Деплой и откат

Определяет стратегию развёртывания, условия автоматического отката, допустимые окна деплоя и требования к согласованию.

| Критерий | critical-high | critical-medium | critical-low |
|----------|--------------|-----------------|--------------|
| Стратегия деплоя | Blue-green, zero downtime | Canary (5% → 25% → 100%) | Rolling update |
| Автооткат | Да, при >1% ошибок | Да, при >5% ошибок | Ручной |
| Окно деплоя | Только low-traffic часы | Рабочие часы, не пятница | Любое время |
| Обязательный smoke test | Да, автоматический | Да, автоматический | Нет (рекомендуется) |
| Approval для деплоя | 2 reviewers + ops | 1 reviewer | Self-merge допустим |

### Данные и восстановление

Определяет целевые показатели восстановления данных (RPO/RTO), политику резервного копирования и требования к плану аварийного восстановления.

| Критерий | critical-high | critical-medium | critical-low |
|----------|--------------|-----------------|--------------|
| RPO (Recovery Point Objective) | ≤1 час | ≤4 часа | ≤24 часа |
| RTO (Recovery Time Objective) | ≤30 мин | ≤2 часа | ≤8 часов |
| Бэкапы | Ежечасные + WAL streaming | Ежедневные + ежечасные инкременты | Ежедневные |
| Disaster Recovery план | Обязателен, тестируется ежеквартально | Обязателен | Рекомендуется |

## 6. Пример

Пример для проекта MyApp (4 сервиса: auth, task, notification, admin).

```markdown
---
description: Инфраструктура MyApp — платформа, окружения, хранилища, сервисы.
standard: specs/.instructions/docs/infrastructure/standard-infrastructure.md
---

# Инфраструктура

## Локальный запуск

Все команды определены в [Makefile](/Makefile) (SSOT). Полный список: `make help`.

**Порядок запуска:** PostgreSQL и Redis стартуют первыми (healthcheck), затем сервисы. RabbitMQ стартует параллельно — сервисы с retry до подключения.

**Переменные окружения:** скопировать `.env.example` → `.env`. Описание каждой переменной — в `.env.example`. Для dev-окружения `.env.example` содержит рабочие значения.

## Сервисы и порты

| Сервис | Хост (docker) | Хост (local) | Порт | URL | Healthcheck |
|--------|--------------|-------------|------|-----|-------------|
| auth | auth | localhost | 8001 | http://localhost:8001 | GET /health |
| task | task | localhost | 8002 | http://localhost:8002 | GET /health |
| notification | notification | localhost | 8003 | http://localhost:8003 | GET /health |
| admin | admin | localhost | 8004 | http://localhost:8004 | GET /health |
| gateway | gateway | localhost | 8000 | http://localhost:8000 | GET /health |

**Gateway** проксирует все запросы: `/api/v1/auth/*` → auth:8001, `/api/v1/tasks/*` → task:8002, и т.д.

## Хранилища

### PostgreSQL (основное хранилище данных)

| Параметр | Значение (dev) | Env-переменная |
|----------|---------------|----------------|
| Хост | postgres | `POSTGRES_HOST` |
| Порт | 5432 | `POSTGRES_PORT` |
| База (auth) | myapp_auth | `AUTH_DB_NAME` |
| База (task) | myapp_task | `TASK_DB_NAME` |
| База (notification) | myapp_notification | `NOTIFICATION_DB_NAME` |
| База (admin) | myapp_admin | `ADMIN_DB_NAME` |
| Пользователь | myapp | `POSTGRES_USER` |
| Пароль | из .env | `POSTGRES_PASSWORD` |

**Connection string:** `postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@${POSTGRES_HOST}:${POSTGRES_PORT}/${*_DB_NAME}`

Каждый сервис имеет свою базу данных. Кросс-сервисные запросы запрещены — только через API.

### Redis (кэш и WS-соединения)

| Параметр | Значение (dev) | Env-переменная |
|----------|---------------|----------------|
| Хост | redis | `REDIS_HOST` |
| Порт | 6379 | `REDIS_PORT` |
| База (notification) | 0 | `NOTIFICATION_REDIS_DB` |
| База (auth sessions) | 1 | `AUTH_REDIS_DB` |

**Connection string:** `redis://${REDIS_HOST}:${REDIS_PORT}/${*_REDIS_DB}`

## Message Broker

### RabbitMQ (асинхронное взаимодействие между сервисами)

| Параметр | Значение (dev) | Env-переменная |
|----------|---------------|----------------|
| URL | amqp://guest:guest@rabbitmq:5672 | `RABBITMQ_URL` |
| Management UI | http://localhost:15672 (*недоступно в staging/prod*) | — |

**Каналы:**

| Канал | Тип | Издатели | Подписчики | Описание |
|-------|-----|---------|------------|----------|
| system.events | fanout exchange | auth, task, admin | notification | Системные события для уведомлений |

**Формат сообщений** (общий формат для всех каналов):

{```}json
{
  "event": "UserRegistered",
  "timestamp": "ISO8601",
  "source": "auth",
  "data": { "...": "event-specific payload" }
}
{```}

## Service Discovery

Сервисы находят друг друга через env-переменные с URL. В Docker — по имени контейнера, локально — localhost с портом.

| Сервис-источник | Сервис-цель | Механизм | Значение (dev) | Env-переменная |
|----------------|------------|----------|---------------|----------------|
| task | auth | env var | http://auth:8001 | `AUTH_SERVICE_URL` |
| notification | auth | env var | http://auth:8001 | `AUTH_SERVICE_URL` |
| admin | auth | env var | http://auth:8001 | `AUTH_SERVICE_URL` |
| gateway | auth | env var | http://auth:8001 | `AUTH_SERVICE_URL` |
| gateway | task | env var | http://task:8002 | `TASK_SERVICE_URL` |
| gateway | notification | env var | http://notification:8003 | `NOTIFICATION_SERVICE_URL` |
| gateway | admin | env var | http://admin:8004 | `ADMIN_SERVICE_URL` |

## Окружения

| Параметр | dev | staging | prod |
|----------|-----|---------|------|
| Реплики сервисов | 1 | 1 | 2-4 (auto-scale) |
| PostgreSQL | Docker container | AWS RDS (shared) | AWS RDS (HA, Multi-AZ) |
| Redis | Docker container | AWS ElastiCache | AWS ElastiCache (cluster) |
| RabbitMQ | Docker container | AWS MQ | AWS MQ (HA) |
| Секреты | .env файл | AWS Secrets Manager | AWS Secrets Manager |
| Домен | localhost:8000 | staging.myapp.com | myapp.com |
| TLS | нет | Let's Encrypt | ACM |
| Логирование | stdout (docker logs) | CloudWatch | CloudWatch + alerts |

## Мониторинг и логи

**Формат логов:** JSON structured

**Где смотреть:**

| Окружение | Логи | Метрики |
|-----------|------|---------|
| dev | `docker-compose logs -f {svc}` | — |
| staging | AWS CloudWatch → Log Group `/myapp/staging/{svc}` | CloudWatch Metrics |
| prod | AWS CloudWatch → Log Group `/myapp/prod/{svc}` | CloudWatch Metrics + Grafana |

**Стандарт логирования:** см. [conventions.md](../conventions.md) секция Логирование.

**Alerting (prod):**
- ERROR rate > 1% за 5 минут → Slack #alerts
- Response time p99 > 2s → Slack #alerts
- Healthcheck fail → PagerDuty

## Секреты

**Правило:** секреты НИКОГДА не коммитятся. Только `.env.example` с пустыми значениями.

| Секрет | Env-переменная | Источник (dev) | Источник (prod) |
|--------|---------------|----------------|-----------------|
| Пароль PostgreSQL | `POSTGRES_PASSWORD` | .env | AWS Secrets Manager |
| JWT signing key | `JWT_SECRET_KEY` | .env | AWS Secrets Manager |
| RabbitMQ credentials | `RABBITMQ_URL` | .env (guest:guest) | AWS Secrets Manager |
| API keys внешних сервисов | `{SERVICE}_API_KEY` | .env | AWS Secrets Manager |
```
