## Принципы

> **Синхронизация с конфигурацией.** `.env.example` и `docker-compose.yml` — SSOT значений. После изменения infrastructure.md проверить соответствие реальным файлам.

> **Каскадное обновление.** Добавление сервиса затрагивает минимум 3 секции: Сервисы и порты, Хранилища (если своя БД), Service Discovery (если вызывает другие сервисы).

> **Валидация в конце.** После любого изменения — запустить валидацию. Не считать обновление завершённым без прохождения.

## Шаги

### Шаг 1: Определить триггер

Определить, какое изменение произошло. Найти соответствующий триггер в [таблице](#триггеры-обновления) и перейти к указанному сценарию.

### Шаг 2: Выполнить обновление по сценарию

Выполнить шаги из соответствующего [сценария](#сценарии).


## Сценарии

### Сценарий 1: Добавлен новый сервис

**Когда:** В проекте появился новый сервис (новый контейнер в docker-compose, новая папка в `/src/`).

**Шаги:**

1. **Обновить таблицу «Сервисы и порты»** — добавить строку с хостом, портом, URL, healthcheck.
2. **Обновить секцию «Хранилища»** — если у сервиса своя база данных, добавить строку в таблицу соответствующего хранилища.
3. **Обновить таблицу «Service Discovery»** — если сервис вызывает другие сервисы или его вызывают, добавить строки.
4. **Обновить таблицу «Секреты»** — если у сервиса есть свои секреты (API-ключи, пароли).
5. **Проверить overview.md** — новый сервис должен быть отражён в карте сервисов (`overview.md`).
6. **Валидация.**

### Сценарий 2: Добавлено хранилище

**Когда:** В проекте появилось новое хранилище (новая БД, кэш, object storage).

**Шаги:**

1. **Добавить h3-подсекцию** в секцию Хранилища:
   ```
   ### {StorageType} ({назначение})
   | Параметр | Значение (dev) | Env-переменная |
   ...
   **Connection string:** ...
   ```
2. **Обновить таблицу «Окружения»** — добавить строку для нового хранилища (dev / staging / prod).
3. **Обновить таблицу «Секреты»** — добавить пароль/credentials хранилища.
4. **Валидация.**

### Сценарий 3: Изменена конфигурация хранилища

**Когда:** Изменены порты, имена баз, env-переменные или connection string существующего хранилища.

**Шаги:**

1. **Обновить таблицу параметров** — изменить затронутые строки.
2. **Обновить connection string** — если изменился формат или переменные.
3. **Проверить `.env.example`** — убедиться что env-переменные в документе совпадают с реальными.
4. **Валидация.**

### Сценарий 4: Добавлен/удалён Message Broker

**Когда:** В проекте появился или удалён брокер сообщений.

**Шаги (добавление):**

1. **Заменить stub-текст** на h3-подсекцию с параметрами брокера.
2. **Добавить таблицу каналов** — для каждого канала/очереди/топика.
3. **Добавить формат сообщений** — code-блок JSON.
4. **Обновить таблицу «Окружения»** — добавить строку для брокера.
5. **Обновить таблицу «Секреты»** — если есть credentials.
6. **Валидация.**

**Шаги (удаление):**

1. **Удалить h3-подсекцию** из секции Message Broker.
2. **Если не осталось брокеров** — добавить stub-текст: `*Брокер сообщений не используется.*`
3. **Обновить таблицу «Окружения»** — удалить строку брокера.
4. **Валидация.**

### Сценарий 5: Изменены окружения

**Когда:** Добавлено новое окружение, изменена инфраструктура существующего (миграция на другой облачный провайдер, смена механизма секретов).

**Шаги:**

1. **Обновить таблицу «Окружения»** — изменить затронутые ячейки.
2. **Обновить таблицу «Мониторинг и логи»** — если изменились инструменты мониторинга.
3. **Обновить таблицу «Секреты»** — если изменился источник секретов.
4. **Валидация.**

### Сценарий 6: Первичное заполнение (из визуализации)

**Когда:** infrastructure.md содержит визуализацию — пример из стандарта (MyApp) с пометкой «Это визуализация». Файл соответствует формату стандарта, но содержание не отражает реальный проект.

**Шаги:**

1. **Прочитать стандарт:** [standard-infrastructure.md](./standard-infrastructure.md) — формат, секции, шаблон.
2. **Удалить blockquote-пометку** `> **Это визуализация.**...` в начале файла.
3. **Заменить содержание секций на актуальные данные проекта** — сохранить структуру (frontmatter, порядок секций, формат таблиц), заменить содержание:
   - Локальный запуск — реальный порядок запуска, зависимости
   - Сервисы и порты — реальные сервисы из docker-compose
   - Хранилища — реальные хранилища с реальными env-переменными
   - Message Broker — реальный брокер (или stub если нет)
   - Service Discovery — реальные межсервисные вызовы
   - Окружения — реальные dev/staging/prod
   - Мониторинг и логи — реальные инструменты
   - Секреты — реальные env-переменные из `.env.example`
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-infrastructure.py
   ```
