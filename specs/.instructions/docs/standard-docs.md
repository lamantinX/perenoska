## 1. Назначение

Документация для поставки (`specs/docs/`) — рабочий контекст, который LLM-разработчик читает при написании кода. Это **результат** аналитической цепочки, а не сама цепочка.

**Что нужно LLM для написания кода:**

| # | Информация | Источник в docs/ |
|---|-----------|-----------------|
| 1 | Мой сервис — API, data model, код, зависимости | `{svc}.md` |
| 2 | Контракты с соседями — форматы, протоколы | `{svc}.md` сервиса-провайдера (того, кто предоставляет API) |
| 3 | Runtime-сценарии — что происходит при X | `{svc}.md` → секция "Потоки" |
| 4 | Обзор системы — кто есть, как связан | `.system/overview.md` |
| 5 | Конвенции API и shared-интерфейсы | `.system/conventions.md` |
| 6 | Инфраструктура — deployment, networking | `.system/infrastructure.md` |
| 7 | Стратегия тестирования | `.system/testing.md` |
| 8 | Стандарты технологий — naming, паттерны | `.technologies/standard-{tech}.md` |

**Что НЕ входит в docs/:**

- Аналитические документы (дискуссии, проектирование, анализ влияния) — `specs/analysis/`
- Тестовые сценарии и задачи конкретного изменения — `specs/analysis/`
- Сам код — `/src/`, `/shared/`, `/tests/`
- Команды запуска — `Makefile` (SSOT для всех команд)

### Разграничение docs/ и analysis/

| Артефакт | docs/ | analysis/ |
|----------|-------|-----------|
| Описание сервиса (API, data model, код) | `{svc}.md` | — |
| Архитектурный обзор системы | `.system/overview.md` | — |
| Конвенции API и shared-интерфейсы | `.system/conventions.md` | — |
| Инфраструктура (deployment, networking) | `.system/infrastructure.md` | — |
| Стратегия тестирования (кросс-сервисная) | `.system/testing.md` | — |
| Стандарты технологий | `.technologies/standard-{tech}.md` | — |
| Дискуссия по изменению | — | `analysis/0001-...` |
| Проектирование (распределение ответственностей) | — | `analysis/0001-...` |
| Тестовые сценарии конкретного PR/фичи | — | `analysis/0001-...` |
| Задачи на разработку | — | `analysis/0001-...` |

**Правило:** docs/ содержит **постоянное описание** системы. Analysis/ содержит **процесс принятия решений** по конкретному изменению. Аналитическая цепочка порождает обновления docs/, но сами аналитические документы в docs/ не хранятся.

## 2. Структура

```
specs/docs/
├── README.md                      # Индекс сервисов + навигация
├── notification.md                # Per-service: всё про notification
├── auth.md                        # Per-service: всё про auth
├── gateway.md                     # Per-service: всё про gateway
├── frontend.md                    # Per-service: всё про frontend
├── .system/                       # Кросс-сервисные документы
│   ├── overview.md                # Архитектура: связи, потоки, контекстная карта
│   ├── conventions.md             # Конвенции API + shared-интерфейсы
│   ├── infrastructure.md          # Платформа: deployment, networking, мониторинг
│   └── testing.md                 # Стратегия тестирования
└── .technologies/                 # Per-tech стандарты кодирования
    ├── standard-postgresql.md     # Конвенции PostgreSQL
    └── standard-redis.md          # Конвенции Redis
```

**Служебные папки** используют точку-префикс (`.system/`, `.technologies/`) — они не отображаются при обычном просмотре и отделены от per-service файлов.

## 3. Принципы

**Документы — для разработчика, не для процесса.** Docs/ содержит только то, что LLM реально читает при написании кода. Промежуточные артефакты планирования (дискуссии, анализ влияния, проектирование) — в `specs/analysis/`.

**Docs/ описывает текущее состояние, не предписывает желаемое.** При расхождении docs/ и кода — актуализировать docs/. Желаемые изменения системы описываются в `specs/analysis/`, не в docs/.

**Один сервис — один файл.** Per-service документ — это `docs/{svc}.md`. Подпапки для сервиса не создаются.

**Провайдер владеет контрактом.** API-контракт (endpoint, формат запроса/ответа, протокол) описывается полностью в документе сервиса-провайдера. Провайдер — сервис, который предоставляет API. Подробнее — [§ 5. Межсервисная информация](#5-межсервисная-информация).

**Потребитель ссылается, не дублирует.** Зависимость от чужого API описывается абзацем + ссылкой на раздел провайдера. Дублирование контракта запрещено.

**Makefile — SSOT для команд.** Docs/ не дублирует команды запуска. `make help` — полный список. Docs/ может ссылаться на Makefile.

**Два уровня для тестирования и логирования.** Кросс-сервисная стратегия тестирования — в `.system/testing.md`. Кросс-сервисная конвенция логирования — в `.system/conventions.md` (секция "Логирование"). Per-tech детали (фреймворк, фикстуры, настройка логгера) — в `.technologies/standard-{tech}.md`.

**Актуализация docs/ при изменении кода.** Обновление specs/docs/ выполняется двумя шагами: (1) `/docs-sync` после аналитической цепочки — service-agent (per-service docs), technology-agent (per-tech стандарты), system-agent mode=sync (overview.md); (2) `/chain-done` при завершении — system-agent mode=done (все 4 .system/ файла из реального кода). Обновление docs/ — автоматизировано через агентов.

## 4. Типы документов

Каждый тип документа имеет своё назначение и формат. Per-document стандарты ещё не созданы — будут созданы из объектных драфтов. До их создания использовать структуру секций, описанную ниже в этом разделе.

**Per-document стандарты и их источники:**

| Тип документа | Стандарт (будет создан) | Драфт-источник |
|---------------|------------------------|---------------|
| README.md | `specs/.instructions/docs/standard-readme.md` | [sdd-docs-readme.md](/.claude/drafts/2026-02-19-sdd-docs-readme.md) |
| {svc}.md | `specs/.instructions/docs/standard-service.md` | [sdd-docs-service.md](/.claude/drafts/2026-02-19-sdd-docs-service.md) |
| overview.md | `specs/.instructions/docs/standard-overview.md` | [sdd-docs-overview.md](/.claude/drafts/2026-02-19-sdd-docs-overview.md) |
| conventions.md | `specs/.instructions/docs/standard-conventions.md` | [sdd-docs-conventions.md](/.claude/drafts/2026-02-19-sdd-docs-conventions.md) |
| infrastructure.md | `specs/.instructions/docs/standard-infrastructure.md` | [sdd-docs-infrastructure.md](/.claude/drafts/2026-02-19-sdd-docs-infrastructure.md) |

### Индекс сервисов

**Файл:** `docs/README.md`

**Назначение:** Точка входа в docs/. Таблица всех сервисов с кратким описанием, навигация к .system/ и .technologies/.

**Содержит:**
- Таблица сервисов (имя, описание, технологии, ссылка на `{svc}.md`)
- Ссылки на системные документы (`.system/`)
- Ссылки на стандарты технологий (`.technologies/`)
- Дерево docs/

**НЕ содержит:** Архитектурное описание системы (это в `overview.md`).

### Сервисный документ

**Файл:** `docs/{svc}.md`

**Назначение:** Полное описание одного сервиса — всё, что LLM-разработчику нужно для работы с этим сервисом.

**Секции (10):**

| # | Секция | Уровень детализации | Содержание |
|---|--------|-------------------|-----------|
| 1 | Назначение | Навигационный | Что делает сервис, зона ответственности |
| 2 | API контракты | Имплементационный | Endpoints, request/response schema, status codes, auth, паттерн, протокол |
| 3 | Data Model | Имплементационный | Таблицы/коллекции: колонки с типами, PK/FK, индексы, constraints |
| 4 | Потоки | Архитектурный | Runtime-сценарии: событие → обработка → результат |
| 5 | Code Map | Навигационный | Tech Stack (ссылки на standard-{tech}.md), пакеты/модули, точки входа |
| 6 | Зависимости | Навигационный | От кого зависит (абзац + ссылка на API провайдера), DDD-паттерн связи |
| 7 | Доменная модель | Архитектурный | Агрегаты, доменные события, инварианты |
| 8 | Границы автономии LLM | Навигационный | Три уровня: Свободно, Флаг, CONFLICT |
| 9 | Planned Changes | Навигационный | Ссылки на активные analysis/ документы, затрагивающие сервис |

**Управление статусами:** [`chain_status.py`](../.scripts/chain_status.py) — SSOT-модуль для переходов статусов analysis chain.
| 10 | Changelog | Навигационный | История изменений сервиса, ссылки на analysis/ |

**Три уровня детализации:**
- **Имплементационный** — request/response с типами, DDL с индексами. LLM может писать код напрямую.
- **Архитектурный** — runtime-сценарии, доменные модели. LLM понимает контекст решений.
- **Навигационный** — ссылки, списки, краткие описания. LLM знает куда смотреть.

**Файл:** `docs/.system/overview.md`

**Назначение:** Системная архитектура — карта сервисов, связи, сквозные потоки данных, контекстная карта доменов, shared-код.

**Секции (6):**

| # | Секция | Содержание |
|---|--------|-----------|
| 1 | Назначение системы | Что делает система в целом |
| 2 | Карта сервисов | Mermaid-диаграмма: сервисы, хранилища, внешние системы |
| 3 | Связи между сервисами | Таблица: кто → кого, протокол, назначение |

**Секции (7):**

| # | Секция | Содержание |
|---|--------|-----------|
| 1 | Формат ответов | Единый JSON-envelope для всех сервисов |
| 2 | Формат ошибок | Структура ошибок, коды, HTTP-статусы |
| 3 | Пагинация | Cursor/offset, заголовки, лимиты |

| 5 | Версионирование API | Стратегия, URL-схема, обратная совместимость |

### Стандарт технологии

**Файл:** `docs/.technologies/standard-{tech}.md`

**Назначение:** Конвенции кодирования для конкретной технологии. Читается LLM при написании кода с этой технологией.

**Секции (8):**

| # | Секция | Содержание |
|---|--------|-----------|
| 1 | Версия и настройка | Версия технологии, установка, конфигурация |
| 2 | Именование | Naming conventions: таблицы, колонки, ключи |
| 3 | Паттерны кода | Типовые конструкции с примерами |
| 4 | Анти-паттерны | Что запрещено с объяснением почему |
| 5 | Структура файлов | Где размещать код технологии |
| 6 | Валидация | Ссылка на скрипт `validate-{tech}.py` |
| 7 | Тестирование | Per-tech: фреймворк, фикстуры, типовые тесты |
| 8 | Логирование | Per-tech: настройка логгера, события, уровни |

**Цепочка автоматизации:** `standard-{tech}.md` (конвенции) → `validate-{tech}.py` (скрипт) → pre-commit hook → `.claude/rules/{tech}.md` (автозагрузка при работе с файлами).

## 5. Межсервисная информация

Некоторая информация межсервисная по природе — контракты, топология, инфраструктура. Правила распределения:

| Информация | Где описывается | Кто владеет |
|------------|----------------|-------------|
| API-контракт (endpoint, формат, протокол) | `docs/{провайдер}.md` → секция "API контракты" | Сервис-провайдер |
| Зависимость от чужого API | `docs/{потребитель}.md` → секция "Зависимости" | Сервис-потребитель |
| Топология, связи, потоки | `docs/.system/overview.md` | Системный документ |
| Инфраструктура, deployment | `docs/.system/infrastructure.md` | Платформенный документ |
| Доменная модель | `docs/{svc}.md` → секция "Доменная модель" | Per-service |
| Контекстная карта доменов | `docs/.system/overview.md` → секция "Контекстная карта" | Системный документ |

**Терминология:** Провайдер — сервис, который предоставляет API (публикует endpoint). Потребитель — сервис, который вызывает API провайдера.

### Контракт у провайдера

Провайдер описывает контракт полностью:

```markdown
<!-- docs/auth.md -->
## API контракты

### POST /api/v1/auth/validate
Валидация JWT-токена. Принимает Bearer token в заголовке.
Возвращает: `{ valid: boolean, user_id: string }`
```

### Зависимость у потребителя

Потребитель описывает зависимость абзацем + ссылкой на раздел провайдера:

```markdown
<!-- docs/notification.md -->
## Зависимости

### auth — валидация JWT
Notification использует auth-сервис для валидации JWT-токенов при WebSocket-подключении
и REST-запросах. См. [POST /api/v1/auth/validate](auth.md#post-apiv1authvalidate).
```

**Запрещено:** Дублировать контракт в документе потребителя. Дублирование создаёт рассинхронизацию.

**При изменении контракта провайдера:** Обновить `docs/{провайдер}.md`. Ссылки в документах потребителей обновлять не нужно — они ссылаются на раздел (anchor), не дублируют содержимое. Если anchor изменился — обновить ссылки у потребителей.

## 6. Именование

### Файлы сервисов

| Правило | Пример |
|---------|--------|
| Имя файла = имя папки сервиса в `/src/` | `notification.md` (= `/src/notification/`) |
| Kebab-case | `user-profile.md` |
| Латиница, нижний регистр | `gateway.md` |
| Без суффикса `-service` | `auth.md`, не `auth-service.md` |

**Откуда берётся имя:** Имя сервисного файла совпадает с именем папки сервиса в `/src/`. Если сервис называется `/src/notification/` — файл `docs/notification.md`.

### Файлы технологий

| Правило | Пример |
|---------|--------|
| Префикс `standard-` | `standard-postgresql.md` |
| Имя = имя технологии (lowercase) | `standard-redis.md` |

### Системные документы

Фиксированные имена файлов:

| Файл | Назначение |
|------|-----------|
| `overview.md` | Архитектура системы |
| `conventions.md` | Конвенции и shared-интерфейсы |
| `infrastructure.md` | Платформа и окружения |
| `testing.md` | Стратегия тестирования |

## 7. Жизненный цикл

### Минимальный стартовый набор

При инициализации docs/ для нового проекта создаются все структурные элементы, чтобы ни одна секция не была пустой:

**Обязательные документы:**

| Документ | Назначение |
|----------|-----------|
| `README.md` | Индекс (таблица сервисов, навигация) |
| `.system/overview.md` | Архитектура системы |
| `.system/conventions.md` | Конвенции API и shared-интерфейсы |
| `.system/infrastructure.md` | Платформа и окружения |
| `.system/testing.md` | Стратегия тестирования |

**Примеры (для наглядности и шаблона):**

| Документ | Назначение |
|----------|-----------|
| `example.md` | Пример сервисного документа — демонстрирует все 10 секций |
| `.technologies/standard-example.md` | Пример per-tech стандарта — демонстрирует все 8 секций |

**Шаблонные per-tech стандарты (создаются при init):**

| Документ | Назначение |
|----------|-----------|
| `.technologies/standard-openapi.md` | Конвенции OpenAPI 3.1 — формат REST-контрактов в `shared/contracts/openapi/` |
| `.technologies/standard-protobuf.md` | Конвенции Protobuf v3 — формат gRPC-контрактов в `shared/contracts/protobuf/` |
| `.technologies/standard-asyncapi.md` | Конвенции AsyncAPI 3.0 — формат event-схем в `shared/events/` |

Примеры (`example.md`, `standard-example.md`) содержат заполненные секции с реалистичными данными. Служат шаблоном при создании новых сервисных и per-tech документов. Шаблонные per-tech стандарты (OpenAPI, Protobuf, AsyncAPI) описывают универсальные конвенции контрактных технологий — создаются при init, не при analysis chain. Per-service и per-tech файлы для реальных сервисов/технологий добавляются по мере их появления.

### Удаление сервиса

При удалении сервиса из системы:

1. Удалить `docs/{svc}.md`
2. Обновить `docs/README.md` — убрать строку из таблицы сервисов
3. Обновить `docs/.system/overview.md` — убрать сервис с карты и из связей
4. Проверить ссылки в документах потребителей (секции "Зависимости") — убрать или перенаправить

### Валидация

Скрипт валидации проверяет наличие обязательных документов docs/:

```bash
python specs/.instructions/.scripts/validate-docs.py
```

Скрипт включён в pre-commit hook. Подробнее — [validation-docs.md](./validation-docs.md).
