## Обновление

### Сценарий 1: Добавлен/изменён endpoint

**Когда:** В сервисе добавлен, изменён или удалён REST endpoint, WebSocket или Event channel.

**Шаги:**

1. **API контракты** — добавить/обновить/удалить h3-подсекцию endpoint-а. Формат — из [standard-service.md § 3 → API контракты](./standard-service.md#api-контракты).
2. **Потоки** — если endpoint участвует в описанных потоках, обновить шаги. Если это новая точка входа — рассмотреть добавление нового потока.
3. **Границы автономии LLM** — при изменении публичного API, уровень = CONFLICT.
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-service.py specs/docs/{svc}.md
   ```

### Сценарий 2: Изменена Data Model

**Когда:** В сервисе добавлена/изменена/удалена таблица, колонка, индекс.

**Шаги:**

1. **Data Model** — обновить h3-подсекцию таблицы/коллекции/ключа. Формат — из [standard-service.md § 3 → Data Model](./standard-service.md#data-model).
2. **Доменная модель** — если изменение затрагивает агрегаты или инварианты, обновить.
3. **Границы автономии LLM** — при изменении Data Model, уровень = CONFLICT.
4. **Валидация.**

### Сценарий 3: Изменены зависимости

**Когда:** Сервис стал зависеть от нового сервиса или перестал зависеть от существующего.

**Шаги:**

1. **Зависимости** — добавить/обновить/удалить h3-подсекцию зависимости. Формат — из [standard-service.md § 3 → Зависимости](./standard-service.md#зависимости).
2. **Вводный абзац Зависимостей** — обновить общую картину (количество, критичность).
3. **Валидация.**

### Сценарий 4: Изменён Tech Stack

**Когда:** Добавлена/удалена/обновлена технология в сервисе.

**Шаги:**

1. **Code Map → Tech Stack** — обновить таблицу технологий.
2. **Code Map → Пакеты** — если новая технология требует новых пакетов, обновить.
3. **Ссылки на standard-{tech}.md** — если создан per-tech стандарт, добавить ссылку. Если файл не создан — оставить `—`.
4. **Валидация.**

### Сценарий 5: Завершён analysis/

**Когда:** analysis/ документ переведён в статус DONE. Выполняется `/chain-done`.

**Шаги:**

1. **Planned Changes** — удалить весь chain-блок: от `<!-- chain: NNNN-{topic} -->` до `<!-- /chain: NNNN-{topic} -->` (включая оба тега).
2. **Changelog** — добавить запись в начало списка:
   ```markdown
   - **[analysis/{NNNN}-{slug}](../analysis/{NNNN}-{slug}/)** | DONE {дата}
     {Краткое описание. Затрагивало: {что}.}
   ```
3. **Обновить затронутые секции** — если analysis/ менял API, Data Model и т.д., внести соответствующие изменения в секции по сценариям 1-4.
4. **Валидация.**

### Сценарий 6: Добавлен новый analysis/

**Когда:** Создан новый analysis/ документ, затрагивающий сервис. Выполняется `/docs-sync` (service-agent).

**Шаги:**

1. **Planned Changes** — добавить запись, обёрнутую в chain-маркер:
   ```markdown
   <!-- chain: NNNN-{topic} -->
   - **[analysis/{NNNN}-{slug}](../analysis/{NNNN}-{slug}/)**
     {Краткое описание. Затрагивает: {что}.}
   <!-- /chain: NNNN-{topic} -->
   ```
2. **Валидация.**

### Сценарий 7: Изменена критичность

**Когда:** Пересмотрен уровень критичности сервиса (изменилась бизнес-роль, зависимости, нагрузка).

**Шаги:**

1. **Frontmatter** — обновить поле `criticality` на новое значение (`critical-high` / `critical-medium` / `critical-low`).
2. **Назначение** — обновить обоснование критичности (почему уровень изменился).
3. **overview.md** — обновить колонку «Критичность» в таблице сервисов через [modify-overview.md → Сценарий 7](../overview/modify-overview.md#сценарий-7-изменена-критичность-сервиса).
4. **Пересмотреть требования** — при повышении уровня проверить соответствие инфраструктуры, тестирования и конвенций новому уровню (см. матрицы в стандартах infrastructure.md, testing.md, conventions.md).
5. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-service.py specs/docs/{svc}.md
   ```

### Сценарий 8: Изменены Makefile таргеты

**Когда:** Добавлен, удалён или изменён per-service Makefile таргет.

**Шаги:**

1. **Code Map → Makefile таргеты** — обновить таблицу таргетов (Таргет, Команда, Описание).
2. **Корневой Makefile** — синхронизировать: добавить/удалить/переименовать таргет `{action}-{svc}`.
3. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-service.py specs/docs/{svc}.md
   ```

## Деактивация

**Когда:** Сервис выведен из эксплуатации (удалён из `/src/`).

### Шаг 1: Удалить файл и Makefile таргеты

```bash
rm specs/docs/{svc}.md
```

Удалить per-service таргеты из корневого Makefile (`test-{svc}`, `lint-{svc}`, `build-{svc}`).

### Шаг 2: Обновить docs/README.md

1. Удалить строку из таблицы «Сервисы».
2. Удалить файл из дерева.

### Шаг 3: Обновить overview.md

Обновить через [modify-overview.md → Сценарий 2](../overview/modify-overview.md#сценарий-2-удалён-сервис).

### Шаг 4: Обновить зависимые документы

Проверить, ссылаются ли другие `{svc}.md` на удалённый сервис (секция Зависимости). Если да — обновить или удалить ссылки.

```bash
# Найти ссылки на удалённый сервис
grep -r "{svc}.md" specs/docs/ --include="*.md"
```

### Шаг 5: Валидация

```bash
python specs/.instructions/.scripts/validate-docs-readme-services.py
```

## Миграция

**Когда:** Сервис переименован (`/src/{old-svc}/` → `/src/{new-svc}/`).

### Шаг 1: Переименовать файл

```bash
mv specs/docs/{old-svc}.md specs/docs/{new-svc}.md
```

### Шаг 2: Обновить frontmatter

Обновить `description` в frontmatter файла.

### Шаг 3: Обновить h1

Заменить `# {old-svc}` на `# {new-svc}`.

### Шаг 3.1: Переименовать Makefile таргеты

В корневом Makefile переименовать per-service таргеты:
- `test-{old-svc}` → `test-{new-svc}`
- `lint-{old-svc}` → `lint-{new-svc}`
- `build-{old-svc}` → `build-{new-svc}` (если есть)

Обновить подсекцию "Makefile таргеты" в Code Map документа `{new-svc}.md`.

### Шаг 4: Обновить docs/README.md

1. Обновить строку в таблице «Сервисы» (имя + ссылка).
2. Обновить дерево.

### Шаг 5: Обновить ссылки

См. [Обновление ссылок](#обновление-ссылок).

### Шаг 6: Валидация

```bash
python specs/.instructions/.scripts/validate-docs-service.py specs/docs/{new-svc}.md
python specs/.instructions/.scripts/validate-docs-readme-services.py
```

## Обновление ссылок

### При переименовании файла

Найти все ссылки на старый файл:

```bash
grep -r "{old-svc}.md" specs/docs/ --include="*.md"
grep -r "{old-svc}.md" specs/.instructions/ --include="*.md"
```

Обновить каждую найденную ссылку на `{new-svc}.md`.

### При удалении файла

Найти и удалить/обновить все ссылки на удалённый файл.

## Примеры

### Добавление endpoint-а в notification.md

```bash
# Триггер: добавлен DELETE /api/v1/notifications/{id}
# Сценарий 1: обновить API контракты + Потоки

# 1. Добавить h3 подсекцию в API контракты
# 2. Обновить поток "Удаление уведомления" (если есть) или добавить новый
# 3. Валидация
```

```bash
# user-auth → auth
mv specs/docs/user-auth.md specs/docs/auth.md
# Обновить frontmatter, h1, README, ссылки
grep -r "user-auth.md" specs/docs/ --include="*.md"
python specs/.instructions/.scripts/validate-docs-service.py specs/docs/auth.md
```
