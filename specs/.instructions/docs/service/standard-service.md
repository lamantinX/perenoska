## 1. Назначение

`docs/{svc}.md` — per-service документ. Содержит всё, что LLM-разработчику нужно знать о конкретном сервисе для написания и изменения кода: API контракты, Data Model, потоки данных, структуру пакетов, зависимости, доменную модель и границы автономии. Для каждого сервиса из `overview.md` ДОЛЖЕН существовать один `{svc}.md`. Если файл отсутствует — создать через [create-service.md](./create-service.md) перед началом работы с сервисом.

**Содержит:** назначение сервиса, полные API контракты (request/response), Data Model (колонки с типами, индексы), потоки обработки, Code Map (пакеты, точки входа), зависимости от других сервисов и shared-кода, доменную модель, границы автономии LLM, Planned Changes и Changelog.

**НЕ содержит:** системную архитектуру и карту всех сервисов — это в `overview.md`. Кросс-сервисные конвенции (формат ответов, пагинация, shared-пакеты) — это в `conventions.md`. Per-tech паттерны кодирования — это в `standard-{tech}.md`. Стратегию тестирования — это в `testing.md`. Инфраструктуру и CI — это в `infrastructure.md`. Информацию, не относящуюся ни к одной из 10 секций (SLA, команда, внешние ссылки) — размещать в `README.md` сервиса, не в `{svc}.md`.

**Целевая аудитория:** LLM-разработчик, получивший задачу в рамках конкретного сервиса. Документ должен быть достаточным для написания нового endpoint-а, добавления колонки в таблицу, обработки нового события — без чтения исходного кода.

**Триггеры обновления:** при добавлении/изменении/удалении API endpoint-а — обновить секцию API контракты; при изменении Data Model (колонки, индексы, таблицы) — обновить секцию Data Model; при изменении зависимостей — обновить секцию Зависимости; при изменении Tech Stack — обновить секцию Code Map; при завершении analysis/ — перенести из Planned Changes в Changelog. При любом обновлении — следовать воркфлоу [modify-service.md](./modify-service.md).

**Индексация:** каждый `{svc}.md` должен быть зарегистрирован в `docs/README.md`. При создании или удалении — обновить индекс.

## 2. Уровни детализации

Секции документа имеют разный уровень детализации. Уровень определяет, насколько подробно описывать содержание.

| Уровень | Что значит | Критерий достаточности |
|---------|-----------|----------------------|
| **Имплементационный** | Достаточно для написания кода по аналогии без чтения исходников | Разработчик может добавить новый endpoint / колонку, глядя только на документ |
| **Архитектурный** | Понимание "как работает" без деталей реализации | Разработчик понимает жизненный цикл запроса и бизнес-правила |
| **Навигационный** | Знать что есть и где искать | Разработчик знает куда смотреть и по каким ссылкам переходить |

Присвоение уровня каждой секции:

| Секция | Уровень |
|--------|---------|
| Назначение | Навигационный |
| API контракты | Имплементационный |
| Data Model | Имплементационный |
| Потоки | Архитектурный |
| Code Map | Навигационный |
| Зависимости | Навигационный |
| Доменная модель | Архитектурный |
| Границы автономии LLM | Навигационный |
| Planned Changes | Навигационный |
| Changelog | Навигационный |

## 3. Секции

10 обязательных секций в фиксированном порядке. Имена секций фиксированы — использовать точные названия из таблицы. Порядок секций в файле фиксирован — изменение порядка запрещено. Дополнительные секции не допускаются — любая информация о сервисе размещается в одной из 10 существующих секций.

| # | Секция | Обязательность | Содержание |
|---|--------|---------------|-----------|
| 1 | Назначение | Обязательна | Что делает сервис, зона ответственности |
| 2 | API контракты | Обязательна | Каждый endpoint подробно: path, method, auth, request/response, errors |
| 3 | Data Model | Обязательна | Каждая таблица/коллекция: колонки, типы, constraints, индексы |
| 4 | Потоки | Обязательна | Ключевые runtime-сценарии обработки данных |
| 5 | Code Map | Обязательна | Tech Stack, пакеты, точки входа, внутренние зависимости |
| 6 | Зависимости | Обязательна | От кого зависит, shared-код, DDD-паттерны связей |
| 7 | Доменная модель | Обязательна | Агрегаты, инварианты, доменные события |
| 8 | Границы автономии LLM | Обязательна | Три уровня: Свободно, Флаг, CONFLICT |
| 9 | Planned Changes | Обязательна | Ссылки на активные analysis/ документы |
| 10 | Changelog | Обязательна | История завершённых изменений |

**Пустые секции:** если секция не применима (критерии неприменимости — в описании каждой секции ниже) — stub-текст в курсиве по шаблону: `*{Название раздела} не применимо — {причина}.*` Удалять секцию запрещено.

**Для BFF и API Gateway:** нормально иметь Data Model = stub и Доменная модель = stub при большом разделе Зависимостей. Это корректное применение стандарта для инфраструктурных сервисов.

### Назначение (секция сервиса)

**Обязательные элементы:**
1. Абзац (2-3 предложения) — что делает сервис, зона ответственности, ключевые потребители (frontend, другие сервисы), основной способ взаимодействия (REST, WS, события)
2. **Обоснование критичности** — абзац (1-3 предложения), объясняющий почему сервису присвоен конкретный уровень `criticality` из frontmatter. Указать бизнес-влияние недоступности: что произойдёт если сервис упадёт, кто пострадает, есть ли обходные пути

**Правила:**
- Описание должно быть конкретным: не "обрабатывает данные", а "управляет WebSocket-соединениями, подписывается на системные события через AMQP, доставляет уведомления через WS и REST API"
- Указать основных потребителей и способ их взаимодействия с сервисом
- Обоснование критичности должно быть привязано к бизнес-влиянию, а не к техническим характеристикам

**Неприменимость:** секция не может быть неприменимой — каждый сервис имеет назначение.

### API контракты

**Уровень:** имплементационный — достаточно для написания нового endpoint-а по аналогии.

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — обзор API сервиса: сколько endpoint-ов, какие группы, общий паттерн. Ссылка на [conventions.md](../../../docs/.system/conventions.md) для общих конвенций (формат ответов, ошибок, пагинация, auth)
2. Подсекции (H3) — по одной на каждый endpoint / WebSocket / Event channel

**Формат REST endpoint-а (H3):**

```
### {METHOD} {/api/v1/path}

{Что делает endpoint. Одно предложение.}

- **Auth:** {Bearer JWT / API Key / нет}
- **Паттерн:** {sync/async} | **Протокол:** {REST/JSON, gRPC, GraphQL}

| Параметр | Тип | Обязательный | Описание |
|----------|-----|-------------|----------|
| {param} | {type} | {да/нет (default X)} | {описание} |

**Request:** *(если POST/PUT/PATCH)*
{json code block}

**Response {code}:**
{json code block}

**Errors:** {code} {описание}, {code} {описание}
```

**Формат WebSocket (H3):**

```
### WebSocket {/ws/path}

{Описание.}

- **Auth:** {как передаётся}
- **Паттерн:** async | **Протокол:** WebSocket/JSON

**Подключение:** `ws://host{/ws/path}?{auth_param}={value}`

**Server → Client messages:**
{json code block}
```

**Формат Event channel (H3):**

```
### Event: {channel} ({publisher/subscriber})

{Описание.}

- **Паттерн:** async | **Протокол:** {AMQP/JSON, Kafka, Redis Pub/Sub}
- **Канал:** `{channel_name}`

**Обрабатываемые события:** *(для subscriber)*
| Событие | Издатель | Действие |
|---------|----------|----------|

**Публикуемые события:** *(для publisher)*
| Событие | Когда | Payload |
|---------|-------|---------|
```

**Правила:**
- Разделитель `---` между endpoint-ами для визуального разграничения
- В Response JSON использовать `"type"` для указания типа поля и `"описание"` для пояснений. Для enum — перечислить все значения через `|`
- Таблица параметров обязательна для GET-запросов с query params. Для GET без query params — таблицу параметров не включать. Для POST/PUT/PATCH — Request body в JSON
- Response: всегда JSON-блок. Исключения: `204 No Content` — указать текстом. Если ответ идентичен другому endpoint-у — ссылка: `Response 200: аналогично [GET /path](#get-apipath).`
- Если сервис одновременно publisher и subscriber для одного канала — использовать `({publisher+subscriber})` в заголовке и включить обе таблицы ("Обрабатываемые события" и "Публикуемые события")
- При поддержке нескольких версий API — группировать endpoint-ы по версиям через комментарий в заголовке. Deprecated endpoint-ы помечать: `### GET /api/v1/path *(deprecated — заменён v2)*`

**Неприменимость:** если сервис не предоставляет публичных API (worker, cron) — stub-текст: `*Сервис не предоставляет публичных API — работает как background worker.*`

### Data Model

**Уровень:** имплементационный — достаточно для добавления колонки по аналогии.

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — какие хранилища использует сервис. Для нескольких хранилищ — как разделяют ответственность. Для одного хранилища — что хранит и почему выбрано это хранилище. Абзац обязателен всегда
2. Подсекции (H3) — по одной на каждую таблицу / коллекцию / ключ Redis

**Формат PostgreSQL/SQL таблицы (H3):**

```
### {table_name} (PostgreSQL)

| Колонка | Тип | Constraints | Описание |
|---------|-----|------------|----------|
| {col} | {TYPE} | {PK/FK/NOT NULL/DEFAULT/CHECK/INDEX} | {описание} |

**Индексы:**
- `idx_{table}_{col}` — по {col} ({зачем})

**Особенности:** {TTL, партиционирование — если есть}
```

**Формат MongoDB коллекции (H3):**

```
### {collection_name} (MongoDB)

| Поле | Тип | Required | Описание |
|------|-----|----------|----------|
| _id | ObjectId | auto | Идентификатор |
| {field} | {String/Number/Boolean/Date/Array/Object} | {да/нет} | {описание} |
| {field.nested} | {type} | {да/нет} | {описание вложенного поля} |

**Индексы:**
- `{field}_1` — по {field} ({зачем})
- `{field1}_1_{field2}_-1` — составной ({зачем})

**Особенности:** {TTL index, capped collection, sharding key — если есть}
```

**Формат Redis (H3):**

```
### {key_pattern} (Redis)

| Ключ | Тип | TTL | Описание |
|------|-----|-----|----------|
| {pattern}:{id} | {SET/HASH/STRING/LIST} | {TTL или "при событии"} | {описание} |
```

**Правила:**
- Constraints перечислять через запятую: `PK, NOT NULL, DEFAULT 'value'`
- CHECK constraints — указывать допустимые значения: `CHECK (read, unread)`
- Индексы — обязательно указывать назначение (для какого запроса / какой операции)
- При изменении Data Model — зафиксировать изменение в Changelog через analysis/

**Неприменимость:** если сервис stateless (не хранит данных) — stub-текст: `*Сервис stateless — не использует хранилища данных.*`

### Потоки

**Уровень:** архитектурный — понимание "как данные проходят через сервис".

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — общий паттерн обработки внутри сервиса: какой пакет принимает запросы, куда передаёт, где хранит. Рецепт "как добавить новый функционал" для разработчика
2. Подсекции (H3) — по одной на каждый ключевой сценарий

**Формат сценария (H3):**

```
### {Название сценария}

{code block — пронумерованные шаги}
1. {Актор/сервис} → {модуль}: {действие}
2. {модуль}: {внутренняя обработка}
3. {модуль} → {хранилище}: {операция}
4. {модуль} → {Актор/сервис}: {результат}
```

**Правила:**
- Описывать ключевые сценарии (2-4): по одному на каждую точку входа (REST, WebSocket, Events), плюс самый сложный сценарий с участием внешних зависимостей. Если точек входа меньше 2 — описать все
- В шагах указывать конкретные модули (пакеты) сервиса, не абстрактные "backend" или "service"
- Формат текстовый (пронумерованные шаги в code-блоке), не диаграммы

**Неприменимость:** если сервис — простой CRUD без бизнес-логики (1 точка входа, 1 хранилище, нет событий) — stub-текст: `*Потоки не описываются — сервис реализует простой CRUD.*`

### Code Map

**Обязательные элементы:**
1. Подсекция **Tech Stack** — таблица технологий со ссылками на `standard-{tech}.md`
2. Подсекция **Пакеты** — вводный абзац + таблица пакетов/модулей
3. Подсекция **Точки входа** — список файлов-точек входа
4. Подсекция **Внутренние зависимости** — граф зависимостей между пакетами + рецепт добавления функционала
5. Подсекция **Makefile таргеты** — per-service таргеты для тестирования, линтинга, сборки

**Формат Tech Stack:**

| Технология | Версия | Назначение | Стандарт |
|-----------|--------|-----------|---------|
| {tech} | {ver} | {назначение} | [standard-{tech}.md](../.technologies/standard-{tech}.md) |

**SSOT-зависимость:** Для каждой технологии в таблице Tech Stack ДОЛЖЕН существовать per-tech стандарт [standard-{tech}.md](/specs/docs/.technologies/) — обязательный SSOT для версий и конвенций кодирования. Если стандарт отсутствует — создать через [create-technology.md](/specs/.instructions/docs/technology/create-technology.md).

**Формат Пакеты:**

| Пакет | Назначение | Ключевые модули |
|-------|-----------|----------------|
| `{svc}.{package}` | {назначение} | `{module1}.py`, `{module2}.py` |

**Правила:**
- Tech Stack: ссылки на `standard-{tech}.md` только для реально существующих файлов. Если файл не создан — оставить ячейку "Стандарт" пустой (`—`)
- Пакеты: вводный абзац объясняет принцип организации кода (по точкам входа, по слоям, по доменам)
- Внутренние зависимости: текстовый формат `packageA → packageB`, не диаграммы
- Рецепт "Как добавить новый функционал" — список сценариев добавления (каждый сценарий: тип функционала → конкретные файлы и действия). Минимум 2 сценария: новый endpoint и новый обработчик событий (если сервис использует события)

**Формат Makefile таргеты:**

| Таргет | Команда | Описание |
|--------|---------|----------|
| test | `make test-{svc}` | Unit + integration тесты сервиса |
| lint | `make lint-{svc}` | Линтинг кода сервиса |
| build | `make build-{svc}` | Сборка Docker-образа (если применимо) |

**Правила Makefile таргеты:**
- Таргеты `test` и `lint` обязательны для каждого сервиса
- Таргет `build` — если сервис имеет Docker-образ
- Дополнительные таргеты (seed, migrate) — по необходимости
- Имя таргета = `{action}-{svc}` (kebab-case)

**Неприменимость:** секция не может быть неприменимой — каждый сервис имеет код.

### Зависимости

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — общая картина зависимостей: от скольких сервисов зависит, какие критические, какие некритические
2. Подсекции (H3) — по одной на каждую зависимость (другой сервис или shared-пакет)

**Формат зависимости от другого сервиса (H3):**

```
### {other_svc} — {зачем (2-3 слова)}
{Абзац: что конкретно использует, как подключается, что произойдёт при недоступности.}
Паттерн: **{Conformist/ACL/Published Language}** ({пояснение}).
См. [{METHOD} {path}]({other_svc}.md#{anchor}).
```

**Формат зависимости от shared-пакета (H3):**

```
### shared/{package}/ — {зачем (2-3 слова)}
{Абзац: что использует, как подключить, роль в обработке запроса.}
Роль: consumer. Владелец: {owner_svc}. Интерфейс: [conventions.md](../.system/conventions.md#{anchor}).
```

**Правила:**
- DDD-паттерн связи обязателен для зависимостей от других сервисов: Conformist (полностью следует API провайдера), ACL (Anti-Corruption Layer — адаптирует API), Published Language (общий стандарт взаимодействия)
- Ссылки на конкретные endpoint-ы/секции зависимости — обязательны (глубокие ссылки, не просто имя файла)
- Критичность зависимости: указать что произойдёт при недоступности (сервис не работает / деградация / ничего)

**Неприменимость:** если сервис полностью автономен (нет зависимостей) — stub-текст: `*Сервис не имеет зависимостей от других сервисов.*`

### Доменная модель

**Уровень:** архитектурный — понимание бизнес-правил и инвариантов.

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — какой домен реализует сервис, основная бизнес-сущность, ключевой жизненный цикл
2. Подсекция **Агрегаты** — таблица агрегатов
3. Подсекция **Инварианты** — маркированный список бизнес-правил
4. Подсекция **Доменные события** — таблица событий

**Формат Агрегаты:**

| Агрегат | Описание |
|---------|----------|
| {Name} | {описание, ключевые атрибуты, жизненный цикл} |

**Формат Доменные события:**

| Событие | Описание |
|---------|----------|
| {EventName} | {когда возникает → что запускает} |

**Правила:**
- Инварианты — бизнес-правила, которые всегда истинны. Формулировка: утверждение, не "нельзя" / "запрещено"
- Жизненный цикл агрегата: перечислить состояния через `→` (created → active → archived)
- Доменные события — бизнес-факты, зафиксированные внутри доменного слоя сервиса (DDD-концепция). Они описывают что произошло в домене. Трансляция во внешние Event channel-ы — отдельный слой (описывается в API контрактах). Одно доменное событие может порождать ноль или более внешних событий

**Неприменимость:** если сервис — чистая инфраструктура (gateway, proxy) без бизнес-логики — stub-текст: `*Доменная модель не применима — сервис инфраструктурный.*`

### Границы автономии LLM

**Обязательные элементы:**
1. Маркированный список из трёх уровней: Свободно, Флаг, CONFLICT

**Формат:**

```
- **Свободно:** {что можно менять без согласования}
- **Флаг:** {что требует проверки — может затронуть другие пакеты/тесты}
- **CONFLICT:** {что нельзя менять без обсуждения — API, data model, протоколы}
```

**Правила:**
- Свободно: реализация внутри одного пакета, рефакторинг, оптимизация
- Флаг: изменения, которые могут затронуть тесты или контракты между пакетами
- CONFLICT: изменения публичных API, Data Model (колонки, индексы), протоколов взаимодействия
- При пересечении критериев Флаг и CONFLICT — применять CONFLICT. Любое изменение, затрагивающее публичные контракты или схему данных — CONFLICT, даже если оно также затрагивает тесты

**Неприменимость:** секция не может быть неприменимой — каждый сервис имеет границы.

## 4. Принципы

> **Один файл — весь сервис.** Всё что LLM-разработчику нужно знать о сервисе — в одном документе. Per-tech паттерны — в `standard-{tech}.md`. Системная архитектура — в `overview.md`. Конвенции — в `conventions.md`.

> **Имплементационный уровень для API и Data Model.** Секции API контракты и Data Model должны быть достаточны для написания кода по аналогии. Полный request/response, все колонки с типами — без чтения исходников.

> **Навигационный уровень для остального.** Секции с навигационным уровнем дают ориентировку "что есть и где искать", не дублируя детали из других документов.

> **Не дублировать overview.md.** Системная архитектура, карта сервисов, сквозные потоки — в `overview.md`. В `{svc}.md` — только per-service детали и ссылки.

> **Не дублировать conventions.md.** Формат ответов API, пагинация, обработка ошибок, shared-пакеты — в `conventions.md`. В `{svc}.md` — ссылка в секции API контракты и Зависимости.

> **Не дублировать standard-{tech}.md.** Конкретные фреймворки, паттерны кодирования, конфигурации — в `standard-{tech}.md`. В Code Map — ссылки.

> **Глубокие ссылки.** Ссылки на другие документы — до конкретной секции (якорь), не просто имя файла. `auth.md#post-apiv1authvalidate`, не `auth.md`.

> **Пустые секции.** Если секция не применима (критерии — в описании каждой секции) — stub-текст в курсиве. Удалять секцию запрещено.

> **10 фиксированных секций.** Порядок и набор секций неизменны. Дополнительные секции не допускаются. Информация, не относящаяся ни к одной из 10 секций, не включается в `{svc}.md`.

## 6. Пример

Пример для проекта MyApp — сервис notification (push-уведомления в реальном времени).

`````markdown
---
description: notification — push-уведомления в реальном времени через WebSocket и REST API.
standard: specs/.instructions/docs/service/standard-service.md
criticality: critical-medium
---

# notification

## Назначение

Сервис push-уведомлений в реальном времени. Управляет WebSocket-соединениями,
подписывается на системные события через AMQP, доставляет уведомления через WS и REST API.
Основные потребители: frontend (REST для истории, WebSocket для push) и все сервисы системы
(публикуют события, которые notification преобразует в уведомления пользователям).

**Обоснование критичности:** Уровень `critical-medium` — при недоступности сервиса пользователи не получают push-уведомления в реальном времени, что приводит к значительной деградации UX. Бизнес продолжает работать: основные операции (авторизация, платежи, работа с данными) не зависят от notification. Обходной путь: пользователи могут проверять изменения вручную через интерфейс.

## API контракты

Сервис предоставляет 3 типа интерфейсов: REST API для получения/обновления уведомлений,
WebSocket для real-time push, и AMQP subscriber для приёма системных событий. Все REST
endpoint-ы требуют Bearer JWT и следуют конвенциям из [conventions.md](../.system/conventions.md).

### GET /api/v1/notifications

Список уведомлений текущего пользователя с пагинацией.

- **Auth:** Bearer JWT (обязательно)
- **Паттерн:** sync | **Протокол:** REST/JSON

| Параметр | Тип | Обязательный | Описание |
|----------|-----|-------------|----------|
| limit | int | нет (default 20, max 100) | Количество |
| offset | int | нет (default 0) | Смещение |
| status | string | нет | Фильтр: `read`, `unread` |

**Response 200:**
```json
{
  "items": [
    {
      "id": "uuid",
      "type": "system|user|admin",
      "title": "string",
      "body": "string",
      "status": "read|unread",
      "created_at": "ISO8601"
    }
  ],
  "total": 42,
  "limit": 20,
  "offset": 0
}
```

**Errors:** 401 Unauthorized, 422 Validation Error

---

### PATCH /api/v1/notifications/{id}

Обновление статуса уведомления.

- **Auth:** Bearer JWT (обязательно, владелец уведомления)
- **Паттерн:** sync | **Протокол:** REST/JSON

**Request:**
```json
{ "status": "read" }
```

**Response 200:** аналогично [GET /api/v1/notifications](#get-apiv1notifications).

**Errors:** 401 Unauthorized, 403 Forbidden (не владелец), 404 Not Found

---

### WebSocket /ws/notifications

Push-уведомления в реальном времени.

- **Auth:** JWT передаётся как query param `?token=...` при подключении
- **Паттерн:** async | **Протокол:** WebSocket/JSON

**Подключение:** `ws://host/ws/notifications?token={jwt}`

**Server → Client messages:**
```json
{
  "event": "notification.created",
  "data": {
    "id": "uuid",
    "type": "system",
    "title": "string",
    "body": "string",
    "created_at": "ISO8601"
  }
}
```

**Rate limit:** max 10 одновременных соединений на пользователя.

---

### Event: system.events (subscriber)

Подписка на системные события через message broker.

- **Паттерн:** async | **Протокол:** AMQP/JSON
- **Канал:** `system.events`

**Обрабатываемые события:**
| Событие | Издатель | Действие |
|---------|----------|----------|
| UserRegistered | auth | Создать welcome-уведомление |
| PasswordChanged | auth | Создать security-уведомление |
| AdminAction | admin | Создать admin-уведомление |
| SystemError | * | Создать alert-уведомление |

## Data Model

PostgreSQL — основное хранилище уведомлений (таблица `notifications`). Redis — кэш активных
WebSocket-соединений для быстрого lookup при push. PostgreSQL-данные персистентны с TTL 90 дней,
Redis-данные эфемерны (удаляются при disconnect).

### notifications (PostgreSQL)

| Колонка | Тип | Constraints | Описание |
|---------|-----|------------|----------|
| id | UUID | PK, default gen_random_uuid() | Идентификатор |
| user_id | UUID | NOT NULL, INDEX | Владелец уведомления |
| type | VARCHAR(20) | NOT NULL, CHECK (system, user, admin) | Тип |
| title | VARCHAR(255) | NOT NULL | Заголовок |
| body | TEXT | NOT NULL | Тело |
| status | VARCHAR(10) | NOT NULL, DEFAULT 'unread', CHECK (read, unread) | Статус |
| created_at | TIMESTAMPTZ | NOT NULL, DEFAULT now(), INDEX | Дата создания |

**Индексы:**
- `idx_notifications_user_id` — по user_id (основной запрос)
- `idx_notifications_created_at` — по created_at (TTL cleanup, сортировка)

**TTL:** Cleanup уведомлений старше 90 дней (cron job).

### ws:connections:{user_id} (Redis)

| Ключ | Тип | TTL | Описание |
|------|-----|-----|----------|
| ws:connections:{user_id} | SET of session_id | При disconnect | Активные WebSocket session ID |

## Потоки

Общий паттерн обработки в notification: внешнее событие или запрос приходит через одну из трёх
точек входа (REST API, WebSocket, AMQP consumer). Бизнес-логика в `events.handlers` или
`api.routes` формирует данные, `storage.repository` сохраняет в PostgreSQL, `ws.hub` отправляет
push через WebSocket если пользователь онлайн. Чтобы добавить новый тип уведомления — достаточно
добавить обработчик в `events.handlers` и маппинг события на шаблон уведомления.

### Доставка уведомления (основной сценарий)

```
1. Другой сервис публикует событие в system.events (AMQP)
2. notification.events.handlers получает событие
3. handlers определяет тип → формирует Notification
4. notification.storage.repository сохраняет в PostgreSQL
5. notification.ws.hub находит активные WS по user_id (Redis lookup)
6. hub отправляет JSON-сообщение через WebSocket
```

### Получение истории (REST)

```
1. frontend → GET /api/v1/notifications?status=unread
2. JWT middleware валидирует токен (shared/auth → auth-сервис)
3. notification.api.routes извлекает user_id из токена
4. notification.storage.repository → SELECT с пагинацией
5. Response: JSON массив уведомлений
```

### Пакеты

Сервис разделён на 4 пакета по принципу "точка входа + хранение": три пакета принимают запросы
каждый по своему каналу (REST, WebSocket, AMQP), четвёртый — общее хранилище, которое используют все три.

| Пакет | Назначение | Ключевые модули |
|-------|-----------|----------------|
| `notification.ws` | WebSocket Hub — управление соединениями, отправка сообщений | `hub.py`, `connections.py` |
| `notification.events` | Event Consumer — подписка на system.events, маршрутизация | `handlers.py`, `router.py` |
| `notification.api` | REST API — endpoints, валидация, пагинация | `routes.py`, `schemas.py` |
| `notification.storage` | Persistence — PostgreSQL repository, Redis connections | `repository.py`, `models.py` |

### Точки входа

- API: `notification/api/routes.py`
- WebSocket: `notification/ws/hub.py`
- Events: `notification/events/handlers.py`

### Внутренние зависимости

notification.api → notification.storage
notification.ws → notification.storage
notification.events → notification.ws (push) + notification.storage (persist)

**Как добавить новый функционал:**
- Новый REST endpoint → создать route в `api/routes.py`, schema в `api/schemas.py`, query в `storage/repository.py`
- Новый тип уведомления из события → добавить case в `events/handlers.py`, маппинг события → шаблон уведомления

## Зависимости

Notification зависит от auth (критическая — без JWT-валидации REST и WS не работают) и от
всех сервисов-издателей через AMQP (некритическая — если издатель упал, notification продолжает
работать, просто новые события не приходят). Прямых sync-вызовов к другим сервисам кроме auth нет.

### auth — валидация JWT
Notification использует auth-сервис для валидации JWT-токенов при WebSocket-подключении
и REST-запросах. Для REST — автоматически через shared/auth middleware (разработчику достаточно
добавить `Depends(get_current_user)` в route). Для WebSocket — явный вызов при подключении.
Если auth недоступен — все запросы возвращают 401, WS-подключения отклоняются.
Паттерн: **Conformist** (notification конформен к API auth).
См. [POST /api/v1/auth/validate](auth.md#post-apiv1authvalidate).

### shared/auth/ — JWT Middleware
Notification использует shared JWT middleware для валидации токенов в HTTP-запросах.
Подключение: `from shared.auth import get_current_user` → `Depends(get_current_user)` в route.
Роль: consumer. Владелец: auth. Интерфейс: [conventions.md](../.system/conventions.md#sharedauth--jwt-middleware).

### * (любой сервис) — системные события
Notification подписывается на канал `system.events` через AMQP. Все сервисы, публикующие
события (auth, task, admin), являются неявными зависимостями. При добавлении нового сервиса,
который публикует события — достаточно добавить handler в `events/handlers.py`.
Паттерн: **Published Language** (стандартные события).
Роль: subscriber. Схемы событий: [conventions.md](../.system/conventions.md#sharedevents--схемы-событий-amqp).

## Доменная модель

Notification реализует домен Notifications. Основная сущность — Notification — проходит
жизненный цикл: создание (из события) → доставка (WS push) → чтение (пользователь пометил
read) → удаление (TTL 90 дней). WebSocketConnection — вспомогательная сущность, живёт
только пока пользователь подключён.

### Агрегаты

| Агрегат | Описание |
|---------|----------|
| Notification | Уведомление (тип, заголовок, тело, статус read/unread, TTL 90 дней). Жизненный цикл: created → delivered → read → expired |
| WebSocketConnection | Активное WS-соединение пользователя. Эфемерное — существует только при подключении |

### Инварианты

- Одно уведомление принадлежит одному пользователю
- Уведомление не может быть помечено read до сохранения в БД
- TTL уведомлений — 90 дней (cleanup удаляет старые)
- Max 10 одновременных WS-соединений на пользователя (при превышении — reject)

### Доменные события

| Событие | Описание |
|---------|----------|
| NotificationCreated | Новое уведомление сохранено → может порождать WS push (внешний Event channel) |
| NotificationRead | Статус обновлён на read |
`````
