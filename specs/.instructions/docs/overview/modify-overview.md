## Принципы

> **Консистентность между секциями.** При обновлении одной секции — проверить все зависимые. Сервис должен присутствовать одновременно в Карте, Связях и Контекстной карте.

> **Сначала таблицы, потом mermaid.** Mermaid-схема генерируется из данных таблиц. Обновлять в порядке: таблицы → mermaid → потоки.

> **Валидация в конце.** После любого изменения — запустить `validate-docs-overview.py`. Не считать обновление завершённым без прохождения валидации.

## Шаги

### Шаг 1: Определить триггер

Определить, какое изменение в системе произошло. Найти соответствующий триггер в [таблице](#триггеры-обновления) и перейти к указанному сценарию.

### Шаг 2: Выполнить обновление по сценарию

Выполнить шаги из соответствующего [сценария](#сценарии). Обновлять секции в порядке, указанном в сценарии — каждый следующий шаг зависит от предыдущего.

### Шаг 3: Валидировать результат

```bash
python specs/.instructions/.scripts/validate-docs-overview.py
```

Убедиться что OVW007 (консистентность сервисов) не нарушена.

## Сценарии

### Сценарий 1: Добавлен новый сервис

**Когда:** В проекте появился новый сервис (`/src/{new-svc}/`).

**Шаги:**

1. **Карта сервисов** — добавить строку в таблицу (алфавитный порядок):
   ```
   | {new-svc} | {зона ответственности} | {критичность} | {данные} | {2-4 endpoints} |
   ```
   Значение критичности берётся из `{new-svc}.md` (SSOT). Допустимые значения: `critical-high`, `critical-medium`, `critical-low`.

2. **Mermaid-схема** — добавить узел в соответствующий `subgraph`:
   ```
   NEW_SVC["{new-svc}"]
   ```
   Добавить рёбра для всех связей нового сервиса.

3. **Связи между сервисами** — добавить строки для каждой связи (алфавитный порядок по Источнику):
   ```
   | {new-svc} | {target} | {протокол} | {назначение} | {паттерн} |
   ```

4. **Контекстная карта доменов** — добавить строку:
   ```
   | {Domain} | {new-svc} | {агрегаты} | {паттерн}: {связь} |
   ```
   Обновить раздел DDD-паттерны, если новый паттерн.

5. **Сквозные потоки** — если новый сервис участвует в существующих потоках, обновить шаги и участников. Если появился новый критичный поток — добавить подсекцию.

6. **Shared-код** — если новый сервис потребляет shared-пакет, добавить в «Потребители». Если создаёт — добавить строку.

7. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-overview.py
   ```

### Сценарий 2: Удалён сервис

**Когда:** Сервис выведен из эксплуатации.

**Шаги:**

1. **Карта сервисов** — удалить строку сервиса из таблицы.
2. **Mermaid-схема** — удалить узел и все рёбра к/от сервиса.
3. **Связи между сервисами** — удалить все строки, где сервис = Источник или Приёмник.
4. **Контекстная карта доменов** — удалить строку домена. Обновить DDD-паттерны.
5. **Сквозные потоки** — удалить или переписать потоки с участием сервиса.
6. **Shared-код** — убрать сервис из «Потребители». Если сервис = Владелец — передать владение или удалить пакет.
7. **Валидация.**

### Сценарий 3: Изменён протокол связи

**Когда:** Связь между сервисами перешла на другой протокол (например, REST → gRPC).

**Шаги:**

1. **Связи между сервисами** — обновить колонку «Протокол» и «Паттерн» в затронутых строках.
2. **Mermaid-схема** — обновить подпись ребра: `протокол: что`.
3. **Сквозные потоки** — обновить протокол в шагах затронутых потоков.
4. **Валидация.**

### Сценарий 4: Добавлен shared-пакет

**Когда:** Создан новый пакет в `/shared/`.

**Шаги:**

1. **Shared-код** — добавить строку в таблицу:
   ```
   | shared/{package} | {назначение} | {владелец} | {потребители} |
   ```
2. Обновить вводный абзац, если новый пакет меняет принципы.
3. **Валидация.**

### Сценарий 5: Изменён endpoint сервиса

**Когда:** В `{svc}.md` изменён, добавлен или удалён endpoint.

**Шаги:**

1. **Карта сервисов** — проверить, входит ли endpoint в 2-4 ключевых. Если да — обновить колонку «Ключевые API».
2. **Сквозные потоки** — проверить, ссылается ли какой-либо поток на этот endpoint в «Ключевые контракты». Если да — обновить ссылку.
3. **Валидация.**

### Сценарий 6: Первичное заполнение (из визуализации)

**Когда:** overview.md содержит визуализацию — пример из стандарта (MyApp, 4 сервиса) с пометкой «Это визуализация». Файл уже соответствует формату стандарта, но содержание не отражает реальный проект.

**Шаги:**

1. **Прочитать стандарт:** [standard-overview.md](./standard-overview.md) — формат, секции, шаблон.
2. **Удалить blockquote-пометку** `> **Это визуализация.**...` в начале файла.
3. **Заменить содержание секций на актуальные данные проекта** — сохранить структуру (frontmatter, порядок секций, формат таблиц), заменить содержание:
   - Назначение системы — реальное описание проекта
   - Карта сервисов — реальные сервисы из `/src/`
   - Связи — реальные протоколы и зависимости
   - Сквозные потоки — реальные end-to-end сценарии
   - Контекстная карта доменов — реальные домены
   - Shared-код — реальные пакеты из `/shared/` (или заглушка, если ещё нет)
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-overview.py
   ```

### Сценарий 7: Изменена критичность сервиса

**Когда:** В `{svc}.md` изменён уровень критичности (`criticality`).

**Шаги:**

1. **Карта сервисов** — обновить значение в колонке «Критичность» для соответствующего сервиса. Допустимые значения: `critical-high`, `critical-medium`, `critical-low`.
2. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-overview.py
   ```
