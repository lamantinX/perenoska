## Когда валидировать

**Обязательно:**
- После создания или изменения любого `specs/docs/.technologies/standard-{tech}.md`
- Перед коммитом (автоматически через pre-commit hook)
- При code review

**Опционально:**
- При обновлении мажорной версии технологии (проверить секцию Версия и настройка)
- При добавлении нового паттерна (проверить секцию Паттерны кода)

| TECH001 | Отсутствует или некорректный frontmatter | Блокирующая |

| TECH003 | Отсутствует обязательная секция | Блокирующая |

| TECH005 | Таблица не содержит обязательных колонок | Предупреждение |

| TECH009 | Code-блок отсутствует в секции, где обязателен | Предупреждение |

## Шаги

### Шаг 0: Автоматическая проверка

```bash
python specs/.instructions/.scripts/validate-docs-technology.py
```

Скрипт проверяет правила TECH001-TECH009 для `standard-{tech}.md` и TECH-SEC001-003 для `security-{tech}.md`. Если скрипт пройден — валидация завершена. Если есть ошибки — исправить по кодам ниже.

**Если скрипт недоступен** — выполнить шаги 1-8 вручную.

### Шаг 1: Проверить frontmatter

**TECH001:** frontmatter должен содержать:

| Поле | Правило |
|------|---------|
| `description` | Непустое, формат: «Стандарт кодирования {Technology} — ...» |
| `standard` | Путь к `specs/.instructions/docs/technology/standard-technology.md` |
| `technology` | Kebab-case, совпадает с `{tech}` в имени файла |

Поле `standard-version` во frontmatter НЕ должно присутствовать — версия указывается в заголовке h1.

### Шаг 2: Проверить заголовок и версию

**TECH002:** Первый h1-заголовок должен соответствовать формату:

```
# Стандарт {Technology} v{X.Y}
```

- `{Technology}` — имя технологии с заглавной буквы (допускается аббревиатура: `PostgreSQL`, `Redis`)
- `v{X.Y}` — версия стандарта (числа, разделённые точкой): `v1.0`, `v2.3`

### Шаг 3: Проверить обязательные секции

**TECH003:** Проверить наличие всех 8 обязательных h2-секций:
1. `## Версия и настройка`
2. `## Конвенции именования`
3. `## Паттерны кода`
4. `## Антипаттерны`
5. `## Структура файлов`
6. `## Валидация`
7. `## Тестирование`
8. `## Логирование`

**TECH004:** Секции должны идти в указанном порядке. Дополнительные h2-секции не допускаются.

### Шаг 4: Проверить таблицы

**TECH005:** Проверить обязательные колонки в таблицах (если секция не stub):

| Секция | Обязательные колонки |
|--------|---------------------|
| Версия и настройка | Параметр, Значение |
| Конвенции именования | Объект, Конвенция, Пример |
| Антипаттерны | Антипаттерн, Почему плохо, Правильно |
| Логирование (таблица событий) | Событие, Уровень, Пример сообщения |
| Тестирование → Фреймворк и плагины | Компонент, Пакет, Назначение |

**Версия и настройка** — проверить обязательные строки в таблице:
- Строка «Версия»
- Строка «Ключевые библиотеки» (допускается `—` если нет)
- Строка «Конфигурация»

### Шаг 5: Проверить подсекции Паттерны кода

**TECH006:** Если секция «Паттерны кода» не stub — проверить:
1. Есть хотя бы одна h3-подсекция
2. Каждая h3-подсекция содержит code-блок (`` ``` ``)

### Шаг 6: Проверить подсекции Тестирование

**TECH007:** Если секция «Тестирование» не stub — проверить наличие h3-подсекций:
1. `### Фреймворк и плагины` — таблица
2. `### Фикстуры` — code-блок
3. `### Мокирование` — code-блок
4. `### Паттерны тестов` — code-блок

### Шаг 7: Проверить stub-текст

**TECH008:** Секции, которые НЕ содержат таблиц, code-блоков, h3-подсекций или обычного текста, должны содержать stub-текст в курсиве (`*...*`).

Секции «Версия и настройка» и «Конвенции именования» не допускают stub — у любой технологии есть версия и конвенции.

### Шаг 8: Проверить code-блоки

**TECH009:** Проверить наличие code-блоков в секциях, где они обязательны (если секция не stub):

| Секция | Code-блок обязателен |
|--------|---------------------|
| Паттерны кода | Да (в каждой h3-подсекции) |
| Структура файлов | Да (дерево файлов) |
| Тестирование → Фикстуры | Да |
| Тестирование → Мокирование | Да |
| Тестирование → Паттерны тестов | Да |
| Логирование | Да (настройка логгера) |

## Типичные ошибки

| Ошибка | Код | Причина | Решение |
|--------|-----|---------|---------|
| Нет frontmatter | TECH001 | Файл начинается без `---` | Добавить frontmatter по [standard-technology.md § 3](./standard-technology.md#3-frontmatter) |
| Нет поля `technology` | TECH001 | Пропущено при создании | Добавить `technology: {tech}` в frontmatter |
| `standard-version` во frontmatter | TECH001 | Скопировали из другого шаблона | Удалить — версия только в заголовке h1 |
| Нет версии в заголовке | TECH002 | Заголовок `# Стандарт PostgreSQL` без `v1.0` | Добавить версию: `# Стандарт PostgreSQL v1.0` |
| Секция «Логирование» отсутствует | TECH003 | Пропущена при создании | Добавить секцию (stub если не применимо) |
| Секции не по порядку | TECH004 | Тестирование перед Антипаттернами | Переставить секции по стандарту (§ 4) |
| Нет колонки «Пример» в Конвенции | TECH005 | Таблица неполная | Добавить колонку «Пример» с backticks |
| Паттерны кода без code-блока | TECH006 | Только текстовое описание | Добавить рабочий code-блок к каждому паттерну |
| Нет подсекции «Мокирование» | TECH007 | Пропущена при заполнении | Добавить все 4 подсекции Тестирования |
| Пустая секция без stub | TECH008 | Секция есть, но пустая | Добавить stub: `*{Секция} не применимо — {причина}.*` |
| Нет `type: security` | TECH-SEC001 | Frontmatter security-файла без type | Добавить `type: security` во frontmatter |
| Нет 5 h2-секций в security | TECH-SEC002 | Пропущена секция | Добавить по шаблону из [standard-security.md § 11](/.github/.instructions/actions/security/standard-security.md#11-per-tech-security-scanning) |
| Неправильное имя файла | TECH-SEC003 | Файл не `security-{tech}.md` | Переименовать в `security-{tech}.md` |
