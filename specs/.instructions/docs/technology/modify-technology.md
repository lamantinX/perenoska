## Обновление

### Сценарий 1: Обновление конвенций

**Когда:** Изменены правила кодирования (новое соглашение, уточнение существующего).

**Шаги:**

1. **Конвенции именования** — обновить/добавить/удалить строку в таблице.
2. **Антипаттерны** — если новая конвенция заменяет старый паттерн, добавить запись.
3. **Версионирование** — обновить минорную версию в заголовке h1 (v1.0 → v1.1). Правила — [standard-technology.md § 8](./standard-technology.md#8-версионирование).
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-technology.py specs/docs/.technologies/standard-{tech}.md
   ```

### Сценарий 2: Обновление мажорной версии технологии

**Когда:** Технология обновлена до новой мажорной версии (PostgreSQL 16 → 17).

**Шаги:**

1. **Версия и настройка** — обновить версию, библиотеки с версиями.
2. **Паттерны кода** — проверить все code-блоки на совместимость с новой версией. Обновить устаревшие API.
3. **Тестирование** — обновить версии тестовых фреймворков/плагинов если изменились.
4. **Антипаттерны** — добавить новые антипаттерны если API изменился.
5. **Версионирование** — обновить мажорную версию в заголовке h1 (v1.x → v2.0).
6. **Валидация.**

### Сценарий 3: Добавление паттерна

**Когда:** Обнаружен новый типовой паттерн, который нужно стандартизировать.

**Шаги:**

1. **Паттерны кода** — добавить h3-подсекцию с описанием «когда использовать» и code-блоком.
2. **Версионирование** — обновить минорную версию (v1.0 → v1.1).
3. **Валидация.**

## Деактивация

**Когда:** Технология выведена из проекта (все сервисы мигрировали на другую технологию).

### Шаг 1: Проверить использование

Убедиться, что технология не упоминается в Tech Stack ни одного `{svc}.md`:

```bash
grep -r "standard-{tech}.md" specs/docs/ --include="*.md"
```

Если есть ссылки — сначала обновить `{svc}.md` через [modify-service.md → Сценарий 4](../service/modify-service.md#сценарий-4-изменён-tech-stack).

### Шаг 2: Удалить файлы

1. Удалить `specs/docs/.technologies/standard-{tech}.md`
2. Удалить `specs/docs/.technologies/security-{tech}.md` (если существует)
3. Удалить `.claude/rules/{tech}.md`

### Шаг 3: Обновить docs/README.md

1. Удалить строку из таблицы «Стандарты технологий».
2. Удалить файл из дерева `.technologies/`.

### Шаг 4: Валидация

```bash
python specs/.instructions/.scripts/validate-docs-technology.py
```

## Миграция

**Когда:** Технология переименована (например, фреймворк поменял имя).

### Шаг 1: Переименовать файлы

```bash
mv specs/docs/.technologies/standard-{old-tech}.md specs/docs/.technologies/standard-{new-tech}.md
# Если существует security companion:
mv specs/docs/.technologies/security-{old-tech}.md specs/docs/.technologies/security-{new-tech}.md
```

### Шаг 2: Обновить frontmatter

Обновить поле `technology: {new-tech}` в frontmatter `standard-{new-tech}.md`.
Если существует `security-{new-tech}.md` — обновить `technology: {new-tech}` и в нём.

### Шаг 3: Обновить h1

Заменить имя технологии в заголовке h1.

### Шаг 4: Переименовать rule

```bash
mv .claude/rules/{old-tech}.md .claude/rules/{new-tech}.md
```

Обновить ссылки внутри rule на новое имя файла.

### Шаг 5: Обновить docs/README.md

1. Обновить строку в таблице «Стандарты технологий» (имя + ссылка).
2. Обновить дерево.

### Шаг 6: Обновить ссылки

См. [Обновление ссылок](#обновление-ссылок).

### Шаг 7: Валидация

```bash
python specs/.instructions/.scripts/validate-docs-technology.py specs/docs/.technologies/standard-{new-tech}.md
```

## Обновление ссылок

### При переименовании файла

Найти все ссылки на старый файл:

```bash
grep -r "standard-{old-tech}.md" specs/ --include="*.md"
grep -r "standard-{old-tech}.md" .claude/rules/ --include="*.md"
```

Обновить каждую найденную ссылку.

### При удалении файла

Найти и удалить/обновить все ссылки на удалённый файл:

```bash
grep -r "standard-{tech}.md" specs/ --include="*.md"
```
