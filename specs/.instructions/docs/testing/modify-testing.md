## Принципы

> **Синхронизация с overview.md.** Список сервисов в дереве файлов и секции Межсервисные тесты должен соответствовать карте сервисов из overview.md.

> **Каскадное обновление.** Добавление сервиса затрагивает минимум 2 секции: Структура файлов и Межсервисные тесты.

> **Валидация в конце.** После любого изменения — запустить валидацию. Не считать обновление завершённым без прохождения.

## Шаги

### Шаг 1: Определить триггер

Определить, какое изменение произошло. Найти соответствующий триггер в [таблице](#триггеры-обновления) и перейти к указанному сценарию.

### Шаг 2: Выполнить обновление по сценарию

Выполнить шаги из соответствующего [сценария](#сценарии).

### Шаг 3: Валидировать результат

```bash
python specs/.instructions/.scripts/validate-docs-testing.py
```

## Сценарии

### Сценарий 1: Добавлен новый тип тестов

**Когда:** В проекте появился новый тип тестов (Contract, Visual Regression, Property-based и т.д.).

**Шаги:**

1. **Обновить таблицу «Типы тестов»** — добавить строку с Тип, Что проверяет, Scope, Внешние зависимости, Когда запускается.
2. **Обновить секцию «Команды запуска»** — добавить пример прямого вызова для нового типа (если отличается от существующих).
3. **Обновить вводный абзац «Типы тестов»** — если количество типов изменилось, обновить число.
4. **Валидация.**

### Сценарий 2: Добавлен/удалён сервис

**Когда:** В проекте появился или удалён сервис (новая папка в `/src/`, новый контейнер в docker-compose).

**Шаги (добавление):**

1. **Обновить дерево «Структура файлов»** — добавить `src/{новый-сервис}/tests/` с комментарием.
2. **Обновить секцию «Межсервисные тесты»** — если сервис участвует в межсервисных сценариях.
3. **Проверить overview.md** — новый сервис должен быть отражён в карте сервисов.
4. **Валидация.**

**Шаги (удаление):**

1. **Удалить из дерева «Структура файлов»** — убрать `src/{удалённый-сервис}/tests/`.
2. **Обновить секцию «Межсервисные тесты»** — если сервис упоминался в паттерне или примерах.
3. **Если остался один сервис** — заменить содержание секции Межсервисные тесты на stub-текст: `*Межсервисные тесты не применимы — один сервис.*`
4. **Валидация.**

### Сценарий 3: Изменена стратегия мокирования

**Когда:** Изменён подход к мокированию на каком-либо уровне (например, перешли с in-memory мока БД на testcontainers для unit-тестов).

**Шаги:**

1. **Обновить таблицу «Стратегия мокирования»** — изменить затронутые ячейки.
2. **Обновить вводный абзац** — если общий принцип изменился.
3. **Валидация.**

### Сценарий 4: Сменён технологический стек

**Когда:** Заменён тестовый фреймворк, язык программирования или добавлен новый технологический стек.

**Шаги:**

1. **Обновить ссылки на per-tech** в секции «Стратегия мокирования» — заменить ссылку на `standard-{tech}.md`.
2. **Обновить примеры в «Команды запуска»** — новый тестовый фреймворк (pytest → jest, и т.д.).
3. **Если добавлен стек** — добавить дополнительную ссылку на per-tech в секции «Стратегия мокирования».
4. **Валидация.**

### Сценарий 5: Изменены команды запуска

**Когда:** Изменены make-таргеты, переименован тестовый фреймворк, изменены флаги.

**Шаги:**

1. **Обновить примеры прямого вызова** — только примеры фильтрации, не дублировать make-таргеты.
2. **Проверить ссылку на Makefile** — убедиться что ссылка актуальна.
3. **Валидация.**

### Сценарий 6: Первичное заполнение (из визуализации)

**Когда:** testing.md содержит визуализацию — пример из стандарта (MyApp) с пометкой «Это визуализация». Файл соответствует формату стандарта, но содержание не отражает реальный проект.

**Шаги:**

1. **Прочитать стандарт:** [standard-testing.md](./standard-testing.md) — формат, секции, шаблон.
2. **Удалить blockquote-пометку** `> **Это визуализация.**...` в начале файла.
3. **Заменить содержание секций на актуальные данные проекта** — сохранить структуру (frontmatter, порядок секций, формат таблиц), заменить содержание:
   - Типы тестов — реальные типы из проекта
   - Структура файлов — реальные сервисы из overview.md
   - Стратегия мокирования — реальная стратегия
   - Межсервисные тесты — реальные инструменты и паттерны
   - Тестовые данные — реальный подход к данным
   - Команды запуска — реальный тестовый фреймворк
4. **Валидация:**
   ```bash
   python specs/.instructions/.scripts/validate-docs-testing.py
   ```
