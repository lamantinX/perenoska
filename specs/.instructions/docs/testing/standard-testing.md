## 1. Назначение

`docs/.system/testing.md` — единый документ со стратегией тестирования системы. Стратегия — это ответы на 6 вопросов: какие тесты, где размещены, что мокировать, как тестировать межсервисное взаимодействие, какие данные использовать, как запускать. Всё остальное — per-tech детали или конфигурация, которые в testing.md не входят.

**Содержит:** стратегию — типы тестов, структуру файлов, стратегию мокирования, межсервисные тесты, тестовые данные, команды запуска.

**НЕ содержит:** per-tech паттерны тестирования (pytest fixtures, mocking asyncpg, Jest configuration) — это в `standard-{tech}.md` секция Тестирование. Структуру сервисов и потоки данных — это в `overview.md`. Конвенции именования тестов в коде — это в `conventions.md`. Конфигурацию CI pipeline — это в `infrastructure.md`. Процесс запуска тестов (make test, make test-e2e) — это в `standard-development.md`.

**Разграничение:** testing.md = стратегия тестирования (какие типы, где размещены, что мокировать), [standard-development.md](/.github/.instructions/development/standard-development.md) = процесс (make test, make test-e2e, порядок запуска при разработке).

**SSOT-зависимости:**
- [standard-development.md](/.github/.instructions/development/standard-development.md) — процесс запуска тестов (make test, make test-e2e)
- [standard-plan-test.md](/specs/.instructions/analysis/plan-test/standard-plan-test.md) — TC-N тестовые сценарии из analysis chain

**Целевая аудитория:** LLM-разработчик, которому нужно написать тест для своего изменения, определить тип теста, разместить файл в правильном месте, настроить моки и запустить.

**Триггеры обновления:** при добавлении нового типа тестов — обновить секции Типы тестов и Команды запуска; при добавлении/удалении сервиса — обновить секции Структура файлов и Межсервисные тесты; при изменении стратегии мокирования — обновить секцию Стратегия мокирования; при смене технологического стека — обновить ссылки на `standard-{tech}.md` и примеры в Командах запуска.

**Индексация:** testing.md должен быть зарегистрирован в `docs/README.md`. При создании или удалении — обновить индекс.

## 2. Секции

6 обязательных секций в фиксированном порядке. Имена секций фиксированы — использовать точные названия из таблицы. Дополнительную секцию создавать после Команд запуска только если тема: (1) не относится ни к одной из 6 обязательных секций ни по смыслу, ни по контексту, и (2) не может быть органично встроена в существующую секцию.

| # | Секция | Обязательность | Содержание |
|---|--------|---------------|-----------|
| 1 | Типы тестов | Обязательна | Какие типы используются, что каждый проверяет, когда запускается |
| 2 | Структура файлов | Обязательна | Где лежат тесты per-service и системные, именование |
| 3 | Стратегия мокирования | Обязательна | Что мокать на каждом уровне |
| 4 | Межсервисные тесты | Обязательна | Как тестировать взаимодействие 2+ сервисов |
| 5 | Тестовые данные | Обязательна | Подход к fixtures, factories, seed data |
| 6 | Команды запуска | Обязательна | Make targets, прямой вызов, фильтрация |

**Пустые секции:** если секция не применима (критерии неприменимости — в описании каждой секции ниже) — stub-текст в курсиве по шаблону: `*{Название раздела} не используется.*`. Правило применяется ко всем секциям документа, включая дополнительные. Удалять секцию запрещено.

### Типы тестов

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — общий принцип тестирования в проекте (unit для бизнес-логики, integration для хранилищ, e2e для полных сценариев)
2. Таблица типов тестов

**Обязательная таблица:**

| Колонка | Описание |
|---------|----------|
| Тип | Название типа теста |
| Что проверяет | Что именно тестируется |
| Scope | Масштаб: один модуль, сервис, вся система |
| Внешние зависимости | Моки или реальные |
| Когда запускается | При коммите, в CI, перед merge |

**Список типов расширяемый.** Шаблон содержит базовый набор (Unit, Integration, E2E, Smoke, Load). Если проект использует дополнительные типы (Contract, Visual Regression, Property-based) — добавить строки в таблицу.

**Критерий разграничения типов:** тест с реальной БД — integration, даже если тестирует один класс. Тест нескольких сервисов через реальные API — e2e, даже если проверяет одну функцию. Классификация определяется по scope и внешним зависимостям, а не по количеству тестируемых классов.

**Неприменимость:** секция Типы тестов не может быть неприменимой — любой проект имеет хотя бы один тип.

### Структура файлов

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — принцип организации (unit-тесты рядом с кодом сервиса, системные тесты отдельно)
2. Дерево файлов в code-блоке — структура тестовых директорий с комментариями

**Правила дерева:**
- Дерево должно отражать реальные сервисы проекта. Каждый сервис из `overview.md` должен присутствовать как `src/{имя-сервиса}/tests/`. Шаблонные заглушки `{svc}` в готовом testing.md недопустимы.
- Показать `tests/` для системных тестов (e2e, integration, smoke, load)
- Каждая строка — с комментарием назначения
- Именование файлов: `test_{module}.py`, `test_{scenario}.py`

**Неприменимость:** секция Структура файлов не может быть неприменимой — если есть тесты, они где-то лежат.

### Стратегия мокирования

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — общий принцип (чем выше уровень теста, тем меньше моков)
2. Таблица мокирования

**Обязательная таблица:**

| Колонка | Описание |
|---------|----------|
| Уровень | Unit / Integration / E2E |
| БД | Мок или реальный |
| Message Broker | Мок или реальный |
| Другие сервисы | Мок или реальный |
| Shared-код | Мок или реальный |
| Внешние API | Мок или реальный |

**Ссылка на per-tech:** после таблицы — ссылка на `standard-{tech}.md` секция Тестирование для конкретных паттернов мокирования. Ссылка должна указывать на реально существующий файл. Если технология не определена — написать: `*Паттерны мокирования будут добавлены при определении технологического стека.*` Если используется несколько технологических стеков — добавить отдельную ссылку для каждого.

**Неприменимость:** секция не может быть неприменимой — даже при отсутствии моков нужно явно указать, что всё реальное.

### Межсервисные тесты

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — ссылка на `overview.md` секция Сквозные потоки (источник e2e-сценариев)
2. Описание инструмента поднятия сервисов (docker-compose, testcontainers)
3. Паттерн написания межсервисного теста: Arrange → Act → Assert. Assert должен проверять результат минимум в двух сервисах — это ключевой критерий межсервисного теста.

**Правила:**
- Секция описывает паттерн написания межсервисного теста, а не реестр тестов. Перечислять конкретные тесты не нужно — сквозные потоки определены в overview.md (SSOT). Каждый поток из overview.md должен быть покрыт e2e-тестом — это требование к тест-сьюту, не к содержанию этой секции.
- Для проверки асинхронных событий — описать механизм polling с конкретными параметрами: интервал проверки и максимальный таймаут ожидания. Значения определяются проектом.

**Если сервис один** (нет межсервисного взаимодействия) — stub-текст: `*Межсервисные тесты не применимы — один сервис.*`

### Тестовые данные

**Обязательные элементы:**
1. Вводный абзац (1-3 предложения) — подход к тестовым данным (factories, fixtures, seed data)
2. Список принципов (маркированный список, минимум 3 пункта)

**Обязательные принципы** (каждый — отдельным пунктом; допустимо расширение формулировки, но смысл должен совпадать):
- Независимость тестов — запрет межтестовых зависимостей, тесты не зависят от порядка запуска
- Минимальность данных — фабрики генерируют минимальный валидный объект, тест переопределяет только нужные поля
- Изоляция данных — каждый тест создаёт и очищает свои данные (rollback, truncate)

**Неприменимость:** секция не может быть неприменимой — любой тест работает с данными.

### Команды запуска

**Обязательные элементы:**
1. Ссылка на Makefile как SSOT команд
2. Код для прямого вызова тестового фреймворка с фильтрацией

**Запрещено:** дублировать полный список make-команд — ссылаться на `make help`. В секции допустимы только примеры прямого вызова для фильтрации (по файлу, по паттерну). Порядок запуска тестов в CI определён в `infrastructure.md` — не дублировать.

**Неприменимость:** секция не может быть неприменимой — тесты нужно запускать.

## 3. Принципы

> **Один файл — вся стратегия.** Стратегия тестирования — ответы на 6 вопросов: какие тесты, где размещены, что мокировать, как тестировать межсервисное взаимодействие, какие данные использовать, как запускать. Всё это в одном документе. Per-tech детали — в `standard-{tech}.md`. CI pipeline — в `infrastructure.md`. Реестр тестов — не здесь.

> **Кросс-сервисный фокус.** testing.md описывает стратегию для всей системы, а не для одного сервиса. Per-service конфигурация тестов — в `{svc}.md`.

> **Не дублировать per-tech.** Конкретные фреймворки, fixtures, конфигурации — в `standard-{tech}.md` секция Тестирование. В testing.md — только ссылки.

> **Не дублировать overview.md.** Сквозные потоки — в `overview.md`. В testing.md — ссылка и описание паттерна написания e2e-тестов. Не перечислять тесты.

> **Не дублировать Makefile.** Команды запуска — через ссылку на `make help`. testing.md описывает примеры прямого вызова для фильтрации.

> **Расширяемость.** 6 обязательных секций в фиксированном порядке. Дополнительные секции — после Команд запуска, только если тема не вписывается ни в одну из 6 обязательных. Новую секцию не создавать, если содержание можно разместить в существующей.

> **Пустые секции.** Если секция не применима (критерии — в описании каждой секции) — stub-текст в курсиве: `*{Название раздела} не используется.*`. Удалять секцию запрещено.

## 5. Пример

Пример для проекта MyApp (4 сервиса: auth, task, notification, admin).

```markdown
---
description: Тестирование MyApp — типы тестов, мокирование, межсервисные тесты, тестовые данные.
standard: specs/.instructions/docs/testing/standard-testing.md

# Тестирование

## Типы тестов

В проекте 5 типов тестов. Unit — основа: быстрые, изолированные, покрывают бизнес-логику. Integration — проверяют работу с реальными БД и брокером. E2E — полные сценарии через HTTP/WS API. Smoke — быстрая проверка что сервисы живы после деплоя. Load — нагрузочное тестирование перед релизом.

| Тип | Что проверяет | Scope | Внешние зависимости | Когда запускается |
|-----|--------------|-------|--------------------|--------------------|
| Unit | Бизнес-логика, валидация, маппинг, утилиты | Один модуль/функция | Все мокаются | При каждом коммите (pre-commit), в CI |
| Integration | Запросы к БД, AMQP pub/sub, repository-слой | Сервис + его хранилища | Реальные БД (Docker), мок других сервисов | В CI на каждый PR |
| E2E | Полные сценарии через API: регистрация → создание задачи → уведомление | Вся система | Всё реальное (docker-compose) | В CI перед merge в main |
| Smoke | Healthcheck всех сервисов, базовый CRUD | Вся система | Всё реальное | После деплоя на staging/prod |
| Load | Пропускная способность, latency под нагрузкой | Вся система | Всё реальное | Перед релизом, по запросу |

## Структура файлов

Unit и integration тесты живут внутри сервиса — рядом с кодом, который тестируют. Системные тесты (e2e, smoke, load) — в корневой `/tests/`. Именование: файл `test_{module}.py`, функция `test_{action}_{scenario}`, класс `Test{Feature}`.

```
/
├── src/auth/tests/                  # Тесты auth-сервиса
│   ├── conftest.py                  # Fixtures: тестовая БД, мок AMQP
│   ├── test_token.py               # Unit: генерация/валидация JWT
│   └── test_repository.py          # Integration: запросы к PostgreSQL
├── src/task/tests/                  # Тесты task-сервиса
│   ├── conftest.py
│   ├── test_routes.py              # Unit: API endpoints (мок repository)
│   └── test_repository.py          # Integration: запросы к PostgreSQL
├── src/notification/tests/          # Тесты notification-сервиса
│   ├── conftest.py                  # Fixtures: тестовая БД, мок AMQP, фабрики
│   ├── test_repository.py          # Integration: запросы к PostgreSQL
│   ├── test_handlers.py            # Unit: обработка событий
│   ├── test_routes.py              # Unit: API endpoints (мок repository)
│   └── test_hub.py                 # Unit: WebSocket Hub логика
├── src/admin/tests/                 # Тесты admin-сервиса
│   ├── conftest.py
│   └── test_routes.py              # Unit: admin API
├── tests/                           # Системные тесты
│   ├── e2e/

## Стратегия мокирования

Принцип: чем выше уровень теста, тем меньше моков. Unit мокает всё за пределами тестируемого модуля. Integration поднимает реальные хранилища (PostgreSQL, Redis в Docker), но мокает другие сервисы. E2E не мокает ничего — все сервисы запущены реально.

| Уровень | БД | Message Broker | Другие сервисы | Shared-код | Внешние API |
|---------|-----|---------------|---------------|------------|-------------|
| Unit | Мок (in-memory) | Мок | Мок | Реальный | Мок |
| Integration | Реальный (Docker) | Реальный (Docker) | Мок (httpx mock) | Реальный | Мок |
| E2E | Реальный | Реальный | Реальный | Реальный | Мок (WireMock) |

Паттерны мокирования (pytest fixtures, AsyncMock, factory_boy) — в [standard-python.md](../.technologies/standard-python.md#тестирование) (backend).

## Межсервисные тесты

Межсервисные сценарии описаны в [overview.md](overview.md#сквозные-потоки) — это SSOT сквозных потоков. Для поднятия всех сервисов используется `docker-compose -f docker-compose.test.yml up`, который запускает все сервисы + хранилища в изолированной сети.

Паттерн написания межсервисного теста:
1. **Arrange:** создать начальное состояние (зарегистрировать пользователя, получить JWT)
2. **Act:** выполнить основное действие (создать задачу через API task-сервиса)
3. **Assert:** проверить результат минимум в двух сервисах (задача создана в task, уведомление появилось в notification, WS-сообщение отправлено)

При проверке асинхронных событий (AMQP → notification) — использовать polling с таймаутом: повторять проверку GET /notifications каждые 100ms, максимальный таймаут ожидания — 5 секунд.

## Тестовые данные

Подход: factory_boy для генерации объектов, pytest fixtures для подготовки состояния БД. Seed data для e2e хранится в `tests/e2e/fixtures/`. Каждый тест создаёт свои данные и очищает после себя (транзакционный rollback для integration, truncate для e2e).

**Принципы:**
- Каждый тест независим — не зависит от порядка запуска и данных других тестов
- Фабрики генерируют минимальный валидный объект, тест переопределяет только нужные поля
- Изоляция данных — каждый тест очищает за собой (транзакционный rollback для integration, truncate для e2e)
- Для e2e: seed data создаётся через API (не напрямую в БД), чтобы проверить полный цикл
- Чувствительные данные (JWT ключи) — фиксированные тестовые значения в `conftest.py`, не из .env

## 6. Требования по уровням критичности

Требования к тестированию зависят от уровня критичности сервиса. Уровень определяется в `{svc}.md` (поле `criticality`).

| Критерий | critical-high | critical-medium | critical-low |
|----------|--------------|-----------------|--------------|
| Unit-test coverage | ≥80% | ≥60% | ≥40% |
| Integration tests | Обязательны | Обязательны | Рекомендуются |
| E2E tests | Обязательны, в CI | Обязательны, в CI | Опционально |
| Load testing | Обязательно перед релизом | При значительных изменениях | Не требуется |
| Chaos testing | Рекомендуется | Не требуется | Не требуется |
